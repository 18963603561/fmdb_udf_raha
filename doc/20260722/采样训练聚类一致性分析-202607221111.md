# 采样与训练聚类一致性分析

生成时间：2026-07-22 11:11

## 结论

采样聚类和训练聚类**可以不同**，但必须明确区分用途，不能把两者当成同一份语义产物混用。

更准确地说：

| 问题 | 结论 |
|---|---|
| 算法上是否允许不同 | 允许 |
| 当前工程是否允许不同 | 允许，当前代码没有强制采样聚类和训练聚类完全相同 |
| 是否建议随意不同 | 不建议 |
| 是否会影响模型 | 会影响，主要通过训练阶段标签传播影响训练样本 |
| 是否会导致标注失效 | 不一定，前提是标注绑定的是稳定行和单元格身份，而不是绑定旧聚类编号 |
| 最推荐方案 | 采样可使用动态粒度，训练使用稳定粒度，并强绑定数据快照、配置指纹和标注批次 |

## 两类聚类的职责不同

采样阶段聚类的主要职责是：在有限标注预算下，挑选更有价值的待标注行。

训练阶段聚类的主要职责是：根据已经获得的直接标签，把标签传播到同簇单元格，扩充训练样本。

所以两者关注点不同：

| 阶段 | 聚类服务对象 | 核心目标 | 错误代价 |
|---|---|---|---|
| 采样 | 标注任务选择 | 尽量覆盖更多未知区域 | 采样效率下降，多问一些价值较低的样本 |
| 训练 | 标签传播和模型训练 | 尽量传播准确，避免污染训练集 | 错误标签进入训练集，模型质量下降 |

因此，采样聚类可以偏探索，训练聚类必须偏稳定和保守。

## 原版 Raha 的设计

原版 `raha/detection.py` 的核心逻辑是先为每列构建一次层次聚类树，然后切出多个粒度：

```text
k = 2, 3, 4, ..., LABELING_BUDGET + 1
```

采样时使用：

```text
k = 已标注行数 + 2
```

标签传播时使用：

```text
k = 已标注行数 + 1
```

这个设计表达的是主动学习中的“从粗到细”：

| 标注阶段 | 聚类粒度 | 目的 |
|---|---|---|
| 早期 | 粗粒度 | 快速覆盖大范围数据分布 |
| 中期 | 中等粒度 | 平衡覆盖率和传播精度 |
| 后期 | 细粒度 | 降低错误传播风险 |

这个设计非常适合小数据和交互式标注，但不适合直接搬到 5 万行以上的 Spark 生产场景。

原因是原版依赖全量层次聚类树，计算和内存接近平方级；同时 `LABELING_BUDGET=20` 时最大只有 21 个簇，对 5 万行会过粗。

## 当前工程的实现现状

当前工程中，采样和训练都会在需要时重新执行聚类。

采样服务中，如果请求没有传入已准备好的聚类结果，会调用聚类服务重新聚类，然后把聚类结果交给覆盖采样服务。

训练服务中，如果准备结果里没有聚类结果，也会重新聚类。训练阶段聚类随后会进入标签传播服务，用于生成传播标签，再构建列级训练数据。

当前代码链路可以概括为：

```text
采样：
特征 -> 聚类 -> 覆盖评分 -> 标注任务

训练：
特征 -> 聚类 -> 标签传播 -> 训练样本 -> 模型
```

这说明当前工程本身已经把“采样聚类”和“训练聚类”作为两个可复用但不强制相同的阶段产物。

## 可以不同的前提

采样和训练聚类可以不同，前提是满足下面条件。

### 1. 标签必须绑定稳定单元格身份

人工标签不能只绑定到旧聚类编号，必须绑定到稳定的单元格身份，例如：

```text
datasetId + snapshotId + rowId + columnName
```

只要标签绑定的是单元格，训练阶段重新聚类后，标签仍然可以映射到新的训练聚类中。

如果标签绑定的是“采样时的 clusterId”，那么训练重新聚类会导致标签语义失效。

### 2. 数据快照必须可追踪

采样和训练可以使用不同数据范围，但必须记录清楚：

| 元数据 | 作用 |
|---|---|
| `datasetId` | 确认逻辑数据集一致 |
| `snapshotId` | 确认数据内容版本 |
| `schemaHash` | 确认字段结构一致 |
| `rowIdentityConfig` | 确认行身份规则一致 |
| `inputSqlHash` | 确认 SQL 输入口径 |
| `executionConfigFingerprint` | 确认策略、特征、聚类、模型配置 |

如果这些信息不清楚，采样和训练聚类不同会变成不可解释差异。

### 3. 训练聚类必须独立版本化

训练聚类用于标签传播，直接影响训练样本。

因此训练聚类必须有自己的版本，包括：

```text
provider
targetClusterCount
maxSampleCount
distanceMetric
randomSeed
featureDictionaryVersion
trainingSnapshotId
```

不能只因为它来自同一个采样批次，就默认训练聚类和采样聚类等价。

## 什么时候不同是合理的

### 场景一：采样小样本，训练全量

这是常见生产场景。

例如：

```text
采样：400 行，用于生成标注任务
训练：5 万行，用于训练最终模型
```

这时采样聚类和训练聚类必然不同。

这种不同是合理的，因为采样阶段关注“问哪些人”，训练阶段关注“如何把标签传播到全量数据”。

但需要注意：如果采样样本太小，直接标签覆盖不到训练全量中的主要分布，训练阶段传播效果可能有限。

### 场景二：采样动态粒度，训练固定粒度

这是比较推荐的方案。

采样阶段可以参考原版 Raha，按标注轮次逐步增加簇数：

```text
samplingClusterCount = min(targetClusterCount, max(2, labeledCount + 2))
```

训练阶段使用固定稳定粒度：

```text
trainingClusterCount = 100
```

这样采样能保留主动学习的探索能力，训练又能保证模型版本稳定。

### 场景三：采样用 Smile，训练用 Scalable

这在数据量差异很大时也合理。

例如：

| 阶段 | 数据量 | 聚类器 |
|---|---:|---|
| 采样 | 400 或 5000 以下 | `SmileHierarchicalColumnClusterer` |
| 训练 | 5 万 | `ScalableColumnClusterer` |

这正是 `raha.clustering.provider=AUTO` 的价值：小样本尽量精确，大样本自动近似。

## 什么时候不同是不合理的

### 1. 数据口径不一致但没有记录

例如：

```text
采样 SQL：select * from dw.person_info limit 400
训练 SQL：select * from dw.person_info where city = 'A'
```

如果没有明确记录这两个 SQL 的关系，训练结果就无法解释。

### 2. 行身份不一致

如果采样阶段的 `rowId` 和训练阶段的 `rowId` 不是同一套规则，人工标签可能映射到错误行。

这是最严重的问题。

### 3. 训练阶段错误复用采样聚类

如果采样只用了 400 行，却把这 400 行的聚类当成 5 万训练数据的聚类使用，这是错误的。

采样聚类只能覆盖采样输入中的单元格，不能代表训练全量。

### 4. 聚类配置变化但模型版本没有变化

如果 `targetClusterCount`、`provider` 或 `maxSampleCount` 变化，但训练模型版本没有体现这些配置变化，会造成模型不可复现。

## 对模型训练的影响

采样聚类不同不会直接改变模型，因为采样聚类只决定哪些行被标注。

训练聚类不同会直接影响模型，因为训练聚类决定标签传播范围。

影响链路是：

```text
训练聚类不同
-> 簇成员不同
-> 同一个直接标签传播到不同单元格
-> 训练样本正负分布不同
-> 列级模型参数不同
-> 预测结果不同
```

所以真正需要重点控制的是训练聚类，而不是采样聚类。

## 对采样效果的影响

采样聚类粒度会影响标注任务的质量。

| 采样簇数 | 效果 | 风险 |
|---:|---|---|
| 太少 | 容易覆盖大类，前期效率高 | 细分模式覆盖不足 |
| 太多 | 探索更细，覆盖更充分 | 每个簇标签少，采样可能过于分散 |
| 动态增加 | 前期粗、后期细 | 实现和版本管理更复杂 |

如果固定 `targetClusterCount=100`，对于 5 万数据是合理的；但对于早期只有少量标注时，100 个簇可能让标注分布过散。

所以采样阶段更适合动态簇数。

## 推荐业务方案

结合当前工程和 `dw.person_info` 这类 Spark 表，推荐采用下面策略。

### 1. 聚类后端

保持：

```properties
raha.clustering.provider=AUTO
raha.clustering.target-cluster-count=100
raha.clustering.max-sample-count=5000
```

语义是：

```text
单列 <= 5000：Smile 精确聚类
单列 > 5000：Scalable 近似聚类
```

### 2. 采样阶段

采样阶段建议后续改为动态粒度：

```text
samplingClusterCount = min(100, max(2, labeledCount + 2))
```

如果一次性生成 300 个待标注任务，也可以按轮次或批次模拟增长，例如：

```text
第 1 批：20 个簇
第 2 批：50 个簇
第 3 批及以后：100 个簇
```

这样比固定 100 更接近原版 Raha 主动学习逻辑。

### 3. 训练阶段

训练阶段建议固定：

```text
trainingClusterCount = 100
```

训练阶段不建议随标注轮次不断变化，否则同一批标注多次训练会产生明显模型漂移。

### 4. 预测阶段

预测阶段不参与聚类。

预测只加载已发布模型并对当前特征打分，所以预测不需要采样聚类或训练聚类一致。

## 最终判断

采样和训练聚类可以不同，而且在生产上经常应该不同。

但必须遵守一条原则：

```text
采样聚类用于选样，训练聚类用于传播，二者可以不同，但标签、快照、配置和模型版本必须强绑定。
```

对当前工程来说，最符合业务的不是完全照搬 Python 原版，也不是永远固定一个簇数，而是：

```text
后端使用 AUTO
采样使用动态粒度
训练使用固定稳定粒度
预测不参与聚类
```

这样既保留 Raha 主动学习的算法思想，又能适配 Spark SQL UDF、5 万数据、任务表审计和模型复现要求。
