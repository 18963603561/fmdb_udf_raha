# 训练按快照复用采样产物落地设计

## 1. 背景

当前训练入口要求传入 `sampleBatchId`，并会根据采样批次找到最新可训练标注批次。训练阶段如果没有显式传入 `datasetId`、`sqlText`、`tableName` 等输入信息，会从 `dw.raha_sample_record` 中还原原始输入，再重新执行：

1. `LOAD_DATA`
2. `MERGE_TRAINING_INPUT`
3. `PROFILE`
4. `GENERATE_STRATEGY`
5. `RUN_STRATEGY`
6. `GENERATE_FEATURE`
7. `CLUSTER`
8. `LABEL`
9. `PROPAGATE`
10. `TRAIN`

用户希望新增一种训练模式：

当训练请求中 `snapshotId` 有值，并且 `datasetId`、`sqlText`、`tableName` 为空时，表示训练数据与采样阶段数据完全一致，此时训练直接复用采样阶段的 `PROFILE`、`RUN_STRATEGY`、`FEATURE`、`CLUSTER`，不再重复执行这些重阶段。

## 2. 目标规则

### 2.1 复用模式触发条件

训练请求同时满足以下条件时，进入快照复用模式：

| 条件 | 要求 |
| --- | --- |
| `sampleBatchId` | 必填 |
| `snapshotId` | 必填 |
| `datasetId` | 为空 |
| `sqlText` 或 `sql` | 为空 |
| `tableName` 或 `table` | 为空 |
| `sampleBatch.snapshotId` | 必须等于请求中的 `snapshotId` |

### 2.2 原流程保留条件

以下任一情况仍走原训练流程：

| 场景 | 行为 |
| --- | --- |
| 请求中传了 `sqlText` | 按 SQL 重新加载并重新计算产物 |
| 请求中传了 `tableName` | 按表重新加载并重新计算产物 |
| 请求中传了 `datasetId` 且带输入覆盖 | 按输入覆盖重新计算产物 |
| `snapshotId` 为空 | 维持当前训练逻辑 |
| 请求 `snapshotId` 与采样批次 `snapshotId` 不一致 | 直接失败，避免错用产物 |

## 3. 当前代码状态

### 3.1 训练入口

入口文件：

```text
src/main/java/com/fiberhome/ml/raha/udf/RahaDetectionUdfService.java
```

当前逻辑：

```java
FmdbInputSpec inputOverride = parser.inputSpec(false);
TrainingRequestOptions options = new TrainingRequestOptions(..., inputOverride, ...);
RahaTaskExecutionRequest taskRequest = requestFactory.training(..., options);
```

问题：

当请求只传 `snapshotId`，没有 `sourceType`、`sqlText`、`tableName` 时，`parser.inputSpec(false)` 返回 `null`，`snapshotId` 不会进入 `TrainingRequestOptions`，后续训练工厂也无法知道用户要按快照复用。

### 3.2 训练工厂

入口文件：

```text
src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskRequestFactory.java
```

当前逻辑：

1. 根据 `sampleBatchId` 读取 `SampleBatch`。
2. 根据 `sampleBatchId` 读取最新标注批次。
3. 调用 `resolveTrainingInput(...)` 还原训练输入。
4. 生成 `RahaTaskExecutionRequest`。

问题：

默认还原出的 `FmdbInputSpec` 不携带请求中的 `snapshotId`，并且训练请求没有“复用采样阶段产物”的显式开关。

### 3.3 训练工作流

入口文件：

```text
src/main/java/com/fiberhome/ml/raha/service/task/TrainingWorkflow.java
```

当前逻辑：

```text
LOAD_DATA
MERGE_TRAINING_INPUT
PROFILE
GENERATE_STRATEGY
RUN_STRATEGY
GENERATE_FEATURE
CLUSTER
LABEL
PROPAGATE
TRAIN
```

问题：

复用模式下不能执行 `MERGE_TRAINING_INPUT`。因为合并会产生新的训练快照，行集合会从“采样阶段快照”变成“采样标注行加原始输入行”的组合快照，导致 `FEATURE` 和 `CLUSTER` 与训练数据不再一致。

## 4. 现有仓储是否支持完整复用

### 4.1 PROFILE

接口：

```text
ColumnProfileRepository.findBySnapshot(datasetId, snapshotId)
```

结论：

可以按 `datasetId + snapshotId` 读取，具备复用基础。

### 4.2 STRATEGY_PLAN

接口：

```text
StrategyRepository.findPlans(datasetId, snapshotId)
```

结论：

可以按 `datasetId + snapshotId` 读取计划，具备复用基础。

### 4.3 RUN_STRATEGY

接口：

```text
StrategyRepository.findHits(jobId)
```

现状：

`FmdbStrategyRepository` 中 `StrategyHit` 当前只保存在 `pendingHits` 内存结构中。代码注释明确说明：策略命中只在特征生成前短暂存在，不写入最终物理表。

结论：

当前不支持跨 SQL 复用。

### 4.4 FEATURE

接口：

```text
FeatureRepository.findDictionary(jobId, columnName)
FeatureRepository.findRows(jobId, columnName)
```

现状：

物理恢复依赖 `training_batch_id`，主要从训练阶段的 `dw.raha_training_column_artifact` 和 `dw.raha_training_cell` 读取。

结论：

当前不支持按采样 `snapshotId` 复用采样阶段特征。

### 4.5 CLUSTER

接口：

```text
ClusterRepository.findResult(jobId, columnName, clusterVersion)
ClusterRepository.findAssignments(jobId, columnName, clusterVersion)
```

现状：

物理恢复同样依赖 `training_batch_id`，聚类成员映射来自 `dw.raha_training_cell`。

结论：

当前不支持按采样 `snapshotId` 复用采样阶段聚类。

## 5. 结论

这个需求在业务规则上可以支持，但当前项目还不能只改训练工作流就完成。

完整复用 `PROFILE / RUN_STRATEGY / FEATURE / CLUSTER` 必须先补齐采样阶段中间产物的物理化，否则训练进程无法从上一条 SQL 的采样任务中恢复 `StrategyHit`、特征向量和聚类成员。

如果现在只做：

```text
snapshotId 有值时跳过 PROFILE / RUN_STRATEGY / FEATURE / CLUSTER
```

训练会在 `LABEL`、`PROPAGATE`、`TRAIN` 阶段拿不到必要上下文，最终要么空训练，要么运行失败，要么训练数据与聚类结果错位。

## 6. 推荐落地方案

### 6.1 表结构调整

用户已经明确“不需要兼容旧表，直接删除表字段”。因此可以采用删表重建或清库后重建 schema，不做旧 schema 兼容。

建议新增统一采样阶段产物表：

```sql
CREATE TABLE dw.raha_snapshot_stage_artifact (
    artifact_batch_id STRING NOT NULL,
    dataset_id STRING NOT NULL,
    snapshot_id STRING NOT NULL,
    source_job_id STRING NOT NULL,
    stage_type STRING NOT NULL,
    column_name STRING,
    artifact_version STRING NOT NULL,
    artifact_json STRING NOT NULL,
    row_set_fingerprint STRING NOT NULL,
    config_fingerprint STRING NOT NULL,
    created_at BIGINT NOT NULL,
    partition_month STRING NOT NULL
)
USING ORC
PARTITIONED BY (dataset_id, partition_month)
TBLPROPERTIES ('raha.schema.version' = '2');
```

建议新增统一采样阶段单元格产物表：

```sql
CREATE TABLE dw.raha_snapshot_cell_artifact (
    artifact_batch_id STRING NOT NULL,
    dataset_id STRING NOT NULL,
    snapshot_id STRING NOT NULL,
    source_job_id STRING NOT NULL,
    row_id STRING NOT NULL,
    column_name STRING NOT NULL,
    cell_id STRING NOT NULL,
    cell_value STRING,
    feature_dictionary_version STRING,
    feature_vector_json STRING,
    feature_summary_json STRING,
    cluster_version STRING,
    cluster_id STRING,
    cluster_distance DOUBLE,
    strategy_hit_json STRING,
    row_set_fingerprint STRING NOT NULL,
    config_fingerprint STRING NOT NULL,
    created_at BIGINT NOT NULL,
    partition_month STRING NOT NULL
)
USING ORC
PARTITIONED BY (dataset_id, partition_month)
TBLPROPERTIES ('raha.schema.version' = '2');
```

### 6.2 采样记录表调整

建议 `dw.raha_sample_record` 增加：

```sql
sample_record_role STRING NOT NULL
```

取值：

| 值 | 含义 |
| --- | --- |
| `ANNOTATION_TASK` | 最终待人工标注数据 |
| `CANDIDATE_POOL` | 采样阶段完整候选数据 |

这样可以区分“需要人工标注的样本”和“采样阶段参与计算的全集”。

### 6.3 仓储接口调整

建议新增统一接口：

```java
public interface SnapshotStageArtifactRepository {
    void saveProfile(...);
    void saveStrategyHits(...);
    void saveFeatureAssembly(...);
    void saveClustering(...);
    Optional<SnapshotPreparedArtifacts> findPreparedArtifacts(
            String datasetId,
            String snapshotId,
            String rowSetFingerprint,
            String configFingerprint);
}
```

核心校验字段：

| 字段 | 作用 |
| --- | --- |
| `datasetId` | 防止跨数据集复用 |
| `snapshotId` | 防止跨快照复用 |
| `rowSetFingerprint` | 防止行集合不一致 |
| `configFingerprint` | 防止策略、特征、聚类配置不一致 |

### 6.4 工作流调整

新增阶段：

```text
ReusePreparedArtifactsStageHandler
```

复用模式训练工作流改为：

```text
LOAD_DATA
REUSE_PREPARED_ARTIFACTS
LABEL
PROPAGATE
TRAIN
RESULT_PERSISTENCE
```

其中 `REUSE_PREPARED_ARTIFACTS` 需要写入以下上下文：

| 上下文键 | 来源 |
| --- | --- |
| `RAHA_DATASET` | `LOAD_DATA` |
| `STRATEGY_PLANS` | 快照产物表 |
| `STRATEGY_PLAN_VERSION` | 快照产物表 |
| `STRATEGY_BATCH_RESULT` | 快照产物表 |
| `STRATEGY_HITS` | 快照单元格产物表 |
| `FEATURE_ASSEMBLY_RESULT` | 快照单元格产物表 |
| `CLUSTERING_BATCH_RESULT` | 快照单元格产物表 |

### 6.5 UDF 参数调整

`TrainingRequestOptions` 建议增加：

```java
private final String requestedSnapshotId;
private final boolean reusePreparedArtifacts;
```

触发判断放在 `RahaDetectionUdfService.train(...)`：

```text
snapshotId 非空
sourceType 为空
sqlText 为空
sql 为空
tableName 为空
table 为空
datasetId 为空
```

并校验：

```text
requestedSnapshotId == sampleBatch.getSnapshotId()
```

不一致时直接返回失败：

```text
SNAPSHOT_MISMATCH
```

```json
{
  "sampleBatchId": "sample_dw.person_info@20260721134516.455",
  "snapshotId": "snapshot_dw.person_info@20260721134516.455",
  "annotationDir": "/fmdb/raha/annotation-upload",
  "allowPartialAnnotation": "false",
  "modelNamePrefix": "person_info",
  "publishZip": "true",
  "artifactBaseDir": "doc/20260722/person-info-reuse-run",
  "localWebRoot": "doc/20260722/person-info-reuse-run/web",
  "hdfsExportPath": "/fmdb/detection/output/person-info/",
```

## 8. 实施顺序

1. 删除或重建当前 FMDB Raha 物理表，升级 schema 到版本 2。
2. 给 `dw.raha_sample_record` 增加 `sample_record_role`，采样阶段同时保存 `ANNOTATION_TASK` 和 `CANDIDATE_POOL`。
3. 新增 `dw.raha_snapshot_stage_artifact` 和 `dw.raha_snapshot_cell_artifact`。
4. 采样阶段在 `FEATURE` 和 `CLUSTER` 后，把完整产物写入快照产物表。
5. 增加 `SnapshotStageArtifactRepository`。
6. 增加 `ReusePreparedArtifactsStageHandler`。
7. 扩展 `TrainingRequestOptions` 和 `RahaTaskRequestFactory`，让 `snapshotId` 独立进入训练请求。
8. `TrainingWorkflow` 根据 `reusePreparedArtifacts` 改用复用流程。
9. 增加单元测试，覆盖复用模式、回退原流程、快照不一致失败。
10. 在 Docker Spark 中按 SQL 重新执行采样、训练、预测，验证训练阶段不再重复执行 `PROFILE / RUN_STRATEGY / FEATURE / CLUSTER`。

## 9. 风险与约束

### 9.1 行集合一致性

复用模式必须保证训练数据就是采样阶段同一个快照的同一个行集合。如果训练阶段重新读取 SQL，即使 SQL 文本相同，也可能因为底表变化或排序变化导致行集合不同。

### 9.2 配置一致性

以下配置变化都会导致旧产物不可复用：

| 配置 | 影响 |
| --- | --- |
| 策略启停 | 改变 `RUN_STRATEGY` |
| 特征生成规则 | 改变 `FEATURE` |
| 聚类 provider | 改变 `CLUSTER` |
| 聚类目标簇数 | 改变 `CLUSTER` |
| 行身份配置 | 改变行和单元格对齐 |

因此需要 `configFingerprint`，不能只靠 `snapshotId` 判断。

### 9.3 当前不能直接代码完成的原因

当前缺少：

1. 采样阶段完整 `StrategyHit` 物理化。
2. 采样阶段完整 `FeatureAssemblyResult` 物理化。
3. 采样阶段完整 `ColumnClusteringResult` 物理化。
4. 按 `snapshotId` 恢复上述产物的仓储接口。
5. 复用阶段向训练上下文注入完整产物的阶段处理器。

## 10. 最终结论

这条规则应该实现，且业务含义是正确的：只有 `snapshotId` 有值、训练输入为空时，才表示“训练使用采样阶段同一份数据”，可以复用采样产物；其他情况继续走原流程。

但当前工程还不具备完整复用底座。建议先完成采样中间产物物理化和按快照恢复能力，再改训练编排。这样训练阶段才能真正跳过耗时的 `PROFILE / RUN_STRATEGY / FEATURE / CLUSTER`，并保证标签传播和模型训练拿到的是同一份数据上的特征与聚类结果。
