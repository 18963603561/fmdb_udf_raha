## 2. 新旧流程对比

### 2.1 采样流程

旧流程：

```text
LOAD_DATA -> PROFILE -> GENERATE_STRATEGY -> RUN_STRATEGY -> GENERATE_FEATURE -> CLUSTER -> SAMPLE -> PERSIST_RESULT
```

新流程：

```text
LOAD_DATA -> PROFILE -> GENERATE_STRATEGY -> RUN_STRATEGY -> GENERATE_FEATURE -> CLUSTER -> SNAPSHOT_CHECKPOINT -> SAMPLE -> PERSIST_RESULT
```

采样阶段会新增一次检查点写入。以历史 `person_info limit 400` 为例，检查点大致会写入特征行、聚类归属、列画像、策略摘要、特征字典和 manifest，数据量约为数千行级别。这个成本会增加采样耗时，但它是一次性成本，换来后续训练跳过重阶段。

### 2.2 训练流程

旧流程：

```text
LOAD_DATA -> MERGE_TRAINING_INPUT -> PROFILE -> GENERATE_STRATEGY -> RUN_STRATEGY -> GENERATE_FEATURE -> CLUSTER -> LABEL -> PROPAGATE -> TRAIN -> PERSIST_RESULT
```

新复用流程：

```text
RESTORE_SNAPSHOT_CHECKPOINT -> LABEL -> PROPAGATE -> TRAIN -> PERSIST_RESULT
```

触发条件：

```sql
SELECT F_DW_DETTRAIN(
    '{"sampleBatchId":"sample_dw.person_info@20260721134516.455","snapshotId":"对应采样快照"}'
);
```

如果训练请求仍然传入 `sourceType`、`sqlText`、`sql`、`tableName`、`table` 或 `datasetId`，则继续走旧流程，不启用检查点复用。

### 2.3 预测流程

本次没有改造预测复用链路。预测仍按现有流程执行 `LOAD_DATA`、`PROFILE`、`GENERATE_STRATEGY`、`RUN_STRATEGY`、`GENERATE_FEATURE`、`PREDICT`、`PERSIST_RESULT`。  
后续如果要进一步优化预测，应另行设计“模型特征字典和预测特征生成复用”方案，不能直接照搬训练的聚类检查点。

## 4. 预期优化收益

训练阶段理论上可少跑约 261.409 秒的重复计算。新增的 `RESTORE_SNAPSHOT_CHECKPOINT` 会读取数千行检查点记录，实际耗时取决于 ORC 分区裁剪、HDFS 读取和 JSON 反序列化，保守估计为 5 到 20 秒。

按当前历史基线估算：

| 项目 | 旧训练 | 新训练预估 |
| --- | ---: | ---: |
| 阶段累计耗时 | 292.059 秒 | 35 到 55 秒 |
| Raha job 耗时 | 333.851 秒 | 45 到 75 秒 |
| SQL 外层耗时 | 354.936 秒 | 65 到 100 秒 |

收益最大的不是模型训练本身，而是跳过了 `CLUSTER`、`RUN_STRATEGY` 和 `MERGE_TRAINING_INPUT`。如果同一次采样后需要多次调整标注并重新训练，收益会继续放大。

需要注意：采样会新增 `SNAPSHOT_CHECKPOINT` 写入成本。对于只采样一次、只训练一次的链路，净收益约等于“训练节省耗时减去采样检查点写入耗时”；对于一次采样多次训练的链路，检查点成本只承担一次。

## 5. 任务表写入优化对比

旧编排器行为：

| 对象 | 旧写入方式 |
| --- | --- |
| `dw.raha_job_run` | 提交写一次，每个阶段开始和结束都可能刷写，任务结束再写 |
| `dw.raha_job_stage_attempt` | 阶段开始写 `RUNNING`，阶段结束再写终态 |

本次落地行为：

| 对象 | 新写入方式 |
| --- | --- |
| `dw.raha_job_run` | 提交写一次，首阶段启动写一次，任务最终成功或失败写一次 |
| `dw.raha_job_stage_attempt` | 每个阶段只在成功、部分成功、跳过或失败时写一次终态 |

这会减少阶段表的 `RUNNING` 记录，也会减少每个阶段结束后的 `job_run` 刷写。  
当前还没有进一步实现跨小阶段的大阶段缓冲写，例如把 `PROFILE + STRATEGY + FEATURE + CLUSTER` 合成一个大阶段状态。后续若继续优化，可把当前“每阶段终态一次”升级为“大阶段终态一次”。

## 6. 阶段耗时观测方式

`RahaUdfDriverApp` 已新增阶段耗时输出。执行 UDF 后会在原始 JSON 结果之后额外打印：

```text
RAHA_STAGE_TIMINGS_JSON={...}
```

输出内容包括：

| 字段 | 含义 |
| --- | --- |
| `jobId` | 当前 UDF 返回的任务标识 |
| `stageTimingAvailable` | 是否成功读取阶段表 |
| `stageCount` | 阶段记录数 |
| `totalStageMillis` | 阶段终态记录的耗时合计 |
| `stages` | 每个阶段的 `stageType`、`status`、开始时间、完成时间和耗时 |

由于用户当前要求三函数执行仍以 SQL 为准，本次没有用 `RahaUdfDriverApp` 重新提交业务任务。该入口主要用于本地调试和补充阶段耗时观测。

## 7. 风险和边界

| 风险 | 说明 | 处理方式 |
| --- | --- | --- |
| 检查点缺失 | 采样没有跑到 `SNAPSHOT_CHECKPOINT` 或检查点表被清理 | 训练返回 `CHECKPOINT_NOT_FOUND`，不静默重跑 |
| 快照不一致 | 请求 `snapshotId` 与采样批次快照不同 | 训练 UDF 返回 `SNAPSHOT_MISMATCH` |
| 配置指纹变化 | 聚类、特征或策略配置变化后不应复用旧产物 | 恢复时按 `config_fingerprint` 匹配 |
| 检查点表变大 | 大数据量下 `CELL_FEATURE` 和 `CLUSTER_ASSIGNMENT` 行数较多 | 后续需要压缩、分区裁剪和必要字段瘦身 |
| 预测尚未复用 | 本次只解决采样到训练复用 | 预测复用需要单独方案 |

## 8. 后续优化优先级

1. 优先验证新训练请求在容器中通过 SQL 执行，确认 `RESTORE_SNAPSHOT_CHECKPOINT` 阶段真实耗时。
2. 观察 `dw.raha_snapshot_checkpoint` 的分区裁剪是否生效，重点看 `dataset_id`、`snapshot_id`、`config_fingerprint` 条件下的读取耗时。
3. 继续优化检查点写入，把单元格级特征和聚类归属压缩存储，减少 ORC 小文件和 JSON 解析开销。
4. 将任务表写入从“每阶段终态一次”进一步升级为“大阶段终态一次”。
5. 单独设计预测链路复用，减少预测阶段重复画像、策略执行和特征生成。

## 9. 结论

本次落地已经把训练最大的重复成本从流程上切掉：训练复用采样检查点后，不再重复执行 `LOAD_DATA`、`MERGE_TRAINING_INPUT`、`PROFILE`、`GENERATE_STRATEGY`、`RUN_STRATEGY`、`GENERATE_FEATURE` 和 `CLUSTER`。  
按历史数据估算，单次训练阶段累计耗时有机会从 292.059 秒降到 35 到 55 秒区间，端到端 SQL 外层耗时预计从约 355 秒降到 65 到 100 秒区间。真实收益需要下一次容器 SQL 实测确认，关键观测点是新增的 `RESTORE_SNAPSHOT_CHECKPOINT` 和采样新增的 `SNAPSHOT_CHECKPOINT`。
