# 标准 XLS 可读性与训练闭环优化分析

## 1. 问题结论

当前标准 XLS 可以支撑程序导入，但对测试人员和标注人员不够友好。

主要问题不是“完全没有元信息”，而是元信息放在隐藏的 `系统信息` sheet 中，并且字段不完整。当前模板里已有 `sampleBatchId`，但没有 `snapshotId`、`annotationDir`、训练请求示例和 UDF 调用说明。

对于测试截断场景，确实没有必要做得过于严格。可以把 `系统信息` sheet 显示出来，并允许编辑，同时增加一个面向人的 `训练说明` sheet，把训练请求、UDF 调用、标注上传和文件命名说明统一放进去。

## 2. 当前 XLS 元信息现状

本次标准 XLS 的 `系统信息` sheet 中已有以下字段：

| 字段 | 当前是否存在 | 说明 |
| --- | --- | --- |
| `templateVersion` | 是 | 模板版本 |
| `datasetId` | 是 | 数据集标识 |
| `samplePartitionMonth` | 是 | 采样分区月份 |
| `sampleBatchId` | 是 | 采样批次标识 |
| `schemaHash` | 是 | 字段结构哈希 |
| `recordCount` | 是 | 标注行数 |
| `businessColumnsJson` | 是 | 业务字段列表 |
| `detectableColumnsJson` | 是 | 可检测字段列表 |
| `snapshotId` | 否 | 复用采样检查点训练需要 |
| `annotationDir` | 否 | 标注上传目录，和部署环境有关 |
| `trainRequestJson` | 否 | 可直接复制的训练请求 |
| `trainSqlExample` | 否 | 可直接执行或参考的 UDF 调用示例 |

## 3. 为什么现在隐藏会影响测试体验

原设计把 `系统信息` 隐藏，是为了防止误改 `sampleBatchId`、`schemaHash`、字段列表等机器校验字段。

但在测试截断场景中，标注人员往往同时也是调试人员。此时隐藏这些字段会带来几个问题：

1. 标注完成后不知道该用哪个 `sampleBatchId` 训练。
2. 复用检查点训练时不知道该传哪个 `snapshotId`。
3. 不知道标注文件应该上传或复制到哪个目录。
4. 不知道 `F_DW_DETTRAIN` 的参数该怎么拼。
5. 需要去 ZIP 里的 `summary.json` 或 `manifest.json` 里找信息，链路不自解释。

因此，测试模式下可以降低严格性，把 `系统信息` 显示出来，并补充训练说明。

## 4. `snapshotId` 中 `@content-...` 是否可以去掉

当前 `snapshotId` 示例：

```text
snapshot_dw.person_info@content-fa37df3849ec096a094e4cb5e98f8a75
```

其中 `snapshot_dw.person_info` 是可读来源名，`content-fa37...` 是内容指纹。

它的作用是区分同一张表在不同时间、不同内容下的快照。如果直接改成：

```text
snapshot_dw.person_info
```

可读性会提升，但会丢失内容版本区分能力。对于真实环境，这有风险：同名表内容变了，但快照标识不变，后续复用检查点可能误用旧数据。

更好的折中方案是允许采样请求传入短的 `sourceVersion`，例如：

```json
{
  "sourceVersion": "person-info-20260722"
}
```

这样生成的快照可以变成：

```text
snapshot_dw.person_info@person-info-20260722
```

既保留版本区分，又比 `content-fa37...` 可读。

## 5. `modelSetVersion` 为什么带 `jobId`

当前训练生成的模型集合版本类似：

```text
dw.person_info@20260722075201.035-job-f439dbf2-fb19-459f-82c0-ab26c6354b05
```

代码里是用数据源名、时间戳和 `jobId` 共同生成。这样做的原因是保证模型集合版本唯一，尤其是在同一数据、同一时间附近重复训练时，避免版本冲突。

但从人工阅读角度看，完整 `jobId` 太长，降低可读性。

测试场景可以改成短版本，例如：

```text
dw.person_info@20260722155228
```

字段模型版本也可以对应简化为：

```text
dw.person_info.age@20260722155228
```

如果担心同秒重复训练冲突，可以加短序号或短哈希：

```text
dw.person_info@20260722155228-r2
```

## 6. 建议的 XLS 改造

### 6.1 显示系统信息

测试模式下建议把 `系统信息` sheet 从隐藏改为可见。

建议新增配置：

```properties
raha.annotation.excel.system-sheet-visible=true
```

默认可以仍保持严格隐藏，测试数据集或本地调试打开显示。

### 6.2 系统信息增加字段

建议在 `系统信息` sheet 只增加：

| 字段 | 示例 |
| --- | --- |
| `snapshotId` | `snapshot_dw.person_info@person-info-20260722` |

`annotationDir`、训练请求模板、UDF 调用示例和标注上传说明不放入 `系统信息`，统一放入可见的 `训练说明` sheet。

### 6.3 增加训练说明 sheet

新增可见 sheet：`训练说明`。

建议内容：

```text
1. 完成标注后，将本文件保存为带 _labeled 后缀的 .xls 文件。
2. 将标注文件上传或复制到 annotationDir 指定目录。
3. 训练请求至少需要 sampleBatchId、snapshotId、annotationDir。
4. 如果要复用采样检查点，不要在训练请求中传 sourceType、datasetId、sqlText、tableName。
```

最小训练请求示例：

```json
{
  "sampleBatchId": "sample_dw.person_info@20260722071220.940",
  "snapshotId": "snapshot_dw.person_info@person-info-20260722",
  "annotationDir": "/fmdb/raha/annotation-upload",
  "forceRun": true,
  "forceRunId": "t1"
}
```

UDF 调用示例：

```sql
SELECT F_DW_DETTRAIN(
  'sampleBatchId=sample_dw.person_info@20260722071220.940&snapshotId=snapshot_dw.person_info@person-info-20260722&annotationDir=/fmdb/raha/annotation-upload&forceRun=true&forceRunId=t1'
);
```

同时在 `训练说明` sheet 中增加标注上传和命名说明：

```text
1. 训练会按 sampleBatchId 匹配标注文件。
2. 文件名中的 @ 会被替换为 _。
3. 示例匹配前缀：raha-annotation_sample_dw.person_info_20260722071220.940_
4. 不要把未标注原始模板放到 annotationDir 中，避免训练选到未标注文件。
5. 同一批次多个标注文件同时存在时，系统会选择修改时间最新的文件。
```

## 7. 说明内容统一放入 Excel

不再额外生成 `train-request-template.json`、`train-call-example.sql`、`annotation-upload-readme.md` 三个旁路文件。

这三类内容统一合并到 Excel 的 `训练说明` sheet 中：

| 内容 | 放置位置 | 用途 |
| --- | --- | --- |
| 训练请求模板 | `训练说明` sheet | 可直接复制构造训练请求 |
| SQL UDF 调用示例 | `训练说明` sheet | 可直接复制执行训练 |
| 标注上传和命名说明 | `训练说明` sheet | 说明标注文件放哪里、文件名如何匹配 |

这样采样标准文件本身就包含完整闭环说明，不需要用户再去 ZIP 中找其它文件。

## 8. 推荐落地顺序

第一步，先做测试友好改造：

1. `系统信息` sheet 改为可见。
2. `系统信息` 只增加 `snapshotId`。
3. 增加 `训练说明` sheet，并合并训练请求、UDF 调用、标注上传和文件命名说明。

第二步，再做版本可读性改造：

1. 采样请求支持用户传短 `sourceVersion`。
2. 本地测试默认生成短 `snapshotId`。
3. 训练模型版本支持测试模式短版本。
4. 结果中保留完整 `jobId` 字段，但不要拼进对用户展示的模型版本。

## 9. 最终建议

测试截断场景下，应优先提高可读性和闭环自解释能力。

建议保留内部字段用于程序校验，但允许在测试模式下显示和编辑。`snapshotId` 不建议完全去掉版本片段，而是把 `content-哈希` 替换成用户可读的 `sourceVersion`。`modelSetVersion` 中完整 `jobId` 可以不再作为展示版本的一部分，改用短时间戳或短运行号。
