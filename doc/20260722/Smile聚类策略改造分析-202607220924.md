# Smile聚类策略改造分析

生成时间：2026-07-22 09:24

## 目标

只保留一个聚类配置参数，通过它选择：

1. 现有自定义实现
2. Smile 成熟实现
3. 自动识别模式

推荐形式：

```properties
raha.clustering.provider=SmileHierarchicalColumnClusterer
```

## 你的期望

你希望配置项只保留一个，不再拆成：

- linkage
- metric
- max-clusters
- 其他 Smile 细项

而是直接写后端名，例如：

- `HierarchicalColumnClusterer`
- `ScalableColumnClusterer`
- `SmileHierarchicalColumnClusterer`
- `SmileBirchColumnClusterer`
- `AUTO`

这个思路是可行的。

## 推荐设计

### 1. 单参数后端选择

只新增一个配置项：

```properties
raha.clustering.provider=AUTO
```

含义：

- `HierarchicalColumnClusterer`，走现有手写层次聚类
- `ScalableColumnClusterer`，走现有可扩展实现
- `SmileHierarchicalColumnClusterer`，走 Smile 层次聚类
- `SmileBirchColumnClusterer`，走 Smile 近似聚类
- `AUTO`，由工程自动判断

### 2. 自动识别规则

建议实现一个统一解析器：

1. 先按字符串匹配类名或别名
2. 再按是否引入 Smile 依赖决定 Smile 后端是否可用
3. 再按数据量选择层次聚类或近似聚类
4. 若都不满足，回退到现有自定义实现

这样对外仍然只有一个参数，但内部可以自动切换。

## 对现有工程的影响

### 1. `pom.xml`

需要：

- 引入 Smile 依赖
- 保持 JDK 8 编译
- 继续使用 `maven-shade-plugin`

### 2. `src/main/resources/raha-defaults.properties`

只新增一个参数即可：

```properties
raha.clustering.provider=AUTO
```

### 3. `ClusteringConfig`

建议只增加一个字段：

- `provider`

不要把 Smile 的 linkage、metric、簇数继续暴露成一堆配置项。
这些默认值放到实现内部即可。

### 4. `RahaConfigFactory`

负责把 `raha.clustering.provider` 读进来。

### 5. `RahaConfigValidator`

只校验后端名是否合法。

### 6. `RahaTaskApplicationServiceFactory`

按 `provider` 选择实现：

- 自定义层次聚类
- 自定义可扩展聚类
- Smile 后端
- `AUTO` 自动选择

### 7. 新增适配器

建议新增统一接口适配层，例如：

- `SmileColumnClusterer`
- `ClusteringProviderResolver`

让工厂只处理一个字符串参数，不直接感知 Smile 细节。

## 为什么这样更合适

1. 配置更简单。
2. 运维切换更直观。
3. 后续扩展不用再加很多参数。
4. 你现在最关心的是“切后端”，不是“细调算法细节”。

## 风险

1. Smile 默认参数不暴露后，灵活度会下降。
2. 如果后端名写错，需要在启动期直接报错。
3. `AUTO` 逻辑要写得足够清楚，否则后面排障会绕。

## 建议落地顺序

1. 先引入 Smile 依赖并打进 shade 包
2. 新增 `raha.clustering.provider`
3. 新增后端解析器
4. 工厂按单参数切换实现
5. 保留回退到自定义实现的路径
6. 回归验证采样、训练、预测耗时

## 本次落地结果

本次已经把 `AUTO` 从“默认等价 Smile”调整为“运行时动态路由”：

```properties
raha.clustering.provider=AUTO
raha.clustering.target-cluster-count=100
raha.clustering.max-sample-count=5000
```

当前实际语义如下：

| 单列样本数 | 实际聚类器 | 说明 |
|---:|---|---|
| `<= max-sample-count` | `SmileHierarchicalColumnClusterer` | 使用 Smile 平均连接层次聚类，优先保证小样本聚类质量 |
| `> max-sample-count` | `ScalableColumnClusterer` | 使用确定性余弦分区近似聚类，避免精确层次聚类平方级开销 |

新增 `AutoColumnClusterer` 作为独立路由器，避免把 `AUTO` 直接映射到某一个具体实现。
这样显式配置 `SmileHierarchicalColumnClusterer`、`ScalableColumnClusterer` 或
`HierarchicalColumnClusterer` 时仍然保持原有语义，只有配置 `AUTO` 时才按数据量自动切换。

需要注意：`max-sample-count=5000` 是进入 Smile 精确聚类的单列样本上限。
如果同时开启多列并发，多个字段同时执行 5000 规模层次聚类会增加 Driver 内存压力，
生产上应结合 `raha.resource.max-parallel-columns` 和实际字段数观察。

## 结论

按你的要求，最佳方案不是多个 Smile 细项，而是**一个后端参数**。

最终默认配置长这样就够了：

```properties
raha.clustering.provider=AUTO
raha.clustering.target-cluster-count=100
raha.clustering.max-sample-count=5000
```

如果需要强制指定实现，也可以显式配置 `SmileHierarchicalColumnClusterer`、
`ScalableColumnClusterer` 或 `HierarchicalColumnClusterer`。
