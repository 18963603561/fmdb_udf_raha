# 训练产物三表默认不入库与恢复链路删减分析

## 一、目标结论

本次目标是把以下三张表降级为“可选审计记录表”，默认不保存入库，并且不再承担恢复、复现或训练主输入职责：

| 表名 | 新定位 |
| --- | --- |
| `dw.raha_training_column_artifact` | 可选保存训练列级审计摘要，默认不入库 |
| `dw.raha_training_cell` | 可选保存训练单元格明细，默认不入库 |
| `dw.raha_training_example` | 可选保存训练入模样本记录，默认不入库 |

对应的主流程应调整为：

```text
训练阶段使用内存中的画像、策略、特征、聚类、标签传播和训练样本。
模型发布只依赖 dw.raha_model_artifact。
预测阶段只读取 dw.raha_model_artifact 和当前输入数据。
三张训练产物表不再被默认读取，不再参与恢复和回读训练。
即使配置开启三表保存，也只作为旁路记录，不参与训练、预测、恢复、幂等和成功校验。
```

因此，当前工程里围绕这三张表建立的“写后回读”“物理恢复”“计数校验”代码可以明显简化。

## 二、开启保存时的硬约束

三张训练产物表的目标定位不是“默认不保存，但开启后恢复旧能力”，而是：

```text
默认不保存。
配置开启时只记录。
记录失败不影响训练成功。
记录内容不作为任何后续阶段输入。
记录表不参与任务幂等、模型加载、预测、断点恢复和结果正确性校验。
```

开启保存时，允许的行为只有：

| 行为 | 是否允许 |
| --- | --- |
| 训练完成后追加审计记录 | 允许 |
| 用户离线查询训练记录 | 允许 |
| 排障时人工查看记录 | 允许 |
| 训练时从 `training_example` 回读样本 | 禁止 |
| 特征阶段从 `training_column_artifact` 恢复字典 | 禁止 |
| 聚类阶段从 `training_cell` 恢复成员 | 禁止 |
| 模型预测读取三张训练产物表 | 禁止 |
| 训练成功校验要求三张表写入成功 | 禁止 |
| 三张表写失败导致训练失败 | 默认禁止 |

因此后续代码设计中，三张表只能作为 `best effort` 的审计写入目标。除非业务另行明确要求强审计，否则写入异常应记录 `warn` 或审计失败摘要，不应中断模型训练和发布。

## 三、当前工程现状

### 3.1 默认配置现状

当前默认配置中：

| 配置项 | 当前默认值 | 与目标是否一致 |
| --- | --- | --- |
| `raha.persistence.table.training-column-artifact.enabled` | `true` | 不一致 |
| `raha.persistence.table.training-cell.enabled` | `false` | 一致 |
| `raha.persistence.table.training-example.enabled` | `true` | 不一致 |

目标应改为：

```properties
raha.persistence.table.training-column-artifact.enabled=false
raha.persistence.table.training-cell.enabled=false
raha.persistence.table.training-example.enabled=false
```

### 3.2 当前训练主流程对三表的依赖

当前 `RahaTrainService` 在存在 `TrainingMergeResult` 时会执行以下逻辑：

```text
构造内存训练样本 builtDatasets
检查 training-example 持久化必须开启
写入 training_column_artifact、training_cell、training_example
再从 training_example 读回 frozenDatasets
使用 frozenDatasets 训练模型
```

这意味着当前代码不是“只记录”，而是把 `dw.raha_training_example` 变成了训练主输入的一部分。

如果三张表只做可选记录，这段逻辑必须改为：

```text
构造内存训练样本 builtDatasets
可选异步或同步写审计记录
直接使用 builtDatasets 训练模型
```

### 3.3 当前读取恢复路径

当前以下仓储会从三张表中读取历史物理产物：

| 类 | 当前读取内容 |
| --- | --- |
| `FmdbFeatureRepository` | 从 `training_column_artifact` 读特征字典，从 `training_cell` 读特征向量 |
| `FmdbClusterRepository` | 从 `training_column_artifact` 读聚类摘要，从 `training_cell` 读聚类成员 |
| `FmdbColumnProfileRepository` | 从 `training_column_artifact` 读字段画像 |
| `FmdbStrategyRepository` | 从 `training_column_artifact` 读策略计划 |
| `FmdbTrainingArtifactRepository` | 读 `training_cell`、`training_example`，并从 `training_example` 恢复训练数据集 |
| `FmdbModelStore` | 具备从 `training_column_artifact` 按字典版本读取特征字典的辅助方法 |

如果三张表只做记录，这些物理读取分支必须删除或改为空实现。不能保留“开启保存后可恢复”的隐含能力，否则三表仍会继续增加主流程复杂度。

## 四、可以直接删除或简化的代码

### 4.1 删除训练样本写后回读链路

涉及文件：

```text
src/main/java/com/fiberhome/ml/raha/service/train/RahaTrainService.java
src/main/java/com/fiberhome/ml/raha/service/train/TrainingArtifactMaterializationService.java
src/main/java/com/fiberhome/ml/raha/service/train/TrainingArtifactMaterializationResult.java
src/main/java/com/fiberhome/ml/raha/service/train/RahaTrainOutput.java
```

建议删除或改造点：

| 位置 | 建议 |
| --- | --- |
| `RahaTrainService.loadFrozenDatasets` | 删除 |
| `RahaTrainService` 中 `isTrainingExamplePersistenceEnabled` 强制校验 | 删除 |
| `RahaTrainService` 中 `frozenDatasets = loadFrozenDatasets(...)` | 改为直接使用 `builtDatasets` |
| `TrainingArtifactMaterializationService.loadFrozenTrainingDataset` | 删除 |
| `TrainingArtifactMaterializationResult.materializedExampleCount` 对训练正确性的强绑定 | 删除或仅保留审计计数 |
| `RahaTrainOutput.materializationResult` | 可删除；若保留，仅作为审计写入摘要 |

推荐训练主流程改为：

```text
Map<String, ColumnTrainingDataset> builtDatasets = buildTrainingDatasets(...)
TrainingArtifactMaterializationResult materialization = tryMaterializeForAudit(...)
final Map<String, ColumnTrainingDataset> trainingDatasets = builtDatasets
```

注意：`tryMaterializeForAudit` 不应影响训练成功与否。审计写入失败最多记录告警和摘要，不能回滚模型训练结果。

### 4.2 删除 `FmdbTrainingArtifactRepository` 的读取能力

涉及文件：

```text
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbTrainingArtifactRepository.java
```

建议删除方法：

| 方法 | 删除原因 |
| --- | --- |
| `findTrainingCells` | 不再从 `training_cell` 恢复单元格明细 |
| `findTrainingExamples` | 不再从 `training_example` 查询冻结样本 |
| `loadFrozenTrainingDataset` | 不再用 `training_example` 回读训练模型 |
| `countColumnArtifacts` | 不再把列级产物作为训练成功必检项 |
| `countTrainingCells` | `training_cell` 默认不入库，不应计数校验 |
| `countTrainingExamples` | `training_example` 默认不入库，不应计数校验 |
| `isColumnArtifactPersistenceEnabled` | 若只保留写入可选逻辑，可用通用 `shouldPersist` 代替 |
| `isTrainingCellPersistenceEnabled` | 同上 |
| `isTrainingExamplePersistenceEnabled` | 同上 |

如果三表仍允许配置开启作为审计记录，则该类可以保留以下写方法：

```text
saveColumnArtifacts
saveTrainingCells
saveTrainingExamples
```

但写入前应先判断三个表是否全部关闭。全部关闭时不应构造大量 `FmdbTrainingCellRecord` 和 `FmdbTrainingExampleRecord`，避免默认路径仍消耗内存。开启写入时也不能提供任何读取方法给主流程调用。

### 4.3 删除训练产物物化服务中的默认构造成本

涉及文件：

```text
src/main/java/com/fiberhome/ml/raha/service/train/TrainingArtifactMaterializationService.java
```

当前服务即使表开关关闭，也会先构造：

```text
List<FmdbTrainingCellRecord>
List<FmdbTrainingExampleRecord>
List<FmdbTrainingColumnArtifactRecord>
```

目标是默认不入库，因此建议：

1. 三表全部关闭时，直接返回空审计结果。
2. 只有某张表开启时，才构造该表对应记录。
3. `training_cell` 记录量最大，必须避免默认构造。
4. `training_example` 不再作为训练输入，只作为可选审计记录。

如果要进一步简化，可以删除整个 `TrainingArtifactMaterializationService`，改为一个轻量的 `TrainingAuditRecordWriter`，职责只做可选写入，不提供读取、恢复、校验和训练输入生成。

### 4.4 删除特征、策略、画像、聚类仓储的物理恢复分支

涉及文件：

```text
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbFeatureRepository.java
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbClusterRepository.java
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbColumnProfileRepository.java
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbStrategyRepository.java
```

建议删除或简化点：

| 类 | 建议 |
| --- | --- |
| `FmdbFeatureRepository.findDictionary` | 只查内存 `pendingDictionaries`，不再查 `training_column_artifact` |
| `FmdbFeatureRepository.findRows` | 只查内存 `pendingRows`，不再查 `training_cell` |
| `FmdbClusterRepository.findResult` | 只查内存 `pendingResults`，不再查 `training_column_artifact` |
| `FmdbClusterRepository.findAssignments` | 只查内存，删除 `physicalAssignments` |
| `FmdbColumnProfileRepository.findBySnapshot` | 只查内存或直接返回空，删除 `physicalProfiles` |
| `FmdbStrategyRepository` 的 `artifactRowsBySnapshot`、`artifactRowsByBatch`、`artifactRows` | 删除物理表读取 |

删除后，这些仓储将回到“当前任务内上下文仓储”的定位。跨进程恢复如仍需要，应由 `dw.raha_snapshot_checkpoint` 负责，而不是由训练最终产物三表负责。

### 4.6 删除训练结果持久化校验中的三表校验




| `TrainingArtifactMaterializationResult` 强依赖 | 不再作为训练成功条件 |

### 4.7 调整持久化依赖校验

涉及文件：

```text
src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/support/FmdbPersistenceConfig.java
```

当前依赖校验中存在：

```text
TRAINING_CELL 依赖 TRAINING_COLUMN_ARTIFACT
TRAINING_EXAMPLE 依赖 TRAINING_COLUMN_ARTIFACT
MODEL_ARTIFACT 依赖 TRAINING_COLUMN_ARTIFACT
MODEL_ARTIFACT 依赖 STRATEGY_PLAN
MODEL_ARTIFACT 依赖 FEATURE_DICTIONARY
```

目标下，模型预测只依赖 `dw.raha_model_artifact`，因此必须删除：

```text

MODEL_ARTIFACT -> STRATEGY_PLAN
MODEL_ARTIFACT -> FEATURE_DICTIONARY
```

三张表只做记录后，三表之间也不应存在强依赖。即使只开启 `training_example` 或只开启 `training_cell`，也应允许任务启动并按配置记录对应内容。审计记录之间的关联完整性可以通过字段冗余和离线查询规则处理，不能用启动期依赖阻塞主流程。

## 六、建议分阶段实施

### 6.1 第一阶段：主流程去恢复化

目标：

```text
训练不再从 dw.raha_training_example 回读样本。
三张表默认关闭。
模型发布和预测只依赖 dw.raha_model_artifact。
```

改动：

1. 修改默认配置，把三表默认值改为 `false`。
2. 删除 `RahaTrainService.loadFrozenDatasets`。
3. 训练时直接使用 `builtDatasets`。
4. 删除 `training-example` 必须开启的强校验。
5. 修改训练结果校验，只校验 `dw.raha_model_artifact`。
6. 修改 `FmdbPersistenceConfig`，删除模型表对训练列级产物的依赖。

收益：

```text
训练主路径变短。
默认不读写三张训练产物表。
预测链路更清晰。
```

### 6.2 第二阶段：删除物理恢复分支

目标：

```text
特征、策略、画像、聚类仓储不再从训练三表恢复历史状态。
```

改动：

1. 删除 `FmdbFeatureRepository` 的训练产物表读取。
2. 删除 `FmdbClusterRepository` 的训练产物表读取。
3. 删除 `FmdbColumnProfileRepository` 的训练产物表读取。
4. 删除 `FmdbStrategyRepository` 的训练产物表读取。
5. 删除 `FmdbTrainingArtifactRepository` 的所有读取和计数方法。

收益：

```text
仓储职责回归当前任务上下文。
跨任务恢复统一交给 snapshot_checkpoint。
训练最终产物三表不再影响主流程。
```

### 6.3 第三阶段：可选审计写入瘦身

目标：

```text
如果三表只是记录，不为默认路径构造记录对象。
```

改动：

1. 三表全部关闭时跳过 `TrainingArtifactMaterializationService`。
2. 只有开启目标表时才构造该表记录。
3. 将 `TrainingArtifactMaterializationService` 改名为 `TrainingAuditRecordWriter`。
4. `TrainingArtifactMaterializationResult` 改为可选审计摘要，不能影响训练成败。

收益：

```text
默认训练不再构造大对象。
审计写入失败不影响模型训练。
```

### 6.4 第四阶段：长期删除三表结构

目标：

```text
如果确认不再需要审计记录，则彻底删除三张物理表。
```

改动：

1. 从 `raha-fmdb-schema.sql` 删除三张表。
2. 从 `FmdbPhysicalTable` 删除三个枚举。
3. 从 `FmdbTableSchemas` 删除三个 schema。
4. 删除 `FmdbTrainingArtifactRepository` 和三类记录对象。
5. 删除对应测试。

风险：

```text
历史环境和历史文档会失去对应表结构。
如果未来重新需要训练审计，需要重新建表或改用其他审计表。
```

## 七、推荐最终保留表关系

如果按“代码简洁、预测可用、训练可追踪到模型”为目标，推荐默认只保留以下核心表入库：

| 表 | 默认保存 | 说明 |
| --- | --- | --- |
| `dw.raha_sample_record` | 是 | 人工标注任务和训练批次关联 |
| `dw.raha_annotation_record` | 是 | 人工标注结果，不可重新计算 |
| `dw.raha_snapshot_checkpoint` | 是或按需 | 采样到训练复用快照 |
| `dw.raha_model_artifact` | 是 | 模型发布和预测核心表 |
| `dw.raha_detection_result` | 是 | 检测错误结果 |
| `dw.raha_job_run` | 是 | 任务状态和幂等 |
| `dw.raha_job_stage_attempt` | 是或按需 | 阶段排障 |

三张训练产物表默认关闭：

| 表 | 默认保存 | 说明 |
| --- | --- | --- |
| `dw.raha_training_column_artifact` | 否 | 可选审计 |
| `dw.raha_training_cell` | 否 | 可选明细审计 |
| `dw.raha_training_example` | 否 | 可选样本审计 |

## 八、最终判断

在当前目标下，以下代码可以优先删除：

1. `RahaTrainService.loadFrozenDatasets` 和训练样本写后回读逻辑。
2. `FmdbTrainingArtifactRepository` 的读取、恢复和计数方法。
3. `FmdbFeatureRepository`、`FmdbClusterRepository`、`FmdbColumnProfileRepository`、`FmdbStrategyRepository` 中从训练产物表恢复的物理读取分支。
4. `FmdbModelStore.loadDictionary` 及其对 `training_column_artifact` 的构造依赖。
5. `FmdbResultPersistenceVerifier` 中对 `training_column_artifact` 和 `training_example` 的训练成功校验。
6. `FmdbPersistenceConfig` 中模型表对训练列级产物表和列级产物开关的依赖。
7. 与上述读取恢复能力绑定的测试用例。

不建议第一步就删除 DDL 和枚举。更稳妥的落地方式是：

```text
先让三表默认关闭且主流程不读。
再删除恢复代码。
最后视历史兼容要求决定是否删除表结构。
```

这样既能快速降低代码复杂度，也避免一次性删除表结构带来发布和回滚风险。
