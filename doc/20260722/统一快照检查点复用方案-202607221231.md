# 统一快照检查点复用方案

## 1. 目标

将采样流程执行到 `CLUSTER` 后的完整上下文固化到统一表：

```text
dw.raha_snapshot_checkpoint
```

后续人工打标、训练都从该检查点恢复，不再重复执行：

```text
PROFILE
GENERATE_STRATEGY
RUN_STRATEGY
GENERATE_FEATURE
CLUSTER
```

这样可以避免把采样阶段所有候选原始行都保存进 `dw.raha_sample_record`。`dw.raha_sample_record` 继续只保存待人工标注的数据，`dw.raha_snapshot_checkpoint` 专门保存可复用的计算产物。

## 2. 核心结论

这个方案可行，而且比“把采样全集保存到 `dw.raha_sample_record`”更合适。

原因如下：

| 对比项 | 保存采样全集 | 统一检查点 |
| --- | --- | --- |
| 职责 | `sample_record` 同时承担标注和候选池职责 | `sample_record` 只服务标注 |
| 数据量 | 原始行全集进入标注表 | 只固化后续训练必要产物 |
| 复用粒度 | 还需要重新生成特征和聚类 | 可从 `CLUSTER` 后直接恢复 |
| 训练速度 | 仍可能重复跑重阶段 | 可跳过重阶段 |
| 表语义 | 容易混乱 | 清晰 |

但要注意：

检查点不能只保存“阶段已完成”状态，也不能只保存 `cluster_summary_json`。必须保存后续 `SAMPLE_TASK`、`LABEL`、`PROPAGATE`、`TRAIN` 所需的完整上下文。

## 3. 推荐流程

### 3.1 首次采样流程

```text
LOAD_DATA
PROFILE
GENERATE_STRATEGY
RUN_STRATEGY
GENERATE_FEATURE
CLUSTER
SAVE_SNAPSHOT_CHECKPOINT
SAMPLE_TASK
SAVE_SAMPLE_RECORD
EXPORT_ANNOTATION
```

说明：

1. `SAVE_SNAPSHOT_CHECKPOINT` 位于 `CLUSTER` 之后。
2. `SAMPLE_TASK` 从内存中的 `FEATURE_ASSEMBLY_RESULT` 和 `CLUSTERING_BATCH_RESULT` 继续执行。
3. `SAVE_SAMPLE_RECORD` 只保存待标注行。
4. 检查点保存完整特征、聚类和必要画像策略产物。

### 3.2 后续打标流程

人工打标仍然只面对 `dw.raha_sample_record` 中的待标注行，不需要感知检查点表。

```text
读取 sampleBatchId
导出待标注 Excel
人工标注
导入 dw.raha_annotation_record
```

### 3.3 训练复用流程

当训练请求满足“快照复用模式”时：

```text
LOAD_CHECKPOINT
LOAD_ANNOTATION
LABEL
PROPAGATE
TRAIN
PERSIST_RESULT
```

训练不再重新加载原始 SQL，也不再执行画像、策略、特征和聚类。

## 4. 触发条件

训练请求同时满足以下条件时，进入检查点复用模式：

| 字段 | 要求 |
| --- | --- |
| `sampleBatchId` | 非空 |
| `snapshotId` | 非空 |
| `datasetId` | 为空 |
| `sourceType` | 为空 |
| `sqlText` 或 `sql` | 为空 |
| `tableName` 或 `table` | 为空 |

还必须满足：

```text
sampleBatch.snapshotId == request.snapshotId
```

如果不一致，直接失败：

```text
SNAPSHOT_MISMATCH
```

如果检查点不存在，建议直接失败：

```text
CHECKPOINT_NOT_FOUND
```

不建议静默回退原流程。因为用户传入 `snapshotId` 且没有输入 SQL，表达的是“必须复用这个快照”，静默重跑会掩盖性能问题和数据一致性问题。

## 5. 表结构设计

统一表建议采用一张宽表，通过 `record_type` 区分不同产物。

```sql
CREATE TABLE dw.raha_snapshot_checkpoint (
    checkpoint_id STRING NOT NULL,
    dataset_id STRING NOT NULL,
    snapshot_id STRING NOT NULL,
    source_job_id STRING NOT NULL,
    sample_batch_id STRING,
    record_type STRING NOT NULL,
    record_scope STRING NOT NULL,
    column_name STRING,
    row_id STRING,
    cell_id STRING,
    cell_value STRING,
    artifact_version STRING,
    profile_json STRING,
    strategy_plan_json STRING,
    strategy_hit_json STRING,
    feature_dictionary_json STRING,
    feature_vector_json STRING,
    feature_summary_json STRING,
    cluster_version STRING,
    cluster_id STRING,
    cluster_distance DOUBLE,
    cluster_summary_json STRING,
    payload_json STRING,
    row_set_fingerprint STRING NOT NULL,
    config_fingerprint STRING NOT NULL,
    schema_hash STRING NOT NULL,
    source_version STRING,
    created_at BIGINT NOT NULL,
    partition_month STRING NOT NULL
)
USING ORC
PARTITIONED BY (dataset_id, partition_month)
TBLPROPERTIES ('raha.schema.version' = '2');
```

## 6. record_type 定义

| record_type | record_scope | 含义 | 是否训练必需 |
| --- | --- | --- | --- |
| `MANIFEST` | `CHECKPOINT` | 检查点元数据 | 是 |
| `PROFILE` | `COLUMN` | 列画像 | 建议保留 |
| `STRATEGY_PLAN` | `COLUMN` | 策略计划 | 是 |
| `STRATEGY_HIT` | `CELL` | 策略命中 | 可选 |
| `FEATURE_DICTIONARY` | `COLUMN` | 列特征字典 | 是 |
| `CELL_FEATURE` | `CELL` | 单元格稀疏特征 | 是 |
| `CLUSTER_SUMMARY` | `COLUMN` | 列聚类摘要 | 是 |
| `CLUSTER_ASSIGNMENT` | `CELL` | 单元格聚类归属 | 是 |

`STRATEGY_HIT` 对训练复用不是绝对必需，因为特征已经生成完毕。但为了审计、排查和未来从 `FEATURE` 前恢复，建议保留。

## 7. 关键字段说明

| 字段 | 说明 |
| --- | --- |
| `checkpoint_id` | 一次快照检查点的唯一标识 |
| `snapshot_id` | 输入数据快照 |
| `source_job_id` | 产生检查点的采样任务 |
| `sample_batch_id` | 采样批次，生成待标注行后回填或同步写入 |
| `record_type` | 产物类型 |
| `record_scope` | 产物粒度 |
| `row_set_fingerprint` | 行集合指纹 |
| `config_fingerprint` | 策略、特征、聚类配置指纹 |
| `schema_hash` | 字段结构指纹 |
| `feature_vector_json` | 单元格特征向量 |
| `cluster_id` | 单元格所属聚类 |
| `cluster_distance` | 单元格到聚类中心或代表点距离 |

## 8. 为什么不需要保存采样全集原始行

训练从 `CLUSTER` 后恢复时，需要的是：

1. 每个单元格的特征向量。
2. 每个单元格的聚类归属。
3. 标注行对应的人工标签。
4. 标签传播所需的聚类关系。
5. 模型训练所需的最终训练样本。

这些数据都可以从检查点表和标注表获得。

因此不需要把所有候选原始行放入 `dw.raha_sample_record`。

但检查点表仍需要保存单元格级别的 `row_id`、`column_name`、`cell_id`、`cell_value`、`feature_vector_json`、`cluster_id`。这不是“保存采样全集给人工标注”，而是“保存训练恢复所需的计算产物”。

## 9. 写入策略

### 9.1 写入时机

在 `ClusterStageHandler` 之后新增：

```text
SnapshotCheckpointStageHandler
```

该阶段读取上下文：

| 上下文键 | 来源 |
| --- | --- |
| `RAHA_DATASET` | `LOAD_DATA` |
| `DATASET_SNAPSHOT` | `LOAD_DATA` |
| `STRATEGY_PLANS` | `GENERATE_STRATEGY` |
| `STRATEGY_BATCH_RESULT` | `RUN_STRATEGY` |
| `STRATEGY_HITS` | `RUN_STRATEGY` |
| `FEATURE_ASSEMBLY_RESULT` | `GENERATE_FEATURE` |
| `CLUSTERING_BATCH_RESULT` | `CLUSTER` |

### 9.2 幂等键

推荐幂等键：

```text
dataset_id
snapshot_id
record_type
column_name
row_id
cell_id
artifact_version
config_fingerprint
```

FMDB 如果不支持高效更新，可以采用追加写入加读取最新版本的方式。

### 9.3 分区建议

使用：

```text
dataset_id
partition_month
```

读取时优先通过 `sampleBatchId` 定位 `partition_month`，再读取对应检查点，避免全表扫描。

## 10. 读取策略

新增仓储接口：

```java
public interface SnapshotCheckpointRepository {
    void save(SnapshotCheckpoint checkpoint);

    Optional<SnapshotCheckpointManifest> findManifest(
            String datasetId,
            String snapshotId,
            String sampleBatchId);

    SnapshotPreparedArtifacts loadPreparedArtifacts(
            SnapshotCheckpointManifest manifest);
}
```

恢复出的 `SnapshotPreparedArtifacts` 至少包含：

| 对象 | 用途 |
| --- | --- |
| `List<StrategyPlan>` | 训练产物血缘和模型元数据 |
| `FeatureAssemblyResult` | 模型训练特征输入 |
| `ClusteringBatchResult` | 标签传播输入 |
| `DatasetSnapshot` | 任务快照绑定 |

## 11. 工作流改造

### 11.1 采样工作流

当前：

```text
preparationStages
CLUSTER
SAMPLE_TASK
RESULT_PERSISTENCE
```

改为：

```text
preparationStages
CLUSTER
SNAPSHOT_CHECKPOINT
SAMPLE_TASK
RESULT_PERSISTENCE
```

### 11.2 训练工作流

普通训练保持不变。

检查点复用训练改为：

```text
RESTORE_SNAPSHOT_CHECKPOINT
DIRECT_LABEL
LABEL_PROPAGATION
MODEL_TRAINING
RESULT_PERSISTENCE
```

注意：

复用模式下不执行 `MERGE_TRAINING_INPUT`。否则会生成新的训练快照，破坏“训练数据等于采样快照”的前提。

## 12. UDF 参数约定

复用模式训练 SQL 参数示例：

```json
{
  "sampleBatchId": "sample_dw.person_info@20260721134516.455",
  "snapshotId": "snapshot_dw.person_info@20260721134516.455",
  "annotationDir": "/fmdb/raha/annotation-upload",
  "allowPartialAnnotation": "false",
  "modelNamePrefix": "person_info",
  "publishZip": "true",
  "requestId": "person-info-train-checkpoint-20260722",
  "forceRun": false
}
```

不要传：

```text
sourceType
datasetId
sqlText
sql
tableName
table
```

只要传了这些输入字段，就走原流程。

## 13. 校验规则

恢复检查点时必须校验：

| 校验 | 失败码 |
| --- | --- |
| `sampleBatchId` 存在 | `SAMPLE_NOT_FOUND` |
| 标注批次存在 | `MANUAL_ANNOTATION_NOT_FOUND` |
| 请求 `snapshotId` 等于采样批次 `snapshotId` | `SNAPSHOT_MISMATCH` |
| 检查点存在 | `CHECKPOINT_NOT_FOUND` |
| 检查点 `datasetId` 等于采样批次 `datasetId` | `CHECKPOINT_DATASET_MISMATCH` |
| 检查点 `rowSetFingerprint` 合法 | `CHECKPOINT_ROWSET_INVALID` |
| 检查点 `configFingerprint` 与当前配置一致 | `CHECKPOINT_CONFIG_MISMATCH` |
| 检查点包含特征字典 | `CHECKPOINT_FEATURE_MISSING` |
| 检查点包含聚类归属 | `CHECKPOINT_CLUSTER_MISSING` |

## 14. 与现有表的关系

| 表 | 新职责 |
| --- | --- |
| `dw.raha_sample_record` | 只保存待标注行 |
| `dw.raha_annotation_record` | 保存人工标注 |
| `dw.raha_snapshot_checkpoint` | 保存采样快照计算检查点 |
| `dw.raha_training_column_artifact` | 保存训练最终列级产物 |
| `dw.raha_training_cell` | 保存训练最终单元格产物 |
| `dw.raha_training_example` | 保存模型实际训练样本 |

检查点表不替代训练产物表。它解决的是“采样到训练之间的中间产物复用”。训练完成后，仍然要写训练产物表，保证模型发布、审计和预测链路稳定。

## 15. 性能收益

按当前链路，耗时主要集中在：

1. `PROFILE`
2. `RUN_STRATEGY`
3. `GENERATE_FEATURE`
4. `CLUSTER`

引入检查点后，训练阶段可跳过这些阶段。训练耗时会主要剩下：

1. 读取检查点。
2. 读取人工标注。
3. 标签传播。
4. 模型训练。
5. 训练产物落库。

对于 `person_info` 这类中小数据，收益主要是减少重复 Spark 作业启动和重复扫描。对于 5 万行以上数据，收益会更明显，尤其是特征生成和聚类阶段。

## 16. 实施步骤

1. 新增 `dw.raha_snapshot_checkpoint` schema。
2. 新增 `SnapshotCheckpointRepository` 接口和 FMDB 实现。
3. 新增 `SnapshotCheckpointStageHandler`，在采样 `CLUSTER` 后写入检查点。
4. 新增 `RestoreSnapshotCheckpointStageHandler`，在训练复用模式下恢复上下文。
5. 扩展 `TrainingRequestOptions`，增加 `requestedSnapshotId` 和 `reuseSnapshotCheckpoint`。
6. 修改 `RahaDetectionUdfService.train(...)`，识别“`snapshotId` 有值且输入为空”的复用模式。
7. 修改 `RahaTaskRequestFactory.training(...)`，把复用模式传入 `RahaTaskExecutionRequest`。
8. 修改 `TrainingWorkflow`，复用模式跳过 `preparationStages` 和 `MERGE_TRAINING_INPUT`。
9. 补单元测试，覆盖检查点存在、缺失、快照不一致、配置不一致。
10. 清库重建 FMDB 表后，在 Docker Spark 中只用 SQL 执行采样、训练、预测验证。

## 17. 推荐最终形态

最终训练有两种模式：

| 模式 | 触发方式 | 执行路径 |
| --- | --- | --- |
| 普通训练 | 传入 SQL 或表 | 重新加载并重新生成产物 |
| 检查点复用训练 | 只传 `sampleBatchId + snapshotId` | 读取 `dw.raha_snapshot_checkpoint`，从标签阶段开始 |

这套方案保留了原流程，也提供了明确的高性能路径。对于“采样、打标、训练使用同一份数据”的业务场景，默认应优先使用检查点复用训练。
