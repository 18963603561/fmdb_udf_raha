# ELKI 组件模型阶段优化分析

生成时间：2026-07-22 16:58

## 一、结论摘要

基于 2026-07-21 的阶段耗时、当前 2026-07-22 工程源码和 ELKI 0.8.0 官方文档，结论如下。

| 结论项 | 判断 |
|---|---|
| 是否有可直接使用的 ELKI 组件 | 有，主要集中在 `CLUSTER`，其次是新增 `RUN_STRATEGY` 中的 OD 策略 |
| 是否适合替换所有阶段 | 不适合，`LOAD_DATA`、`PROFILE`、`GENERATE_FEATURE`、`SAMPLE`、`LABEL`、`PROPAGATE`、`TRAIN` 没有明显可直接替换收益 |
| 当前最大收益点 | 先优化 `CLUSTER`，但要同时区分算法耗时、列并发和阶段复用 |
| 是否建议立即把 ELKI 打进生产 all jar | 暂不建议，需要先确认 AGPL 许可证影响并做隔离基准 |
| 最推荐路线 | 先做 `ElkiSphericalKMeansColumnClusterer` 与 `ElkiAnderbergAverageCosineColumnClusterer` 的 POC，再根据耗时和模型效果决定是否进入主链路 |

一句话判断：**ELKI 可以作为聚类后端候选，不应作为当前模型流水线的整体替换方案。**

## 二、输入耗时解读

用户提供的阶段耗时主要包含两段。

| 阶段 | 耗时 | 占比 | 初步定位 |
|---|---:|---:|---|
| `CLUSTER` | 157.315 秒 | 60.07% | 最大瓶颈，适合评估 ELKI 聚类后端 |
| `RUN_STRATEGY` | 31.786 秒 | 12.14% | 第二瓶颈，ELKI 可作为新增 OD 算法来源，但不一定更快 |
| `TRAIN` | 25.848 秒 | 7.28% | Spark MLlib 逻辑回归训练，不适合用 ELKI 替换 |
| `PROFILE` | 14.415 秒 | 5.50% | Spark 画像统计，不适合用 ELKI 替换 |
| `LOAD_DATA` | 8.786 秒 | 3.36% | FMDB/Spark 数据加载，不适合用 ELKI 替换 |
| `SAMPLE` | 3.430 秒 | 1.31% | 使用聚类结果做覆盖采样，耗时很小 |
| `GENERATE_FEATURE` | 2.304 秒 | 0.88% | Spark 与本地特征组装，ELKI 只能间接影响向量表示 |
| `PERSIST_RESULT` | 1.477 秒 | 0.56% | 持久化和产物发布，不适合用 ELKI 替换 |
| `PROPAGATE` | 1.197 秒 | 0.34% | 按簇多数或同质传播，耗时很小 |

| `GENERATE_STRATEGY` | 1.332 秒 | 0.51% | 策略计划生成，ELKI 只会增加新策略类型 |

注意：上表中的时间发生在 2026-07-21。当前源码已经有 2026-07-22 的 Smile 与 `AUTO` 聚类改造记录，默认配置也已经变为 `raha.clustering.provider=AUTO`、`raha.clustering.target-cluster-count=100`、`raha.clustering.max-sample-count=5000`。因此，ELKI 方案如果要比较收益，应使用 2026-07-22 当前代码重新跑一组同口径基准。

## 三、当前工程关键实现

当前聚类链路已经抽象出统一接口，ELKI 可以通过新增实现类接入。

| 文件 | 关键事实 |
|---|---|
| `src/main/java/com/fiberhome/ml/raha/cluster/algorithm/ColumnClusterer.java` | 单列聚类统一接口，ELKI 可以实现同一接口 |
| `src/main/java/com/fiberhome/ml/raha/cluster/algorithm/ClusteringProviderResolver.java` | 当前支持 `AUTO`、`HierarchicalColumnClusterer`、`ScalableColumnClusterer`、`SmileHierarchicalColumnClusterer` |
| `src/main/java/com/fiberhome/ml/raha/cluster/algorithm/SmileHierarchicalColumnClusterer.java` | 小样本使用 Smile 平均链接层次聚类，先把稀疏特征转为稠密矩阵 |
| `src/main/java/com/fiberhome/ml/raha/cluster/algorithm/ScalableColumnClusterer.java` | 大样本使用自研确定性余弦分区聚类，保持稀疏表示 |
| `src/main/java/com/fiberhome/ml/raha/job/stage/feature/ClusterStageHandler.java` | 独立 `CLUSTER` 阶段当前调用串行 `clusterAndSave` |
| `src/main/java/com/fiberhome/ml/raha/cluster/ColumnClusteringService.java` | 已经存在并行入口 `clusterAndSaveParallel`，但独立阶段没有使用 |
| `src/main/java/com/fiberhome/ml/raha/service/sample/RahaSampleService.java` | 采样服务内部已经使用并行聚类入口 |
| `src/main/java/com/fiberhome/ml/raha/service/train/RahaTrainService.java` | 训练服务内部在缺少聚类时也使用并行聚类入口 |
| `src/main/java/com/fiberhome/ml/raha/label/propagation/LabelPropagationService.java` | 标签传播只依赖 `clusterId`、`clusterVersion` 和成员列表 |
| `src/main/java/com/fiberhome/ml/raha/model/training/SparkMllibLogisticRegressionTrainer.java` | 训练使用 Spark MLlib 逻辑回归，不是 ELKI 擅长替换点 |

最重要的工程判断：当前独立 `CLUSTER` 阶段是串行调用，而工程内部已有并行聚类方法。即使不引入 ELKI，也应优先评估把 `ClusterStageHandler` 改为调用 `clusterAndSaveParallel`，并补充每列耗时日志。

## 四、ELKI 官方能力与约束

ELKI 0.8.0 官方发布页列出的 Maven 坐标是：

```xml
<dependency>
    <groupId>io.github.elki-project</groupId>
    <artifactId>elki</artifactId>
    <version>0.8.0</version>
</dependency>
```

ELKI 官方文档说明它可以通过 Java API 使用，也可以通过命令行使用；但官方更推荐松耦合方式，因为 Java API 仍在变化，嵌入式使用会带来内存和 API 稳定性问题。

更关键的是许可证：ELKI 是 AGPL 许可证。若把 ELKI 直接嵌入当前 UDF all jar 并分发到 FMDB/Spark 集群，需要先做许可证合规确认。若只把 ELKI 作为外部命令行进程运行，许可证限制相对不同，但这会增加数据导出、进程启动和结果回读成本，不适合每次 UDF 小任务高频调用。

## 五、阶段级适配结论

| 阶段 | 是否可直接用 ELKI | 可用组件 | 优化判断 | 建议 |
|---|---|---|---|---|
| `LOAD_DATA` | 否 | `DatabaseConnection`、解析器 | 与 FMDB/Spark 数据加载职责不同，引入 ELKI 会增加数据搬运 | 不接入 |
| `PROFILE` | 否 | 统计类或距离统计 | 当前是列画像、频率和分位数统计，Spark 更适合 | 不接入 |
| `GENERATE_STRATEGY` | 有限 | ELKI OD 算法枚举 | 可以新增策略类型，但不会减少生成计划耗时 | 仅作为新策略扩展 |
| `RUN_STRATEGY` | 有限 | `DBOutlierDetection`、`KNNOutlier`、`ParallelLOF`、`LOF` | 适合补充多维离群点特征，不适合作为当前简单 OD 的加速替代 | 可做实验策略 |
| `GENERATE_FEATURE` | 间接 | `SparseDoubleVector` | ELKI 可复用稀疏向量表示，但特征组装仍在本工程 | 不单独改造 |
| `CLUSTER` | 是 | `SphericalKMeans`、`SphericalSimplifiedHamerlyKMeans`、`Anderberg`、`NNChain`、`CutDendrogramByNumberOfClusters` | 最直接、最高价值，但需要基准验证和许可证确认 | POC 优先 |
| `SAMPLE` | 否 | 无直接组件 | 采样只是使用聚类结果评分，当前耗时低 | 只接受聚类结果变化 |
| `PERSIST_RESULT` | 否 | 无 | 持久化是项目仓储职责 | 不接入 |
| `LABEL` | 否 | 无 | 标签展开和读取是业务逻辑 | 不接入 |
| `PROPAGATE` | 否 | 无 | 当前按簇多数或同质传播，耗时低，ELKI 不提供直接替换收益 | 不接入 |
| `TRAIN` | 不建议 | `KNNClassifier`、`PriorProbabilityClassifier` | ELKI 分类器能力有限，当前 Spark MLlib 逻辑回归更合适 | 不替换 |

## 六、`CLUSTER` 阶段可直接使用的 ELKI 组件

### 6.1 精确层次聚类路径

候选组件：

| 目标 | ELKI 组件 |
|---|---|
| 余弦距离 | `CosineDistance` 或单位化后的 `CosineUnitlengthDistance` |
| 稀疏向量 | `SparseDoubleVector` |
| 平均链接层次聚类 | `Anderberg` 或 `NNChain` 配合 `AverageLinkage` |
| 按目标簇数切平层次树 | `CutDendrogramByNumberOfClusters` |

适配方式：

1. 新增 `ElkiAverageCosineColumnClusterer implements ColumnClusterer`。
2. 将 `SparseFeatureRow.values` 转为 `SparseDoubleVector` 或 `double[][]` 数据库。
3. 使用 ELKI 层次聚类生成层次结果。
4. 使用目标簇数切分为平面簇。
5. 按当前工程规则生成稳定 `cluster-0001`、`cluster-0002`。
6. 继续使用当前 `ClusterVersioner` 生成版本。

收益判断：

| 项 | 判断 |
|---|---|
| 语义贴合度 | 高，最接近当前 Smile 平均链接余弦聚类 |
| 性能收益 | 不确定，需要基准。层次聚类仍有平方级甚至更高内存和时间压力 |
| 适合默认生产 | 暂不适合 |
| 适合用途 | 质量对照、算法基准、少量字段小样本 |

这个路径“可直接使用”，但不一定是性能最优路径。它更适合回答“ELKI 与 Smile 结果谁更稳定”，不一定能解决 157 秒瓶颈。

### 6.2 近似余弦分区路径

候选组件：

| 目标 | ELKI 组件 |
|---|---|
| 固定目标簇数 | `SphericalKMeans` |
| 更快的球面 k-means | `SphericalSimplifiedHamerlyKMeans`、`SphericalHamerlyKMeans` |
| 初始化 | `SphericalKMeansPlusPlus` 或固定随机种子初始化 |
| 稀疏输入 | `SparseDoubleVector` |

适配方式：

1. 新增 `ElkiSphericalKMeansColumnClusterer implements ColumnClusterer`。
2. 对每列特征向量做单位化，保持与当前余弦距离语义接近。
3. 使用 `targetClusterCount` 作为 `k`。
4. 使用 `randomSeed` 创建确定性初始化。
5. 将 ELKI 结果映射回 `SparseFeatureRow.cellId`。
6. 保持当前簇编号稳定排序规则，避免采样和传播版本不可复现。

收益判断：

| 项 | 判断 |
|---|---|
| 语义贴合度 | 中高，适合当前“固定目标簇数”的工程配置 |
| 性能收益 | 比精确层次聚类更值得期待 |
| 适合默认生产 | 需要 POC 通过后再考虑 |
| 适合用途 | 替代当前 `ScalableColumnClusterer`，或作为 `AUTO` 大样本路径 |

这是最值得做的 ELKI POC。它匹配当前 `targetClusterCount` 配置，也比 DBSCAN、HDBSCAN 更符合 Raha 标签传播对固定簇数量的需求。

### 6.3 BIRCH 或 BETULA 路径

候选组件：

| 目标 | ELKI 组件 |
|---|---|
| 大数据预聚类 | `BIRCHLeafClustering` |
| BIRCH/BETULA 后续 k-means | `BetulaLloydKMeans` |

收益判断：

| 项 | 判断 |
|---|---|
| 语义贴合度 | 中，BIRCH 更偏欧氏空间和聚类特征树 |
| 性能收益 | 对大数据可能有价值 |
| 风险 | 与当前余弦特征、标签传播语义不完全一致 |
| 建议 | 作为第二批 POC，不作为首选 |

若使用 BIRCH 类组件，需要特别验证簇成员是否过粗，否则会放大错误标签传播风险。

### 6.4 DBSCAN、OPTICS、HDBSCAN 路径

这些组件虽然在 ELKI 中很成熟，但不建议作为当前默认路径。

原因：

1. 当前工程配置以 `targetClusterCount` 为核心，而 DBSCAN、OPTICS、HDBSCAN 需要密度参数。
2. Raha 标签传播依赖每列有稳定簇成员，密度算法可能产生噪声点或过多小簇。
3. 配置自由度增加会违背当前“少参数、可解释”的工程方向。

适合用途：离线探索字段结构或高噪声列，不适合作为默认 `CLUSTER` 后端。

## 七、`RUN_STRATEGY` 阶段可用的 ELKI OD 组件

当前工程启用的策略族主要是 `OD`、`PVD`，已有 OD 策略包括：

| 当前策略 | 文件 | 当前语义 |
|---|---|---|
| `LowFrequencyStrategy` | `strategy/impl/od/LowFrequencyStrategy.java` | 低频值异常 |
| `NumericDistanceStrategy` | `strategy/impl/od/NumericDistanceStrategy.java` | 数值标准差距离异常 |
| `QuantileOutlierStrategy` | `strategy/impl/od/QuantileOutlierStrategy.java` | 四分位距异常 |

ELKI 可补充的 OD 组件：

| ELKI 组件 | 可用方式 | 适合场景 |
|---|---|---|
| `DBOutlierDetection` | 按距离半径和比例判定异常 | 多维数值特征中的全局离群 |
| `KNNOutlier` | 使用第 k 近邻距离作为分数 | 局部稀疏区域异常 |
| `ParallelLOF` 或 `LOF` | 局部密度离群分数 | 多维上下文异常 |
| `KMeansOutlierDetection` | 基于簇距离生成异常分数 | 与聚类特征联动 |

但这些不应直接替代现有 OD 简单策略，原因如下。

1. 当前 OD 多数是单列统计规则，Spark groupBy 和分位数更直接。
2. ELKI 需要构造数据库、运行近邻查询、回填 `StrategyHit`，小数据下启动和转换成本可能高于收益。
3. `RUN_STRATEGY` 当前不是最大瓶颈，优先级低于 `CLUSTER`。
4. 新增 OD 策略会改变特征字典、模型版本和训练效果，需要完整回归。

建议：把 ELKI OD 作为“新增策略族或新增 OD 类型”做实验，不作为性能优化主线。

## 八、`TRAIN` 阶段不建议使用 ELKI 替换

当前训练阶段使用 Spark MLlib 逻辑回归：

| 当前实现 | 说明 |
|---|---|
| `SparkMllibLogisticRegressionTrainer` | 用 Spark MLlib 训练列级二分类逻辑回归 |
| `WeightedRuleFallbackTrainer` | 训练不可用时走规则模型降级 |
| `AdaptiveColumnModelTrainer` | 在 MLlib 与规则模型之间自适应选择 |

ELKI 官方分类包主要包含 `KNNClassifier` 和 `PriorProbabilityClassifier`。这与当前列级二分类模型需求不匹配。

不建议替换原因：

1. 当前 `TRAIN` 只有 25.848 秒，不是最大瓶颈。
2. Spark MLlib 适合当前 Spark/FMDB 环境和稀疏特征训练。
3. ELKI 分类器不提供当前所需的逻辑回归可解释系数、权重列、阈值和模型发布契约。
4. 替换后会影响模型兼容校验、预测器、模型存储和 UDF 输出字段。

因此，ELKI 对 `TRAIN` 的更合理作用是提供额外离群分数特征，而不是替换分类器。

## 九、许可证与打包风险

当前工程通过 `maven-shade-plugin` 打包 all jar，并已经 relocation 了 `smile`、`poi`、`zip4j` 等依赖。

若直接引入 ELKI，需要关注：

| 风险 | 说明 | 建议 |
|---|---|---|
| AGPL 许可证 | 嵌入 ELKI 并分发组合包可能影响当前代码授权要求 | 先做法务或开源合规确认 |
| API 稳定性 | 官方文档明确 Java API 在 1.0 前仍可能变化 | 用 POC 隔离，不直接绑定生产主流程 |
| 包体积 | `io.github.elki-project:elki:0.8.0` 是完整包 | POC 可先用完整包，生产再瘦身 |
| shade 资源 | ELKI 有服务发现和参数化机制 | 若使用 Parameterization API，需要验证 `META-INF/services` 处理 |
| 内存 | ELKI 官方也提示嵌入式使用要注意内存释放 | 每列运行后清理数据库对象，做堆内存压测 |
| Spark 环境 | ELKI 是单 JVM 数据挖掘库，不是 Spark 分布式算法 | 不要在 Executor 大量嵌入运行 |

如果许可证无法接受，直接嵌入式方案应停止。此时最多保留“外部命令行实验工具”，不进入 UDF 生产 jar。

## 十、推荐落地路线

### 10.1 先做非 ELKI 快速优化

优先级最高，不涉及新增依赖。

1. 将 `ClusterStageHandler` 从 `clusterAndSave` 改为 `clusterAndSaveParallel`。
2. 参数使用现有 `ResourceConfig.maxParallelColumns` 和 `stageTimeoutMillis`。
3. 补充每列聚类开始、结束、耗时、样本数、特征维度、算法名日志。
4. 在 `ClusteringBatchResult` 或阶段摘要里输出每列耗时。
5. 使用 2026-07-21 同口径输入重新压测，确认 157.315 秒中算法与调度各占多少。

这一步不影响聚类算法语义，收益和风险比优于直接换 ELKI。

### 10.2 第一批 ELKI POC

新增两个 provider，但默认不启用。

| provider | 用途 |
|---|---|
| `ElkiSphericalKMeansColumnClusterer` | 评估固定簇数余弦近似聚类性能 |
| `ElkiAnderbergAverageCosineColumnClusterer` | 评估 ELKI 层次聚类与 Smile 层次聚类差异 |

建议只在测试 profile 或实验分支中引入依赖：

```xml
<dependency>
    <groupId>io.github.elki-project</groupId>
    <artifactId>elki</artifactId>
    <version>0.8.0</version>
    <scope>test</scope>
</dependency>
```

通过后再决定是否调整 scope。

### 10.3 基准验证口径

每个 provider 至少比较以下指标。

| 指标 | 验收标准 |
|---|---|
| `CLUSTER` 阶段耗时 | 比当前 `AUTO` 路径降低至少 30% 才值得继续 |
| 最大堆内存 | 不高于当前 Smile 路径，或在可接受范围内 |
| `assignmentCount` | 不少于当前路径 |
| `effectiveClusterCount` | 接近 `targetClusterCount`，不能大量空簇 |
| 采样结果 | 抽样行覆盖率不能明显下降 |
| 标签传播 | 冲突簇比例不能明显上升 |
| 训练结果 | 已发布模型字段数不能下降 |
| 检测效果 | 在有真值数据集上 F1 不能下降，或下降需有业务解释 |
| 可复现性 | 同一输入、同一 seed 下 clusterVersion 和 clusterId 稳定 |

### 10.4 主链路接入条件

满足全部条件后才建议进入主链路。

1. 开源许可证确认可接受。
2. 2026-07-22 当前代码同口径基准证明有明显收益。
3. 聚类结果可复现。
4. 采样和训练指标不劣化。
5. all jar 打包、shade、Spark SQL UDF 加载全部通过。
6. 故障时能回退到 `AUTO` 当前路径。

## 十一、建议的代码改动边界

若进入 POC，建议新增文件而不是改动现有 Smile 实现。

| 文件 | 动作 |
|---|---|
| `pom.xml` | 实验 profile 引入 ELKI 依赖 |
| `ClusteringProviderResolver.java` | 增加 `ElkiSphericalKMeansColumnClusterer`、`ElkiAnderbergAverageCosineColumnClusterer` provider |
| `cluster/algorithm/ElkiSphericalKMeansColumnClusterer.java` | 新增 ELKI 球面 k-means 适配器 |
| `cluster/algorithm/ElkiAnderbergAverageCosineColumnClusterer.java` | 新增 ELKI 平均链接层次聚类适配器 |
| `cluster/algorithm/ElkiVectorAdapter.java` | 新增内部向量转换工具 |
| `ClusterStageHandler.java` | 独立阶段改用并行入口 |
| `ColumnClusteringService.java` | 增加每列耗时指标 |
| `src/test/java/...` | 增加 deterministic、空输入、单样本、全零特征、稀疏高维测试 |

新增类和关键方法必须使用中文注释，保持 UTF-8 无 BOM，符合当前项目注释约定。

## 十二、阶段最终建议

| 阶段 | 最终建议 |
|---|---|
| `LOAD_DATA` | 不使用 ELKI |
| `PROFILE` | 不使用 ELKI |
| `GENERATE_STRATEGY` | 可新增 ELKI OD 策略计划类型，但不优先 |
| `RUN_STRATEGY` | 可实验 `DBOutlierDetection`、`KNNOutlier`、`ParallelLOF`，定位为特征增强 |
| `GENERATE_FEATURE` | 不单独使用 ELKI，必要时给 ELKI 聚类适配 `SparseDoubleVector` |
| `CLUSTER` | 优先 POC `ElkiSphericalKMeansColumnClusterer`，其次 POC `ElkiAnderbergAverageCosineColumnClusterer` |
| `SAMPLE` | 不使用 ELKI，只消费聚类结果 |
| `PERSIST_RESULT` | 不使用 ELKI |
| `LABEL` | 不使用 ELKI |
| `PROPAGATE` | 不使用 ELKI |
| `TRAIN` | 不使用 ELKI 替换 Spark MLlib，最多消费 ELKI OD 产生的新特征 |

## 十三、最终判断

ELKI 不是当前模型阶段的全链路优化组件，而是一个候选算法库。它最值得用于 `CLUSTER` 阶段的实验性后端，尤其是球面 k-means 类组件；也可以用于 `RUN_STRATEGY` 阶段新增多维离群点策略。

当前不建议直接把 ELKI 嵌入生产 UDF all jar。更稳妥的做法是：

1. 先修正独立 `CLUSTER` 阶段的串行调用问题。
2. 再做 ELKI 聚类 provider POC。
3. 使用同口径数据比较耗时、采样、传播、训练和检测结果。
4. 许可证和打包验证通过后，再决定是否进入默认配置。
