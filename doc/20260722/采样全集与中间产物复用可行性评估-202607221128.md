# 采样全集与中间产物复用可行性评估

生成时间：2026-07-22 11:28

## 结论

你的判断是对的：如果后续希望训练、预测尽量复用采样阶段产物，仅保存“待标注数据”是不够的。

当前 `dw.raha_sample_record` 保存的是被选中的待标注行，也就是 C1 标注样本。它适合支撑人工标注和训练标签回放，但不足以表达“采样阶段处理过的全部候选数据”。

建议把 `dw.raha_sample_record` 扩展为同时保存两类行：

| 行类型 | 含义 | 是否进入标注 Excel | 是否进入训练 C1 优先合并 |
|---|---|---|---|
| `ANNOTATION_TASK` | 被采样算法选中的待标注行 | 是 | 是 |
| `CANDIDATE_POOL` | 本次采样输入全集或候选全集行 | 否 | 默认否 |

同时，PROFILE / RUN_STRATEGY / FEATURE / CLUSTER 的保存和复用**具备可行性**，但当前还没有完整实现“训练、预测按采样批次直接复用并跳过阶段”的工作流。

## 一、为什么需要保存采样全集

当前 `dw.raha_sample_record` 主要用于：

1. 导出标注 Excel。
2. 导入标注时校验行身份和原始行快照。
3. 训练时按 `sampleBatchId` 读取 C1 行。
4. 和 O1 原始表合并生成训练快照。

这要求它至少保存被标注任务选中的行。

但如果目标变成：

```text
采样阶段跑完整 PROFILE / RUN_STRATEGY / FEATURE / CLUSTER
训练阶段直接复用采样阶段的这些结果
```

那么训练阶段必须知道采样阶段公共流水线处理过哪些行。仅有 300 条待标注行，无法代表采样输入全集，例如 `select * from dw.person_info limit 400` 中全部 400 行。

如果只保存 300 条待标注行，后续最多能复用这 300 条的原始数据和标注，不能可靠复用 400 行上的策略命中、特征行、聚类成员关系。

## 二、建议增加的标记

建议给 `dw.raha_sample_record` 增加一个字段：

```sql
sample_record_role STRING NOT NULL
```

可选值建议：

| 值 | 说明 |
|---|---|
| `ANNOTATION_TASK` | 采样算法最终选中的待标注行 |
| `CANDIDATE_POOL` | 本次采样输入全集或候选全集行 |

为了支持后续更细的筛选，也可以增加：

```sql
annotation_required BOOLEAN NOT NULL
```

但如果只想增加一个标记，优先使用 `sample_record_role`，表达能力更清楚。

推荐语义：

```text
sample_record_role = ANNOTATION_TASK
-> 导出 Excel
-> 允许生成 annotation_task_id
-> 允许进入训练 C1 优先合并

sample_record_role = CANDIDATE_POOL
-> 不导出 Excel
-> 不要求 annotation_task_id
-> 用于复用画像、策略、特征、聚类、候选全集审计
```

## 三、表结构影响

当前 `dw.raha_sample_record` 的唯一写入键是：

```text
sample_batch_id + row_id
```

如果同一个 row 同时作为候选全集行和待标注行，不能插入两条记录，否则会被幂等键冲突跳过。

所以推荐做法不是保存两条，而是保存一条，并用字段表达角色：

```text
未被选中：CANDIDATE_POOL
被选中：ANNOTATION_TASK
```

这样主键仍然保持：

```text
sample_batch_id + row_id
```

优点是：

1. 不破坏现有全局唯一性。
2. 训练读取 C1 时只过滤 `ANNOTATION_TASK`。
3. 标注导出时只过滤 `ANNOTATION_TASK`。
4. 候选全集仍可按同一个 `sampleBatchId` 追溯。

## 四、需要改造的代码点

### 4.1 领域对象

需要改：

```text
SampleRecord
SampleBatch
SampleRecordService
```

`SampleRecord` 增加：

```java
private final String sampleRecordRole;
```

或者新增枚举：

```java
SampleRecordRole.ANNOTATION_TASK
SampleRecordRole.CANDIDATE_POOL
```

更推荐枚举，避免字符串散落。

### 4.2 FMDB 表结构

需要改：

```text
src/main/resources/db/fmdb/raha-fmdb-schema.sql
FmdbTableSchemas.SAMPLE_RECORD
FmdbSampleRecordRepository.toTableRecord
FmdbSampleRecordRepository.fromRow
```

新增字段：

```sql
sample_record_role STRING NOT NULL
```

如果需要兼容旧表，读取旧记录时可以默认：

```text
sample_record_role = ANNOTATION_TASK
```

### 4.3 标注导出

`findForAnnotation` 必须增加过滤：

```text
sample_record_role = 'ANNOTATION_TASK'
```

否则 Excel 会把候选全集都导出，标注量会爆炸。

### 4.4 训练输入合并

训练阶段读取 C1 时，也必须只读取：

```text
sample_record_role = 'ANNOTATION_TASK'
```

否则候选全集会被当成 C1 标注样本进入训练合并，导致训练数据语义错乱。

### 4.5 采样物化

当前 `SampleRecordService` 只物化 `sampling.getTasks()` 对应的行。

如果要保存全集，需要新增物化模式：

```text
1. 先保存本次采样输入全集为 CANDIDATE_POOL
2. 再把被选中的行角色标记为 ANNOTATION_TASK
```

由于当前表是追加写，不适合更新同一行角色，推荐在生成 `SampleBatch` 时一次性决定每行角色，然后一次性写入。

## 五、保存 PROFILE 的可行性

可行性：高。

当前工程已经有：

```text
ColumnProfileRepository
FmdbColumnProfileRepository
ColumnProfileService
dw.raha_training_column_artifact.profile_json
```

当前问题不是不能保存，而是训练、预测工作流没有按 `sampleBatchId` 查找并复用采样阶段 profile。

复用前提：

| 条件 | 要求 |
|---|---|
| `datasetId` | 必须一致 |
| `snapshotId` | 必须一致 |
| `schemaHash` | 必须一致 |
| `inputSqlHash` | 建议一致 |
| 采样全集行数 | 必须能覆盖训练或预测输入 |

如果训练输入是“采样全集 400 行”，可以复用采样 profile。

如果训练输入是“全量 5 万行”，不能复用 `limit 400` 的 profile，只能作为采样侧审计产物。

## 六、保存 RUN_STRATEGY 的可行性

可行性：中高。

当前工程已经有：

```text
StrategyRepository
FmdbStrategyRepository
StrategyPlanService
StrategyExecutionService
```

策略计划和策略命中理论上可以保存并复用。

风险点是策略命中绑定的是单元格：

```text
datasetId + snapshotId + rowId + columnName
```

所以复用必须保证运行时输入和采样阶段处理过的输入是同一批行。

如果训练输入变成全量 5 万行，复用 400 行策略命中会导致其余 4.96 万行没有策略特征。此时只能：

1. 只训练 400 行采样全集。
2. 或者对全量重新跑策略。
3. 或者引入增量策略执行，只对未覆盖行补跑。

第三种收益最大，但实现复杂度最高。

## 七、保存 FEATURE 的可行性

可行性：中。

当前工程已经有：

```text
FeatureRepository
FmdbFeatureRepository
FeatureService
dw.raha_training_column_artifact.feature_dictionary_json
dw.raha_training_cell.feature_vector_json
```

问题是 feature 不是完全独立产物，它依赖：

1. 策略计划版本。
2. 策略命中结果。
3. 列画像。
4. 特征配置。
5. 本次输入数据的频率类上下文。

尤其当前特征里有：

```text
context.column.frequency
context.column.frequency_ratio
context.column.frequency_bucket.rare
```

这些依赖本次输入全集。采样 400 行和训练 5 万行的频率特征一定不同。

因此：

| 场景 | 是否适合复用特征 |
|---|---|
| 训练仍使用采样全集 400 行 | 可以 |
| 训练使用全量 5 万行 | 不建议直接复用 |
| 预测使用同一采样快照 | 可以 |
| 预测使用新数据窗口 | 不建议直接复用 |

## 八、保存 CLUSTER 的可行性

可行性：中。

当前工程已经有：

```text
ClusterRepository
FmdbClusterRepository
ColumnClusteringService
dw.raha_training_column_artifact.cluster_summary_json
```

聚类 assignments 可以保存并复用，但限制更强：

1. 聚类必须基于同一份特征行。
2. `featureDictionaryVersion` 必须一致。
3. `clustering.provider`、`targetClusterCount`、`maxSampleCount`、`randomSeed` 必须一致。
4. 输入行集合必须一致。

如果采样保存的是 400 行聚类，训练是 5 万行，就不能直接复用。

如果你的目标是减少训练阶段重复聚类，最可行的是：

```text
训练输入也限定为采样全集
或
采样阶段直接按训练全集执行公共流水线
```

否则训练全量仍然需要重新聚类。

## 九、预测阶段复用的特殊问题

预测阶段目前不跑 CLUSTER，但会跑：

```text
LOAD_DATA
PROFILE
GENERATE_STRATEGY
RUN_STRATEGY
GENERATE_FEATURE
PREDICT
```

要让预测复用采样或训练产物，比训练更敏感。

原因是预测输入可能是新数据、新窗口、新行集合。只要预测数据和训练数据不同，直接复用旧 feature 就会漏掉新行。

更合理的预测优化不是复用采样产物，而是：

1. 复用训练时发布的策略计划和特征字典。
2. 预测只生成当前输入的 feature rows。
3. 不重新生成策略计划。
4. 尽量减少 profile 对策略计划的依赖。

也就是说，预测优化方向应该是“冻结计划与字典”，不是“读取采样阶段 feature rows”。

## 十、推荐分阶段实施方案

### 第一阶段：扩展 sample_record 角色标记

目标：

```text
同一个 sampleBatchId 下同时保存候选全集和待标注行。
```

改造内容：

1. 增加 `sample_record_role`。
2. 标注导出过滤 `ANNOTATION_TASK`。
3. 训练 C1 合并过滤 `ANNOTATION_TASK`。
4. 采样物化时保存全集行，并标记被选中行。

收益：

1. 可以审计采样全集。
2. 为后续公共产物复用提供行集合基础。
3. 不改变现有人工标注体验。

### 第二阶段：复用 PROFILE 和 STRATEGY_PLAN

目标：

```text
相同 datasetId + snapshotId + configFingerprint 时，训练跳过画像和策略计划生成。
```

这两个产物体积小，复用风险相对低，适合作为第一批复用目标。

### 第三阶段：复用 RUN_STRATEGY 和 FEATURE

目标：

```text
输入行集合完全一致时，训练跳过策略执行和特征生成。
```

需要新增产物索引：

```text
artifact_scope = SAMPLE
sample_batch_id
snapshot_id
feature_dictionary_version
strategy_plan_version
row_set_fingerprint
```

### 第四阶段：复用 CLUSTER

目标：

```text
输入行集合、特征字典和聚类配置完全一致时，训练跳过聚类。
```

这是收益最大但约束也最强的一步。

如果训练全量和采样全集不同，则不建议强行复用采样聚类。

## 十一、推荐新增的产物复用键

建议后续所有可复用产物至少绑定：

```text
datasetId
snapshotId
schemaHash
inputReference
inputSqlHash
rowIdentityConfig
rowSetFingerprint
executionConfigFingerprint
artifactType
artifactVersion
sampleBatchId
```

其中 `rowSetFingerprint` 非常关键，用于判断两次任务处理的行集合是否完全一致。

没有这个字段时，很容易误把 `limit 400` 的产物复用到 5 万行训练。

## 十二、最终建议

短期建议：

1. 可以扩展 `dw.raha_sample_record`，增加 `sample_record_role`。
2. 保存采样输入全集时，把未选中行标为 `CANDIDATE_POOL`，选中行标为 `ANNOTATION_TASK`。
3. 标注导出和训练 C1 合并必须只使用 `ANNOTATION_TASK`。

中期建议：

1. 先复用 PROFILE 和 STRATEGY_PLAN。
2. 再复用 RUN_STRATEGY 和 FEATURE。
3. 最后复用 CLUSTER。

关键边界：

```text
只有当训练或预测输入行集合与采样阶段行集合一致时，才能直接复用 RUN_STRATEGY / FEATURE / CLUSTER。
```

如果训练是 5 万全量，而采样是 400 行，那么采样阶段的 FEATURE 和 CLUSTER 不能直接复用到全量训练。

对当前 person_info 场景，最稳妥的路线是：

```text
先保存采样全集 + 角色标记
再增加产物复用索引
最后让训练在 rowSetFingerprint 一致时跳过公共阶段
```
