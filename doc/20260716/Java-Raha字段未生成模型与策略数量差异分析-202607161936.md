# Java Raha 字段未生成模型与策略数量差异分析

## 一、结论摘要

本次 `flights` 复测中，Java 并不是因为“总标签数不足”而少生成两个模型，而是因为 `src` 和 `flight` 两个字段的字段级标签只有负类，没有正类。Java 要求一个字段至少同时存在正、负两类标签且具有有效特征，才允许训练二分类模型；因此这两个字段被判定为 `SINGLE_CLASS` 并跳过。

20 个采样元组共生成 120 个直接标签，数量上并不少，但标签按字段分布后并不均衡。`src` 和 `flight` 各有 20 个负标签、0 个正标签，传播标签也没有使它们达到双类别可训练条件。最终 Java 生成 4 个候选模型，跳过 2 个字段，训练任务仍返回 `SUCCEEDED`。

不生成模型在当前数据集上没有造成真实错误漏检，因为 `src` 和 `flight` 的完整真值本身就是 0 个错误；但这不等于没有风险。当前检测服务会把这两个预期跳过字段当成“没有已发布模型”的执行错误，任务返回 `PARTIAL_SUCCESS`，并且只产生 9504 条评分结果，而不是 6 个业务字段应有的 14256 条结果。若后续数据漂移导致这两个字段出现错误，当前模型链路无法检测这些错误。

Java 的 60 个策略与 Python 的 308 个策略不是同一种计数口径。Python 使用 34 组全表 dBoost 参数、232 个“字段与实际字符”组合和 42 个有向字段对；Java 使用 6 个字段级低频策略、24 个字段级聚合模式策略和 30 个业务字段有向关系策略。策略数少不代表最终特征一定少，也不能直接推出检测效果更差。

本次实际结果中，Java 的 F1 为 0.831625，Python 的 F1 为 0.823194。由于双方主动采样的 20 行交集为 0，不能把该差异归因于策略数量。要回答某类策略是否有效，必须先使用完全相同的冻结直接标签进行对照实验。

## 三、需要区分的四种“没有模型”

“没有模型”不能只用一个状态表达，至少应区分以下四类：

| 分类 | 含义 | 是否异常 | 当前例子 |
| --- | --- | --- | --- |
| 不参与训练 | 字段被定义为行号、不可检测字段或被配置排除 | 否 | `tuple_id` |
| 训练数据不可用 | 标签或特征不满足二分类训练条件 | 通常是预期跳过 | `src`、`flight` |
| 训练执行失败 | 分类器、Spark 作业、超时、模型保存或质量门禁失败 | 是 | 本次没有 |
| 检测时模型不可用 | 模型未发布、已停用、文件缺失或版本不兼容 | 是或发布流程未完成 | 当前检测中的 `src`、`flight` 被错误归入此类 |

当前主要问题不是 Java 坚持双类别训练，而是训练阶段知道字段属于“预期跳过”，检测阶段却丢失了这个语义，把它重新解释成模型加载错误。

## 四、字段不生成模型的完整条件

### 4.1 字段不进入训练范围

以下字段不会进入列级模型训练循环：

1. 行号字段或 `detectable=false` 的字段。
2. 未包含在任务 `includedColumns` 中的字段。
3. 被任务 `excludedColumns` 排除的字段。
4. 没有生成特征字典的字段。

这类字段属于 `NOT_APPLICABLE`，不应计入模型覆盖率分母，也不应在检测阶段报错。当前 `tuple_id` 是稳定行号，Java 将其排除是正确设计。

### 4.2 字段训练数据不可用

进入训练范围后，`ColumnTrainingDataBuilder` 可能产生以下不可训练状态：

| 状态 | 触发条件 | 结果 |
| --- | --- | --- |
| `EMPTY_FEATURES` | 特征字典没有任何有效特征定义 | 不调用有效分类训练，字段跳过 |
| `NO_LABELS` | 没有标签能与该字段的特征行关联 | 字段跳过 |
| `LABEL_CONFLICT` | 同一单元格的同优先级标签冲突，冲突标签剔除后没有剩余样本 | 字段跳过 |
| `SINGLE_CLASS` | 只有正标签或只有负标签 | 字段跳过 |
| `TRAINABLE` | 同时存在正、负标签并具有有效特征 | 进入模型训练 |

直接标签优先于传播标签。传播训练集不可用，或者正例比例小于 5% 或大于 95% 时，Java 会尝试回退到直接标签；只有直接标签本身达到 `TRAINABLE`，才会替换传播训练集。如果传播标签和直接标签都是单类别，字段仍不会生成模型。

需要特别说明：当前实现没有“最少正样本数”和“最少负样本数”的统计门槛。理论上 1 个正样本加 19 个负样本也会进入训练，虽然技术上可训练，但统计可靠性可能很弱。因此“能生成模型”和“模型值得发布”仍是两个问题。

### 4.3 训练器或质量门禁失败

即使训练数据是 `TRAINABLE`，以下情况仍不会形成候选模型：

1. Spark MLlib 逻辑回归训练异常，并且 `raha.model.fallback-enabled=false`。
2. 配置了尚未接入的分类器，例如当前未接入的 `DECISION_TREE` 或 `GBT`，同时禁止规则降级。
3. 字段训练达到阶段超时，或并行训练工作项异常。
4. 模型质量门禁拒绝退化模型。
5. 模型文件保存失败，或候选模型元数据写入失败。

当前质量门禁默认开启，主要拒绝以下模型：

| 门禁 | 默认值 | 拒绝条件 |
| --- | ---: | --- |
| 分数标准差 | `0.000001` | 模型对样本几乎输出恒定分数 |
| 预测正例比例 | 最大 `0.98` | 正例比例小于等于 0.02 或大于等于 0.98 |
| 训练集 F1 | 最小 `0.0` | 低于配置阈值 |

因此，一个字段即使同时有正、负标签，也可能因为模型输出退化而不生成候选模型。

### 4.4 已训练但检测时不可用

已生成模型文件不代表检测一定可用。检测只加载 `PUBLISHED` 状态的模型，并且同时校验：

1. 模型元数据与模型文件中的模型版本、字段、分类器和特征字典一致。
2. 当前数据模式哈希与训练时一致。
3. 当前特征字典版本与训练时一致。
4. 当前策略计划版本与训练时一致。
5. 模型文件路径存在且能够加载。

候选模型未发布、模型已停用、模型文件缺失，或者任一兼容性版本变化，都会表现为“当前字段没有可用模型”。这类情况不能与 `SINGLE_CLASS` 等预期跳过混在一起。

## 五、本次 flights 为什么只有 4 个模型

本次 Java 主动采样了 20 个元组，每个元组标注 6 个业务字段，共 120 个直接标签。按字段重新核验脏数据和干净数据后，标签分布如下：

| 字段 | 直接正标签 | 直接负标签 | 是否双类别 |
| --- | ---: | ---: | --- |
| `src` | 0 | 20 | 否 |
| `flight` | 0 | 20 | 否 |
| `sched_dep_time` | 9 | 11 | 是 |
| `act_dep_time` | 13 | 7 | 是 |
| `sched_arr_time` | 10 | 10 | 是 |
| `act_arr_time` | 12 | 8 | 是 |

完整真值中，`src` 和 `flight` 也都是 0 个错误。因此，无论继续随机采样多少行，这个特定数据集都不可能为这两个字段提供真实正标签。对这类字段无限增加采样预算没有意义，应该使用明确的“已观测单类别正常基线”状态，而不是反复重试训练。

训练日志最终记录：

```text
status=SUCCEEDED，candidateCount=4，skippedCount=2，failedCount=0
```

这说明两个字段属于跳过而非训练失败。四个时间字段均生成并发布了模型，`src` 和 `flight` 没有候选模型。

## 六、不生成模型的实际影响

### 6.1 对训练任务状态的影响

当前训练状态规则为：

1. 所有字段都没有候选模型时，任务返回 `FAILED`，错误码为 `NO_CANDIDATE_MODEL`。
2. 至少有一个候选模型，且只有不可训练字段被跳过时，任务仍返回 `SUCCEEDED`。
3. 至少有一个候选模型，但部分字段训练失败或部分策略失败时，任务返回 `PARTIAL_SUCCESS`。

因此，本次训练返回成功是符合现有实现的，但成功状态本身不能表达只有 4/6 的字段生成模型。

### 6.2 对检测任务状态的影响

检测阶段仍会遍历所有具有特征字典的字段。`src` 和 `flight` 没有已发布模型，因此各抛出一次：

```text
java.lang.IllegalStateException: 当前字段没有已发布模型
```

最终检测任务返回：

```text
status=PARTIAL_SUCCESS，errorCode=PARTIAL_DETECTION_FAILURE，failedColumnCount=2
```

这会产生三个问题：

1. 预期跳过被记成错误日志，污染监控、告警和失败率统计。
2. 操作人员无法区分“该字段确认只有一个类别”与“模型文件意外丢失”。
3. 每次检测都重复进入已知失败路径，产生无意义的并行任务和异常堆栈。

### 6.3 对检测覆盖率和效果的影响

本次有 2376 行、6 个业务字段，理论评分单元格为 14256。实际只对 4 个有模型字段生成 9504 条结果：

| 指标 | 数值 |
| --- | ---: |
| 业务字段数 | 6 |
| 有模型字段数 | 4 |
| 字段模型覆盖率 | 66.67% |
| 理论评分单元格 | 14256 |
| 实际评分单元格 | 9504 |
| 单元格评分覆盖率 | 66.67% |

当前 `src` 和 `flight` 没有真实错误，所以总体 TP、FP、FN 没有受到影响。但是这是一种数据集特例，不应据此认为缺失模型永远安全。后续批次一旦在这两个字段产生错误，缺失模型会形成未评分区域；若评估程序把未评分默认当成“正确”，这些错误会全部变成漏检。

### 6.4 对模型治理的影响

没有模型意味着该字段缺少以下能力：

1. 没有可发布、回滚和版本追踪的字段模型。
2. 没有字段阈值和分数分布。
3. 无法监控模型漂移和预测覆盖变化。
4. 无法证明“没有检出”是模型判断为正常，还是字段根本未执行。

因此，输出层必须把“预测为正常”和“未评分”明确区分。

## 七、Python 对单类别字段的处理差异

Python 参考实现不会为单类别字段训练分类器对象，而是直接生成常量预测：

1. 训练标签全部为 1 时，整列预测为 1。
2. 训练标签全部为 0，或者字段没有有效特征时，整列预测为 0。
3. 同时有正、负标签且有特征时，才训练配置的分类器。

所以本次 `src` 和 `flight` 在 Python 中同样没有真正训练分类器，只是通过全 0 预测完成整列处理。Java 则明确拒绝创建单类别模型，并在检测时缺少评分结果。

Python 参考实现还存在一个边界问题：当 `y_train` 为空时，`sum(y_train) == len(y_train)` 仍为 `0 == 0`，代码会进入“全部为错误”的分支，把整列预测为 1。本次每个字段都有 20 个直接标签，没有触发该问题，但后续对齐测试应加入零标签用例。

从生产语义看，Java 不伪造二分类模型更稳妥；但应补充明确的常量基线或跳过状态，并避免把预期状态转换为检测异常。

## 八、Java 60 个策略与 Python 308 个策略的组成

### 8.1 总体拆分

| 策略族 | Python | Java | 数量差 | 主要原因 |
| --- | ---: | ---: | ---: | --- |
| OD | 34 | 6 | 28 | Python 枚举 dBoost 参数，Java 按字段画像生成 Spark 原生策略 |
| PVD | 232 | 24 | 208 | Python 每个实际字符一个策略，Java 每字段四类聚合策略 |
| RVD | 42 | 30 | 12 | Python 包含 `tuple_id`，Java 排除行号字段 |
| 合计 | 308 | 60 | 248 | 生成粒度和适用字段不同 |

### 8.2 OD 差异

Python 的 34 个 OD 由固定参数网格组成：

| Python OD 类型 | 参数组合 | 数量 |
| --- | --- | ---: |
| `histogram` | 5 个参数值乘 5 个参数值 | 25 |
| `gaussian` | 9 个标准差参数 | 9 |
| 合计 | 全表 dBoost 配置 | 34 |

每一组配置都对整个数据表运行 dBoost，可能同时输出多个字段的候选坐标，因此这 34 个不是“每列 34 个”。

Java 的 OD 按字段画像生成，支持以下类型：

1. `OD_LOW_FREQUENCY`：按字段值频率识别低频值。
2. `OD_NUMERIC_DISTANCE`：数值字段基于均值和标准差识别离群值。
3. `OD_QUANTILE`：数值字段基于四分位距识别离群值。

`flights` 的 6 个业务字段均按字符串读取，没有字段满足数值策略生成条件，因此每字段只生成 1 个低频策略，共 6 个。Java OD 是 Spark 原生近似实现，不是 dBoost 的逐参数等价移植，不能把 Python 的某一条 OD 与 Java 某一条 OD 直接一一配对。

### 8.3 PVD 差异

Python 为每个字段中实际出现过的每一个字符生成一条 PVD 配置。本次 232 条分布如下：

| 字段 | 不同字符数及 PVD 数 |
| --- | ---: |
| `tuple_id` | 10 |
| `src` | 29 |
| `flight` | 35 |
| `sched_dep_time` | 35 |
| `act_dep_time` | 42 |
| `sched_arr_time` | 35 |
| `act_arr_time` | 46 |
| 合计 | 232 |

Python PVD 的实际含义是“值中是否包含指定字符”。它为分类器提供细粒度字符存在特征，但单条策略本身并不判断字符是否真的异常。实现使用字符拼接正则字符类，特殊字符没有统一转义，部分字符可能改变正则含义或触发异常后被跳过。

Java 对每个业务字段生成以下 4 类 PVD：

1. `PVD_CHARACTER_SET`：把数字、拉丁字母、中文、空格和符号组合成字符签名，检测少数签名。
2. `PVD_LENGTH`：根据长度分布和四分位距检测异常长度。
3. `PVD_NULL_PLACEHOLDER`：检测空值、空白和配置的占位符。
4. `PVD_TYPE_FORMAT`：检测字段类型和格式少数模式。

6 个业务字段乘 4 类等于 24。Java 用聚合语义降低策略数量和计算量，但失去了 Python 的逐字符粒度，因此当前对比状态被标记为“缺少 Python 已观察字符动态网格”。

### 8.4 RVD 差异

Python 对 7 个字段枚举所有有向字段对：

```text
7 × 6 = 42
```

Java 把 `tuple_id` 排除，只对 6 个业务字段枚举：

```text
6 × 5 = 30
```

Python 多出的 12 个字段对全部涉及 `tuple_id`。稳定唯一行号几乎会为每一行建立唯一关系，不应作为错误检测特征，Java 排除它是合理的。

对双方共有的 30 个业务字段对逐坐标比较后：

| 对比项 | 结果 |
| --- | ---: |
| 坐标完全一致的字段对 | 11 |
| Java 为 Python 子集的字段对 | 19 |
| Java 独有坐标 | 0 |
| Python 独有坐标出现次数 | 11768 |
| Python 独有的字段对与行组合 | 5884 |
| Python 独有坐标中右值为空的比例 | 100% |

根因已经通过脏数据逐坐标核验：

1. 双方都用非空左值建立“左值对应多个右值”的冲突关系。
2. Java 在关系计算前过滤左右任一侧为空的行，所以不会输出右值为空的行。
3. Python 建立冲突关系时忽略空右值，但只要该左值在其他行形成一对多冲突，输出阶段仍会把当前行的左、右单元格都标记，即使当前右值为空。

因此 19 个近似字段对不是随机误差，而是明确的空右值输出语义差异。是否对齐应由业务定义决定：如果空值已由 PVD 负责，保留 Java 逻辑更清晰；如果要求完全复现 Python，则 Java 应在判定左值冲突后把同一左值对应的空右值行也输出。

## 九、为什么不能用 308 大于 60 判断优劣

策略数量只是策略画像数量，不等于有效特征数量、模型容量或最终效果，原因如下：

1. Python 一条字符 PVD 只表示一个字符是否出现，232 条中可能有大量常量或高度相关特征。
2. Python 在生成列特征后会删除整列恒定的策略特征，所以 308 条策略不一定全部进入每个字段分类器。
3. Java 一条聚合 PVD 可以编码一类模式分布，语义范围可能覆盖多个 Python 字符特征。
4. Java 还会生成上下文特征，本次最终特征数为 157，不能直接用 60 代替最终特征维度。
5. OD 的运行粒度不同，Python 34 条是全表参数组合，Java 6 条是字段级计划。
6. RVD 的 12 条数量差来自正确排除行号，不属于能力缺失。

本次完整系统效果为：

| 实现 | 精确率 | 召回率 | F1 | 策略数 |
| --- | ---: | ---: | ---: | ---: |
| Python | 0.938818 | 0.732927 | 0.823194 | 308 |
| Java | 0.807427 | 0.857317 | 0.831625 | 60 |

Java 策略更少但 F1 略高，同时精确率明显更低、召回率更高。这说明策略数量与最终指标没有简单的单调关系。更重要的是，两边的 20 个主动采样行交集为 0，标签传播数量也不同，因此当前指标只能评价各自完整流程，不能证明 60 条策略优于或劣于 308 条策略。

## 十、当前工程存在的主要问题

### 10.1 无模型状态没有跨阶段持久化

训练阶段正确区分 `SINGLE_CLASS`、`NO_LABELS`、`EMPTY_FEATURES` 和 `LABEL_CONFLICT`，但检测阶段只查询已发布模型。未找到模型时统一抛出异常，导致预期跳过与生产故障无法区分。

### 10.2 任务成功状态没有体现字段覆盖率

本次训练返回 `SUCCEEDED`，但只有 4/6 字段有模型。调用方若只检查任务状态，会误以为所有字段都已完成训练。

### 10.3 缺少单类别字段的生产处理策略

对真实无错误字段，继续采样无法制造正例。工程目前既没有显式常量正常基线，也没有带有效期的“单类别已确认”状态，最终只能在检测时失败。

### 10.4 可训练门槛过低

当前只要求同时出现两个类别，没有配置最少正样本、最少负样本和最少直接标签比例。极少数正样本也可能进入逻辑回归，模型稳定性依赖后续质量门禁，但现有门禁主要检查训练集分数分布，不足以替代独立验证。

### 10.5 Java 与 Python 策略语义仍未完全对齐

主要剩余差异是 PVD 的逐字符粒度、OD 的 dBoost 参数语义和 RVD 对空右值的处理。当前已有数量和坐标证据，但尚未在同一冻结标签下量化每项差异对最终模型的单独贡献。

### 10.6 Python 参考实现存在零标签边界缺陷

Python 对空标签列表会误判为“全部正类”。如果继续把 Python 作为对齐基准，必须增加零标签测试，否则可能把参考实现缺陷当作 Java 应模仿的行为。

## 十一、建议的落地调整

### 11.1 第一优先级：持久化字段模型处置状态

为每个业务字段保存独立状态，至少包括：

| 建议状态 | 含义 | 检测处理 |
| --- | --- | --- |
| `NOT_APPLICABLE` | 行号或明确排除字段 | 不进入检测，不计覆盖率分母 |
| `SKIPPED_NO_LABELS` | 没有可关联标签 | 正常跳过并告警标签覆盖不足 |
| `SKIPPED_SINGLE_CLASS` | 只有一个标签类别 | 使用配置的基线策略或标记未评分 |
| `SKIPPED_EMPTY_FEATURES` | 没有有效特征 | 正常跳过并告警特征覆盖不足 |
| `SKIPPED_LABEL_CONFLICT` | 标签冲突后无样本 | 阻止自动发布并要求处理标签 |
| `FAILED_TRAINING` | 分类器或保存失败 | 检测任务按异常处理 |
| `PUBLISHED` | 已发布且兼容 | 正常评分 |
| `UNAVAILABLE_INCOMPATIBLE` | 有模型但版本不兼容 | 检测任务按异常处理 |

检测开始前先执行模型可用性预检。预期跳过字段不要提交给并行模型加载器，不打印错误堆栈；意外缺失、文件损坏和版本不兼容仍必须返回失败。

### 11.2 第一优先级：输出模型覆盖率和评分覆盖率

训练、检测和评估结果至少补充：

1. 期望训练字段数。
2. 已发布模型字段数。
3. 按原因跳过的字段清单。
4. 模型字段覆盖率。
5. 期望评分单元格数和实际评分单元格数。
6. 未评分单元格数及原因。

总体 F1 必须与覆盖率同时展示，禁止把未评分单元格静默当成预测正常。

### 11.3 第一优先级：增加字段级采样充足性门槛

建议新增可配置项：

```properties
raha.model.minimum-positive-labels=5
raha.model.minimum-negative-labels=5
raha.model.minimum-direct-labels=10
raha.sampling.maximum-field-resample-rounds=2
```

首轮采样后，对样本不足但可能存在错误的字段执行定向补采样，优先选择稀有模式、策略分歧大和低置信度元组。达到最大补采样轮数仍无正例时，转为 `SKIPPED_SINGLE_CLASS`，不要无限扩大预算。

具体阈值不能直接以 5、5 作为生产定值，应在多个数据集上通过学习曲线确定；上述数值只适合作为初始实验配置。

### 11.4 第二优先级：为真实单类别字段定义显式基线

可提供两种模式，由业务配置选择：

1. `UNSCORED`：字段明确标记为未评分，不生成检测结果，适合高风险字段。
2. `CONSTANT_BASELINE`：全负标签生成常量正常基线，全正标签生成常量异常基线；结果必须标记为基线而不是机器学习模型。

常量正常基线应设置有效期、数据分布漂移监控和重新采样条件。字段字符集、空值率、频率分布或策略命中分布发生明显变化时，必须失效并重新采样。

### 11.5 第二优先级：明确 RVD 空值业务语义

先确定以下二选一规则，再修改实现：

1. 保持 Java 规则：RVD 只负责非空关系冲突，空值由 PVD 负责。此时把 19 个近似结果记录为有意差异。
2. 对齐 Python 规则：非空右值形成一对多冲突后，同一左值对应的空右值行也输出左右坐标。应增加配置开关和回归测试，避免直接改变生产结果。

不建议仅为了让策略数量或坐标完全一致而默认扩大 RVD 命中范围，因为 Python 独有的 11768 个坐标全部与空右值有关，可能显著提高候选密度和误报风险。

### 11.6 第二优先级：按收益补充字符级 PVD

如果冻结标签实验确认 Python 字符特征有独立收益，可在 Java 中新增受控的“已观察字符”特征：

1. 只保留达到最小支持度且具有区分度的字符。
2. 设置每字段最大字符特征数，避免高基数字段膨胀。
3. 使用字面量匹配，不拼接未转义正则表达式。
4. 排除或单独编码控制字符、代理字符和不可见字符。
5. 保留现有字符签名、长度、占位符和类型格式特征，不直接替换。

### 11.7 第三优先级：以冻结标签评估 OD 对齐价值

不要先把 34 组 dBoost 配置全部机械移植到 Java。应在同一批直接标签、同一评估集下分别运行：

1. 当前 Java OD。
2. 当前 Java OD 加直方图参数网格。
3. 当前 Java OD 加高斯距离参数网格。
4. 不使用 OD 的基线。

只有当新增策略对字段级精确率、召回率或平均精确率有稳定增益时才进入生产。否则 34 变 6 只是实现压缩，不是功能缺陷。

### 11.8 第三优先级：修复 Python 零标签参考行为

Python `predict_labels` 应先判断 `len(y_train) == 0`，并显式返回未评分或全 0 基线，不能利用空列表满足“全部为 1”的条件。该修改属于参考基准修正，应单独记录，避免影响历史结果解释。

## 十二、建议验收标准

完成调整后至少满足以下验收项：

1. `src` 和 `flight` 在训练输出中明确记录为 `SKIPPED_SINGLE_CLASS`。
2. 检测阶段不再为预期跳过字段打印异常堆栈，也不因此返回 `PARTIAL_SUCCESS`。
3. 检测结果明确报告 4/6 模型覆盖和 9504/14256 评分覆盖，或在启用常量基线时报告基线覆盖。
4. 所有字段无模型时，训练仍正确返回 `NO_CANDIDATE_MODEL`，除非业务明确启用常量基线模式。
5. 模型文件缺失、未发布和版本不兼容仍返回失败，不能被预期跳过逻辑吞掉。
6. 增加 `NO_LABELS`、空标签、全负、全正、标签冲突、空特征、质量门禁失败和模型不兼容的自动化测试。
7. 增加 RVD 空右值用例，分别验证 Java 现有模式和 Python 兼容模式。
8. Java 与 Python 使用同一批冻结直接标签重新评测，分别输出策略、传播、字段模型和最终检测差异。
9. 新增字符级 PVD 或 OD 参数网格前后必须进行消融实验，不能只以策略数量作为验收标准。

## 十三、最终判断

本次“采样标签不足未生成模型”是字段级单类别问题，不是总采样数量异常。对 `src` 和 `flight` 而言，由于完整数据真值都没有错误，继续加大随机采样也不会产生正例。Java 跳过二分类训练本身合理，真正需要修复的是字段状态持久化、检测预检、覆盖率表达和单类别生产策略。

Java 60 与 Python 308 的主要差异来自策略展开粒度，而不是简单缺少 248 个算法。RVD 的业务字段对数量已经一致，12 条差异来自正确排除 `tuple_id`；PVD 的 208 条数量差主要来自 Python 按实际字符展开；OD 的 28 条差来自 dBoost 参数网格与 Spark 字段级策略的不同设计。

当前最优先工作不是把 Java 策略数量扩到 308，而是先用冻结标签建立可归因对比，并修复无模型字段的运行语义。之后再根据消融实验决定是否补充字符级 PVD、OD 参数网格或 Python 兼容的 RVD 空值模式。
