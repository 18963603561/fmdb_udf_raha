当前工程删除 `security`、`audit`、`retention`、`performance`、`production` 五个治理包后，核心算法与 UDF 全链路仍可正常运行。

## 十二、当前问题

### 12.1 回归评测没有冻结训练样本

主动采样是完整算法的一部分，但它也会把策略、聚类、资源并发和排序变化传导到训练样本。当前历史回归直接比较最终 F1，无法区分策略变化、采样变化和模型变化。

### 12.2 Java 精确率偏低

Java 在 `act_dep_time` 和 `act_arr_time` 上分别产生 637 和 282 个假阳性。两个字段贡献了绝大多数误报，阈值继续降低不适合作为主要优化手段。

### 12.3 预期跳过字段被记录为错误

`src` 和 `flight` 没有正样本，训练阶段正确跳过，但检测阶段仍以错误日志记录“没有已发布模型”，并将任务标记为部分成功。该状态会污染监控和告警。

### 12.4 Java 与 Python 的训练样本完全不同

Java 的一基行号转换为零基后，与 Python 的 20 个标注行交集为 0。当前总体对比适合评估完整系统效果，不适合验证相同标签下的模型等价性。

### 12.5 策略覆盖仍有差距

Java OD 和 PVD 数量明显少于 Python。RVD 的 19 个近似映射均为 Java 真子集，会影响关系信号覆盖范围。

### 12.6 治理能力已外移但尚未由宿主补齐

当前验证版本不再具备权限、审计、自动清理、性能基线和生产门禁。核心算法可以验证，但不能直接按生产版本发布。

## 十三、改进建议

### 13.1 第一优先级：建立冻结标签回归集

固定一组 20 行或更大的标注坐标，Java 与 Python 都使用同一组直接标签执行传播、训练和检测。主动采样结果另设一组独立指标。这样才能分别评估策略、采样和模型。

### 13.2 第二优先级：固化聚类和采样排序

在聚类输入、聚类归属和候选元组评分后统一按稳定行号排序；所有分数相同的候选必须使用稳定行号作为最终决胜字段。摘要中应记录聚类版本、采样版本、Spark 分区数和并发参数。

### 13.3 第三优先级：降低两个实际时间字段误报

对 `act_dep_time` 和 `act_arr_time` 分别增加概率校准、精确率下限和误报上限。阈值选择应同时满足召回下限与精确率下限，无法同时满足时应显式回退并记录原因。

### 13.4 第四优先级：调整无模型字段状态

训练输出应明确保存 `SKIPPED_SINGLE_CLASS` 或同类状态。检测前先做模型可用性预检，对预期跳过字段返回正常跳过，不进入并行执行器错误路径，也不把整个检测任务标记为部分失败。

### 13.5 第五优先级：继续策略语义对齐

优先分析 19 个 RVD 真子集差异的空值、规范化和冲突两侧定义。PVD 若需要接近 Python，应实现带字符上限、控制字符过滤和稳定配置标识的动态字符网格。

### 13.6 第六优先级：由宿主环境补齐治理

验证环境至少需要外部账户隔离、结果目录清理脚本、集中日志和资源监控。进入生产前必须重新接入权限、审计、保留策略、性能门禁和上线检查。

## 十四、最终判断

五个治理包删除后，Java Raha 的核心编译、测试、Spark UDF、训练、预测和评估链路均通过，删除方案对当前隔离验证阶段是可行的。

从算法效果看，当前 Java 的 F1 略高于 Python，但精确率明显较低，不能仅凭 F1 判断已经完全优化。当前最需要解决的是主动采样回归口径、两个实际时间字段的误报和无模型字段的状态语义。

从删除影响看，策略层 60 个结果与删除前完全一致，性能也处于同一量级。历史最终指标变化主要伴随训练样本和阈值变化，没有证据表明是删除治理包直接改变了算法计算。

从发布边界看，当前工程应继续定位为核心算法和 FMDB 适配验证组件，不应直接作为具备生产治理能力的版本上线。
