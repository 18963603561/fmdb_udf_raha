# Python Raha 与 Java Spark hospital 结果对比分析

## 一、结论摘要

本次使用两边内容完全一致的 `hospital` 数据，完成了 Python Raha 原始流程和 Java Spark UDF 的采样、训练、发布、预测、评估及策略坐标对比。

最终结果为：Python 精确率 0.981250、召回率 0.616896、F1 0.757539；Java 精确率 0.838828、召回率 0.449902、F1 0.585678。Java F1 比 Python 低 0.171861，精确率低 0.142422，召回率低 0.166994。

主要差距不是 RVD 策略不一致。Python 与 Java 共有的 342 个业务字段 RVD 有向字段对，策略坐标全部完全一致；Python 多出的 38 个 RVD 全部涉及行标识 `index`，Java 排除行标识是正确设计。

Java 效果较低的首要原因是模型字段覆盖不足。19 个业务字段中只有 9 个生成模型，模型覆盖率为 47.37%；10 个无模型字段中有 8 个字段实际存在错误，共包含 233 个真实错误，占 Java 280 个全量假阴性的 83.21%。检测结果缺失主要发生在训练阶段没有采到字段正例，而不是模型预测阈值稍有偏差。

Python 与 Java 均采样 20 行，但两边采样行只有 2 行重合。Python 采到了 11 个错误字段的正例，Java 只采到了 9 个错误字段的正例。Java 采到了 `state`、`owner`，Python 没有；Python 采到了 `city`、`phone`、`type`、`condition`，Java 没有。最终字段检测能力基本沿着这组采样差异展开。

本次还发现并修复了一个 Java 容器验证性能问题：`InMemoryFmdbTableGateway` 连续追加特征字典时不断累积 Spark 血缘，写入耗时呈指数增长。修复前 14 分钟仍只保存 13/19 个字典，前 13 个字典保存跨度已达 376 秒；修复后 19 个字典在 17 秒内全部保存，完整 Java 流程约 278 秒完成。

此外进行了全字段阈值调优实验。该实验使 Java 全量 F1 从 0.585678 降到 0.582051，阈值大量跳到 0.99，并在少数字段降到 0.01、0.03，表现出小验证集过拟合。因此该实验代码已经撤销，最终工程只保留内存网关血缘修复。

## 九、Java 性能问题与修复

### 9.1 问题现象

第一次 Java 运行中，模型训练已经生成 9 个候选模型，但在保存 19 个特征字典时耗时持续翻倍：

| 状态 | 结果 |
| --- | --- |
| 已运行时间 | 约 14 分钟 |
| 已保存字典 | 13/19 |
| 前 13 个保存跨度 | 376 秒 |
| 完整任务 | 未完成，主动中止 |

### 9.2 根因

`InMemoryFmdbTableGateway.appendIdempotent` 每次写入都保存以下惰性数据集：

```text
existing union pending
```

下一次写入为了判断幂等键又要读取 `existing`，Spark 会递归重算全部历史 `union` 和 `left_anti join` 血缘。随着写入批次数增加，逻辑计划和计算量快速膨胀。

该问题只存在于开发和测试使用的内存 FMDB 网关，不是生产 `SparkSqlFmdbTableGateway` 的 SQL 写入路径。

### 9.3 修复

每次追加成功后执行以下处理：

1. 对合并后的数据集执行立即本地检查点。
2. 物化当前完整表数据并截断历史 Spark 血缘。
3. 释放旧数据集缓存。
4. 将物化后的数据集重新注册为当前临时视图。

新增集成测试验证连续追加后：

1. 幂等追加行数正确。
2. 最终逻辑计划不再保留 `Union` 历史血缘。

### 9.4 修复效果

| 指标 | 修复前 | 修复后 |
| --- | ---: | ---: |
| 已保存字典 | 13/19 | 19/19 |
| 字典保存跨度 | 376 秒 | 17 秒 |
| 完整 Java 流程 | 约 14 分钟仍未完成 | 277.566 秒完成 |

修复后连续两次完整 Java 运行的采样行、436 个策略、526593 次命中、9 个模型、9000 条评分及最终指标完全一致，说明性能修复没有改变算法结果。

## 十一、当前主要问题

### 11.1 主动采样没有保证字段级正例覆盖

hospital 总错误数为 509，但单字段错误率大多只有 2% 至 4%。随机或聚类覆盖型 20 行采样很容易在部分字段完全没有正例。当前采样目标更偏向元组整体覆盖，没有显式优化“每个字段至少采到一个或多个正例”。

### 11.2 无模型字段仍被当成检测异常

训练阶段 `skippedCount=10`、`failedCount=0`，说明这些字段是预期跳过；检测阶段却输出 10 个错误堆栈，并把任务标记为 `PARTIAL_SUCCESS`。预期跳过、模型文件缺失和版本不兼容仍未区分。

### 11.3 一个正例即可训练，统计可靠性不足

当前只要求同时存在正负两类，没有最少正例数限制。1 个正例加 19 个负例即可训练逻辑回归，训练 F1 可以达到 1，但不能证明泛化可靠。

### 11.4 质量门禁主要检查训练集

`sample` 和 `state_average` 训练 F1 都为 1，但全量 F1 分别只有 0.539683 和 0.581818。需要把独立验证指标、样本量和类别覆盖纳入模型发布门禁。

### 11.5 阈值调优仍带有数据集特例

容器验证程序只调优 `act_dep_time`，不具备通用数据集能力；但简单扩展到所有字段又会在小验证集上过拟合。需要以配置、最少验证样本和交叉验证替代字段名硬编码。

### 11.6 双边对比仍受主动采样差异干扰

双方采样行只有 2/20 重合。当前结果适合评价完整系统，不适合把最终差异精确归因到策略、传播算法或分类器。

## 十二、调整建议

### 12.1 第一优先级：增加字段级定向补采样

首轮 20 行采样后，统计每个字段的直接正、负标签数量：

1. 没有正例但策略存在明显异常簇的字段，进入定向补采样。
2. 优先选择该字段策略分歧大、模式稀有或聚类未覆盖的元组。
3. 设置每字段最少正例数、最少负例数和最大补采样轮数。
4. 达到最大轮数仍无正例时，明确记录 `SKIPPED_SINGLE_CLASS`。

对 hospital，优先补采样字段应包括 `provider_number`、`name`、`city`、`phone`、`type`、`emergency_service`、`condition`、`score`。

### 12.2 第一优先级：增加模型覆盖率门禁

训练任务必须输出并校验：

1. 期望模型字段数。
2. 实际候选和已发布模型字段数。
3. 字段模型覆盖率。
4. 无模型字段对应的历史错误量或策略异常量。

对于本次 9/19 的覆盖率，任务不应只返回普通 `SUCCEEDED`。至少应返回带覆盖告警的成功状态，或根据业务阈值阻止自动发布。

### 12.3 第一优先级：修复无模型字段状态语义

训练阶段持久化 `SKIPPED_SINGLE_CLASS`、`SKIPPED_NO_LABELS`、`SKIPPED_EMPTY_FEATURES` 等状态。检测阶段预检后正常跳过预期字段，不再打印模型加载异常；意外丢失和版本不兼容仍保留失败。

### 12.4 第二优先级：提高模型发布门槛

建议加入以下可配置条件：

```properties
raha.model.minimum-positive-labels=5
raha.model.minimum-negative-labels=5
raha.model.minimum-validation-positive-labels=5
raha.model.minimum-validation-negative-labels=20
```

具体数值应通过多数据集学习曲线确定。没有达到门槛的字段可以保存为实验候选，但不应直接自动发布。

### 12.5 第二优先级：重做通用阈值选择

通用阈值机制应满足：

1. 不使用字段名硬编码。
2. 验证集正负样本达到门槛后才允许调优。
3. 使用分层交叉验证或多次切分，而不是一次小验证集。
4. 对阈值偏离 0.5 设置最大变化限制。
5. 调优后必须通过独立保留集性能不下降门禁。

### 12.6 第二优先级：建立冻结标签消融实验

固定相同的 20 行或更大直接标签，让 Java 与 Python 分别运行：

1. 相同直接标签，不传播。
2. 相同直接标签，加各自传播。
3. 相同标签和传播结果，只比较模型。
4. 逐步移除 OD、PVD、RVD，观察字段指标变化。

只有这样才能判断 Python 361 个字符 PVD 相对 Java 69 个聚合 PVD 是否真正贡献了额外效果。

### 12.7 第三优先级：保留内存网关血缘回归测试

后续任何内存表追加、更新或删除改造都必须验证 Spark 逻辑计划不会随写入次数无限增长。该问题在 toy 和 flights 的少字段测试中不明显，hospital 的 19 个字段才暴露出来。

## 十三、最终判断

从策略一致性看，Java 在 hospital 上的 RVD 已达到完整业务语义对齐，342 个字段对全部精确一致。策略数量从 Python 775 减少到 Java 436，主要来自 PVD 聚合和排除行标识，并不能直接解释 0.171861 的 F1 差距。

从模型效果看，Java 当前最大问题是主动采样和模型覆盖。只要字段没有在 20 行中采到正例，Java 就不生成模型；本次 233 个真实错误位于无模型字段，占 Java 假阴性的 83.21%。因此下一步投入应优先放在字段级补采样、模型覆盖门禁和无模型状态语义，而不是先扩充策略总数。

从工程性能看，内存 FMDB 网关血缘累积问题已经修复，字典保存从指数增长恢复为稳定耗时，最终 Jar 已通过 136 个测试并在 Spark Standalone 上完成精确复现。

全字段阈值调优实验已经证明，缺少样本量约束时，通用化阈值选择反而会过拟合。该实验未进入最终代码，后续必须以最少验证样本、交叉验证和独立保留集门禁重新设计。
