# 工作流默认构造器与存储实现切换分析

## 结论

这三个工作流类可以做成“默认构造器可用”，但**不建议直接在工作流类里内置默认对象**。
更合理的方式，是新增一个**最外层装配工厂**，由它根据配置决定使用内存实现还是 FMDB 实现，再把已经组装好的依赖传给工作流。

## 现状

### 1. 三个工作流都不是轻量对象

它们都继承了 `AbstractRahaWorkflow`，公共父类已经强依赖下面五个对象：

- `RahaDatasetLoader`
- `ColumnProfileService`
- `StrategyPlanService`
- `StrategyExecutionService`
- `FeatureService`

也就是说，工作流本身一出生就要具备“数据加载、画像、策略、执行、特征”这一整条准备链。

### 2. 各自还额外依赖一批业务对象

`TrainingWorkflow` 还需要：

- `ColumnClusteringService`
- `LabelPropagationService`
- `RahaTrainService`
- `TrainingInputMergeService`，可选
- `ResultPersistenceVerifier`，可选

`SamplingWorkflow` 还需要：

- `ColumnClusteringService`
- `RahaSampleService`
- `SampleRecordService`
- `ResultPersistenceVerifier`，可选

`DetectionWorkflow` 还需要：

- `RahaDetectService`
- `ResultPersistenceVerifier`，可选

所以这三个类如果强行补默认构造器，默认逻辑最终还是要把这一长串依赖组装出来。

## 为什么不建议把默认构造器直接加进这三个类

### 1. 职责会变重

工作流类现在的职责是“按任务类型拼装阶段处理器”。
如果把对象创建也塞进去，它们就会变成“半个装配容器”，可读性会下降。

### 2. 会把存储选择扩散到业务层

你提到的“内存存储还是 FMDB 实现”，本质上不是工作流选择问题，而是**仓储和持久化层选择问题**。

如果把这个开关放进工作流类，后果通常是：

- 工作流知道太多底层实现细节
- 训练、采样、检测三条链路都要重复做选择
- 以后新增别的存储实现时，改动面会扩大

### 3. 默认对象很难真正“默认”

这条链路不是只有一个 `FmdbTableGateway`。
它还会继续影响：

- `StageRepository`
- `JobRepository`
- `RahaRepository`
- `SampleRecordService`
- `FmdbModelStore`
- `FmdbSampleRecordRepository`

也就是说，真正的切换点不是工作流，而是**最外层装配**。

## 更合适的设计

### 1. 新增一个装配工厂

建议新增类似下面的工厂职责：

- 根据配置读取运行模式
- 创建内存版或 FMDB 版仓储
- 创建对应的 `RahaTaskApplicationService`
- 再把工作流实例统一注册进去

这样工作流类依旧保持纯粹，只管业务阶段编排。

### 2. 配置应该放在更高层

如果要在 `raha-defaults.properties` 里加开关，建议加的是**运行模式**，而不是直接加“某个工作流使用哪个实现”。

更合适的配置语义类似：

- `raha.runtime.storage-mode=FMDB`
- 或 `raha.persistence.backend=fmdb`
- 或 `raha.task.runtime-mode=production`

然后由工厂去解释这个开关：

- `IN_MEMORY`：走内存仓储、内存网关、测试友好实现
- `FMDB`：走 Spark SQL 和 FMDB 持久化实现

## 对这三个工作流的建议

### 1. 不建议直接新增无参构造器

原因是它们需要的依赖太多，无参构造器最后只会把“默认对象创建”转移到类内部静态方法，反而更难维护。

### 2. 可以考虑新增静态工厂方法，但只做薄封装

如果确实想让外部创建更方便，可以加：

- `of(...)`
- `createDefault(...)`

但这些方法最好仍然接收已组装好的核心依赖，不要直接在工作流类里 new 出整条链。

### 3. 存储实现切换不要放进工作流类

工作流类只关心“怎么跑阶段”。
存储实现切换应该放在：

- 启动装配类
- 应用服务工厂
- 或 Spring 配置类

## 最终建议

最稳妥的方案是：

1. 保持 `TrainingWorkflow`、`SamplingWorkflow`、`DetectionWorkflow` 现有显式构造器不变
2. 新增一个顶层工厂或启动装配类
3. 在 `raha-defaults.properties` 增加运行模式配置
4. 由工厂统一选择内存实现或 FMDB 实现

这样能把“切换策略”放在最外层，把“业务编排”留在工作流类本身。
