# FMDB 仓储版本实现状态分析

## 一、结论

针对本次核查的六类仓储：

- 阶段仓储 `StageRepository`
- 策略仓储 `StrategyRepository`
- 特征仓储 `FeatureRepository`
- 画像仓储 `ColumnProfileRepository`
- 聚类仓储 `ClusterRepository`
- 检测结果仓储 `DetectionResultRepository`

结论是：这些仓储目前都没有独立、完整的 FMDB 生产版实现。

当前代码中存在的是 `Default*Repository` 适配器，它们都基于统一的 `RahaRepository` 抽象；默认落地对象是 `InMemoryRahaRepository`。
即使 `raha.runtime.storage-mode=FMDB`，默认工厂当前也仍然为这些仓储注入 `Default*Repository + InMemoryRahaRepository`。

检测结果比较特殊：项目已有 `SparkSqlFmdbResultWriter` 和 `FmdbResultPersistenceVerifier` 可以把最终错误结果写入 `dw.raha_detection_result`。
但这不是 `DetectionResultRepository` 的 FMDB 实现。
`DetectionResultRepository` 在默认工厂里仍然使用 `DefaultDetectionResultRepository`。

## 四、是否已经实现

### 1. 阶段仓储

尚未实现 FMDB 版本。

当前只有：

- `StageRepository`
- `DefaultStageRepository`
- `DefaultStageCheckpointRepository`
- `StageCheckpointRunner`

但没有 `FmdbStageRepository` 或等价实现。
默认工厂仍注入 `DefaultStageRepository`。

任务清单中对应 `RUN-003`，要求实现 FMDB 阶段和检查点仓储，并与 `raha_job_stage_attempt` 共用物理表。
当前未完成。

### 2. 策略仓储

默认工厂中策略计划和策略执行服务都使用 `DefaultStrategyRepository`。

任务清单中要求策略计划和摘要进入 `raha_training_column_artifact`。
当前未完成。

### 3. 特征仓储

### 6. 检测结果仓储

仓储层尚未实现 FMDB 版本，但物理写入链路已有部分实现。

当前默认工厂中：

```java
DetectionResultRepository detectionRepository =
        new DefaultDetectionResultRepository(repository);
```

所以 `RahaDetectService` 内部保存的检测结果仍然进入内存仓储。

同时，FMDB 模式下 `DetectionWorkflow` 会接入 `FmdbResultPersistenceVerifier`，它会调用 `SparkSqlFmdbResultWriter.writeDetectionResults` 把错误结果写入 `dw.raha_detection_result`。

这表示：

- 检测错误表物理写入：已有部分实现
- `DetectionResultRepository` 的 FMDB 版本：没有实现
- 默认检测服务内部仓储：仍是内存

任务清单中 `DB-006` 和 `DET-001` 至 `DET-005` 覆盖该工作。
当前只能算部分实现，不能算仓储实现完成。

## 五、和当前默认工厂的关系

当前默认工厂在 `raha.runtime.storage-mode=FMDB` 时已经切换的内容包括：

- `SparkSqlFmdbTableGateway`
- `FmdbJobRepository`
- `FmdbModelMetadataRepository`
- `FmdbModelStore`
- `FmdbSampleRecordRepository`
- `FmdbResultPersistenceVerifier`
- `SparkSqlFmdbResultWriter`

但以下内容没有随 `FMDB` 模式切换：

- `StageRepository`
- `StrategyRepository`
- `FeatureRepository`
- `ColumnProfileRepository`
- `ClusterRepository`
- `DetectionResultRepository`

因此，`FMDB` 模式当前是混合后端：

```text
任务、采样、模型、检测错误结果：部分 FMDB
阶段、策略、特征、画像、聚类、检测服务内部结果：仍是内存
```

## 六、最终判断

这六类仓储的 FMDB 版本确实还没有完整实现。

任务清单里已经有对应任务，而且描述比较明确。
但从当前代码看，除检测错误物理写入链路已有部分落地外，其余五类仍停留在内存仓储。

如果后续要继续落地，建议优先顺序是：

1. `StageRepository` 和检查点仓储：对应 `RUN-003`，决定任务能否跨进程恢复。
2. `ColumnProfileRepository`、`StrategyRepository`、`FeatureRepository`、`ClusterRepository`：对应 `ART-003` 到 `ART-008`，决定训练列级产物和训练单元格能否落库。
3. `DetectionResultRepository`：对应 `DET-001` 到 `DET-005`，决定检测查询是否从内存仓储切到最终错误表。

其中不要为每个仓储单独新建独立物理表。
应按清单要求映射到最终九张 FMDB 物理表，避免形成第二套数据库协议。
