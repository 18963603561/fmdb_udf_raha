# Raha 首个可用版本里程碑核对

日期：2026-07-19

## 一、环境和构建结论

1. 本次实现不依赖新增网络下载或外部网络服务。
2. 首次 Maven 失败原因为当前默认 Java 版本为 JDK 17，而项目强制 JDK 8。
3. 已使用 `D:\Program Files\java\jdk8u492-b09` 完成编译、测试和打包。
4. 全量命令为 `mvn -q test`，退出码为 0。
5. 打包命令为 `mvn -q -DskipTests package`，退出码为 0。
6. `mvn dependency:tree -Dincludes=org.apache.poi` 仅发现 `org.apache.poi:poi:5.2.5`。
7. Maven Shade 配置使用 `<include>org.apache.poi:poi</include>`。
8. 产物中存在 HSSF 类，不存在 XSSF 和 XWPF 类。
9. Excel 适配器统一使用 `Workbook workbook = new HSSFWorkbook()`。
10. 输入和输出均强制使用 `.xls` 后缀，`.xlsx` 会被拒绝。

## 二、M3 标注闭环

1. 已实现标注批次、行标注、导入结果、行错误和修订关系对象。
2. 已实现 `标注数据`、`标注说明`、`导入校验`、`系统信息` 四张工作表。
3. 工作簿读取会强制检查四张工作表，不信任隐藏元数据绕过模板校验。
4. 业务列、系统列、行标识、内容哈希和模式哈希均有明确导入校验。
5. 导入校验失败会返回稳定错误码，并可回写错误工作簿。
6. 原始业务值只从可信 c1 仓储回填，用户编辑业务值不能绕过校验。
7. 用户排序后仍按 `_row_id` 关联，列号不参与关联。
8. `rowLabel`、`reviewedColumns`、`errorColumns` 会确定性展开为直接 `CellLabel`。
9. 直接标签来源快照写入 `annotation_json`，跨进程读取可以恢复标签坐标。
10. 标注批次通过 FMDB 追加式仓储幂等写入，旧批次不更新。
11. 已覆盖正常往返、排序、业务值篡改、隐藏哈希篡改、重复导入和错误回写测试。
12. M3 定向测试：`AnnotationImportServiceIntegrationTest`，4 个测试全部通过。

## 三、M4 合并闭环

1. 训练请求支持采样批次、标注批次、月分区和行身份配置。
2. 统一训练工作流在加载 o1 后插入 `MERGE_TRAINING_INPUT` 阶段。
3. 后续画像、策略、特征、聚类和传播均接收新的训练快照。
4. c1 和 o1 使用同一行身份模式、字段、算法和规范版本。
5. SOURCE_KEY 和 CONTENT_HASH 均支持 c1 优先去重。
6. c1 不在 o1 中时仍保留，联合键内容冲突选择规则稳定并记录汇总 WARN。
7. 小 c1 按资源配置估算字节数，满足阈值时广播身份键，超过阈值自动回退普通连接。
8. 合并训练快照绑定新的 `training_snapshot_id`，下游不再读取原始 o1 快照。
9. 标注标签按数据集、训练快照、行标识和字段重新计算单元格坐标。
10. 合并后重新生成训练画像，不能复用采样画像。
11. M4 定向测试：`TrainingInputMergeServiceIntegrationTest`，1 个测试通过。

## 四、M5 训练产物闭环

1. 已实现训练列级产物、训练单元格和最终训练样本三类记录对象。
2. 列级画像、策略、特征字典、聚类和传播摘要分别受独立开关控制。
3. 关闭的 JSON 字段会写空值，不会误写被关闭的产物。
4. 特征字典版本纳入训练批次、策略计划和执行配置指纹。
5. 训练单元格从可信训练快照回填 `cell_value`，找不到原值直接拒绝物化。
6. 最终训练样本只保存实际进入列模型训练的数据。
7. 训练样本写入后按模型集合版本和字段重新读取，再启动模型训练。
8. 物理写入数量和重新读取数量不一致时训练失败，不发布模型。
9. 训练产物追加使用业务键幂等，重复执行会校验已有物理记录而不是误报失败。
10. M5 定向测试：`FmdbTrainingArtifactRepositoryTest`，2 个测试通过。

## 五、M6、M7 和 M8 首版范围

1. `FmdbModelMetadataRepository` 已基于 `raha_model_artifact` 追加模型状态快照。
2. 模型表增加 `schema_hash` 和 `state_version`，模型状态可跨实例恢复。
3. 已覆盖候选、发布和新仓储实例读取测试。
4. `FmdbJobRepository` 已实现任务幂等查询、状态追加和单调状态版本。
5. 任务恢复使用领域状态机校验，不直接反序列化可变字段。
6. `ResultPersistenceStageHandler` 支持真实物理表验证器，不能只凭 URI 报成功。
7. FMDB 结果验证器覆盖采样、训练列级产物、最终训练样本、模型和检测错误表。
8. 检测写入器只写 `isError=true` 的单元格，零错误批次不写错误表但可成功。
9. 原始值访问策略覆盖 c1、标注、检测原值写入，并记录主体、数据集和用途。
10. 所有热路径查询均使用显式投影和条件过滤，训练读取按批次、月份、模型集合和字段裁剪。
11. M6/M7 定向测试：`FmdbRuntimeRepositoryIntegrationTest`，2 个测试通过。
12. 全量测试共覆盖现有工程和新增 M3 至 M8 回归，全部通过。

## 六、仍需真实环境执行的验收

1. 当前工作区没有真实 FMDB 连接参数、Catalog 或 ORC 集群，因此未伪造 OPS-007 的真实 FMDB 验收结论。
2. 真实环境需要执行九张表初始化、追加、幂等、分区读取、重启读取和失败恢复。
3. 真实环境还需要按 OPS-008 执行十万采样源、一千 c1、两百万 o1 的规模基线。
4. 本机已完成 Spark 内存网关、Spark Catalog 适配、表契约和跨实例对象恢复验证。
5. 真实 FMDB 上线前必须使用部署侧授权策略配置原始值访问主体和数据集范围。
