# Raha 包结构分层重构落地报告

## 一、落地目标

本次重构按照 `Raha包结构分层重构分析-202607181231.md` 执行，目标是把原来集中在一层包中的接口、领域对象、请求结果、业务服务和基础设施实现拆分到职责明确的二、三层包中。

本次只调整代码组织和依赖表达，不改变算法语义，不增加旧包兼容类，也不重新引入 UDF、文件队列或常驻 Worker。

## 四、重点迁移

### 4.1 任务模型

| 新包 | 主要内容 |
| --- | --- |
| `job.domain` | `RahaJob`、`RahaStage`、`JobRunResult` |
| `job.execution` | `RahaJobOrchestrator`、`StageFailureDecider`、`FailureDecision` |
| `job.id` | 任务标识、阶段标识和幂等键生成 |
| `job.stage` | 阶段处理器、执行上下文、属性键和阶段结果 |

任务模型主线保持为：

```text
RahaJobOrchestrator.submit
  -> RahaJobOrchestrator.execute
  -> StageHandler.execute
  -> 领域服务
  -> 仓储端口
```

`submit` 和 `execute` 均由同一调用进程直接触发，不依赖任务文件和外部消费者。

### 4.2 服务层

| 新包 | 主要内容 |
| --- | --- |
| `service.common` | 公共任务状态、类型、摘要和结果 |
| `service.prepare` | 特征准备请求、结果和服务 |
| `service.sample` | 采样请求、输出、服务和主动采样编排 |
| `service.train` | 训练请求、输出和服务 |
| `service.detect` | 检测请求、输出和服务 |

服务请求、返回对象和业务服务按用例归组，不再集中在 `service` 根包。

### 4.3 仓储层

| 新包 | 主要内容 |
| --- | --- |
| `repository.core` | 通用仓储接口、键、记录、事务、版本和保存结果 |
| `repository.port` | 任务、阶段、特征、模型、标签等领域仓储端口 |
| `repository.adapter` | 默认类型化适配器、内存仓储和持久化结构 |

核心服务依赖 `repository.port`，具体实现放在 `repository.adapter`，接口和实现边界已经明确。

### 4.4 领域能力

| 能力 | 分层结果 |
| --- | --- |
| 数据 | `data.domain`、`data.type`、`data.loader`、`data.profile` |
| 策略 | `strategy.api`、`strategy.plan`、`strategy.execution`、`strategy.domain`、`strategy.impl` |
| 特征 | `feature.domain`、`feature.assembly` |
| 聚类 | `cluster.domain`、`cluster.algorithm` |
| 采样 | `sampling.domain`、`sampling.service` |
| 标签 | `label.propagation` |
| 模型 | `model.domain`、`model.training`、`model.prediction`、`model.release` |
| 检测 | `detection.scoring`、`detection.explanation`、`detection.service` |
| 评估 | `evaluation.metrics`、`evaluation.threshold` |
| FMDB | `fmdb.gateway` 和 FMDB 根包中的其他适配能力 |

## 五、可见性调整

拆包后，原先依赖包内可见性的配置工具需要成为明确契约。本次做了两项必要调整：

1. `ConfigTextUtils` 改为公共工具类，三个稳定文本方法改为公共静态方法，并补充用途和边界注释。
2. `RahaJobConfig.toCanonicalString` 改为公共只读方法，仅用于配置版本和摘要计算，不作为展示格式或持久化协议。

除此之外未修改领域对象和算法的访问级别。

## 九、当前边界

本次完成的是包结构和依赖表达落地，不新增生产启动入口。当前工程仍由外部调用方负责：

1. 创建 Spark 会话。
2. 选择并创建仓储实现。
3. 组装领域服务和阶段处理器。
4. 调用 `RahaJobOrchestrator.submit` 和 `execute`。

如果后续需要命令行、批处理平台或 HTTP 入口，应新增薄入口并调用现有任务模型，不应把任务编排重新复制到入口类中。

包结构重构已经完整落地，工程不再以一层包混放接口、实现和数据对象。任务编排、业务服务、领域能力、仓储端口和基础设施实现的边界已经通过目录和包名明确表达，并通过从零编译和完整测试验证。
