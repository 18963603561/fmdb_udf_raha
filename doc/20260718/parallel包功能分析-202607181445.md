# parallel 包功能分析与示例

## 一、包定位

`src/main/java/com/fiberhome/ml/raha/parallel` 是项目里的通用并行与 Spark 资源控制包，主要解决两个问题：

1. 对一批独立任务做受限并发执行，避免字段、策略等批处理任务无限制创建线程或拖垮资源。
2. 对 Spark 广播变量和数据集缓存做大小阈值保护，避免误把过大对象广播或缓存到 Spark 内存中。

这个包不是具体业务算法包。它更像底层执行基础设施，被策略执行、特征组装、列内聚类、训练、检测等业务服务复用。

## 二、类职责总览

| 类名 | 主要职责 | 典型使用方 |
| --- | --- | --- |
| `BoundedParallelExecutor` | 用固定线程池执行一批工作项，限制最大并发，统一批次超时，隔离单项异常。 | 策略、特征、聚类、训练、检测 |
| `ParallelWorkItem<K, V>` | 描述一个带稳定唯一业务键的并行任务。 | 所有需要组装并行任务列表的服务 |
| `ParallelBatchResult<K, V>` | 保存成功结果、失败摘要和实际观察到的并发峰值。 | 调用并行执行器后的结果汇总逻辑 |
| `ParallelFailure` | 保存某个任务失败的类型、摘要和是否超时。 | 失败统计、错误码转换、部分成功判断 |
| `SparkResourceManager` | 按 `ResourceConfig` 判断是否允许广播小对象或缓存数据集。 | Spark 预测、特征准备、资源保护相关流程 |

## 三、BoundedParallelExecutor 的核心行为

`BoundedParallelExecutor.execute(items, maxConcurrency, timeoutMillis)` 是整个包最核心的方法。

它接收：

| 参数 | 含义 |
| --- | --- |
| `items` | 待执行的任务列表，每个任务有一个唯一 `key` 和一个 `Callable` |
| `maxConcurrency` | 最大并发线程数 |
| `timeoutMillis` | 整个批次的总超时时间，不是每个任务各自独立计时 |

它返回：

| 返回内容 | 含义 |
| --- | --- |
| `successes` | 成功任务结果，按输入顺序保存 |
| `failures` | 失败任务摘要，按输入顺序保存 |
| `maxObservedConcurrency` | 执行期间实际达到的最大并发数 |

关键设计点：

1. `items` 不能为空，`maxConcurrency` 和 `timeoutMillis` 必须大于零。
2. 每个 `ParallelWorkItem` 的 `key` 必须唯一，重复会直接抛出 `IllegalArgumentException`。
3. 线程池大小取 `min(maxConcurrency, items.size())`，不会创建超过任务数的线程。
4. 每个任务的异常会被转换为 `ParallelFailure`，不会直接导致其他任务结果丢失。
5. 批次超时后，会取消尚未完成的任务，但会尽量保留已经完成的任务结果。
6. 最终一定会取消未完成任务并关闭线程池。

## 四、通用并行执行示例

下面是一个简化示例：同时计算多个字段的得分，最多两个字段并发，整个批次最多三秒。

```java
List<ParallelWorkItem<String, Integer>> items =
        new ArrayList<ParallelWorkItem<String, Integer>>();

items.add(new ParallelWorkItem<String, Integer>("name", () -> scoreColumn("name")));
items.add(new ParallelWorkItem<String, Integer>("age", () -> scoreColumn("age")));
items.add(new ParallelWorkItem<String, Integer>("email", () -> scoreColumn("email")));

ParallelBatchResult<String, Integer> result =
        new BoundedParallelExecutor().execute(items, 2, 3000L);

for (Map.Entry<String, Integer> entry : result.getSuccesses().entrySet()) {
    System.out.println(entry.getKey() + "=" + entry.getValue());
}

for (Map.Entry<String, ParallelFailure> entry : result.getFailures().entrySet()) {
    System.out.println(entry.getKey() + " failed: "
            + entry.getValue().getErrorType());
}
```

这个示例里的行为是：

1. `name`、`age`、`email` 三个任务会进入同一个批次。
2. 任意时刻最多两个任务并发。
3. 某一列失败只会进入 `failures`，其他成功列仍保留在 `successes`。
4. 超过三秒后，尚未完成的列会被标记为超时失败。

## 五、在项目里的真实使用方式

### 1. 策略并行执行

`StrategyExecutionService` 会把普通策略计划转换成 `ParallelWorkItem<String, StrategyExecutionResult>`。

业务键是 `strategyId`，任务体是单策略执行：

```java
workItems.add(new ParallelWorkItem<String, StrategyExecutionResult>(
        plan.getStrategyId(), () -> executor.execute(jobId, stageId,
        dataset, plan, strategyTimeoutMillis)));
```

然后使用配置里的策略并发数和批次超时执行：

```java
ParallelBatchResult<String, StrategyExecutionResult> parallelResult =
        parallelExecutor.execute(workItems, maxParallelStrategies, batchTimeoutMillis);
```

这里的目的：

1. 限制同一阶段同时运行的策略数量。
2. 避免单个慢策略或异常策略阻塞所有策略。
3. 按原始策略计划顺序保存执行结果。

注意：`RVD_ONE_TO_MANY` 类型策略被单独归到 `rvdBatchExecutor` 做批量执行，不走普通任务列表。

### 2. 特征按列并行组装

`FeatureService.assembleAndSaveParallel` 会按可检测字段拆成列级任务。

业务键是字段名，任务体是只针对单列隔离后的特征组装：

```java
items.add(new ParallelWorkItem<String, FeatureAssemblyResult>(
        column.getName(), () -> assembler.assemble(
        isolate(dataset, column), plansForColumn(plans, column.getName()),
        hitsForColumn(hits, column.getName()), config)));
```

这里的 `isolate(dataset, column)` 很重要：它只保留行号列和目标列，让每个并行任务处理自己的字段范围。

执行完成后，如果有并行调度失败，服务会直接抛出异常：

```java
if (!parallel.getFailures().isEmpty()) {
    throw new IllegalStateException("列特征并行执行失败："
            + parallel.getFailures().keySet());
}
```

这说明特征组装阶段对完整性要求较高：有列失败时，不产出合并特征结果。

### 3. 列内聚类并行

`ColumnClusteringService.clusterAndSaveParallel` 会把每个字段的特征字典和稀疏特征行拆成一个任务。

业务键是字段名，任务体是 `clusterColumn(...)`：

```java
items.add(new ParallelWorkItem<String, ColumnClusteringResult>(
        columnName, () -> clusterColumn(jobId, entry.getValue(), rows,
        config, randomSeed)));
```

执行完成后按 `features.getDictionaries().keySet()` 的字段顺序保存结果。

这里的目的：

1. 各字段聚类互不影响。
2. 聚类结果保存仍保持稳定字段顺序。
3. `maxObservedConcurrency` 被写入日志，便于确认并发限制是否生效。

### 4. 列级模型训练并行

`RahaTrainService` 在完成策略、特征、聚类和标签传播后，会把每个字段的训练构造成列级任务。

业务键是字段名，任务体是 `trainColumn(...)`：

```java
trainingItems.add(new ParallelWorkItem<String, ColumnTrainingOutcome>(
        entry.getKey(), () -> trainColumn(request, features,
        trainingPropagation, planVersion, entry.getKey(), entry.getValue())));
```

训练阶段对失败的处理比特征、聚类更温和：

1. 成功列进入 `trainingResults`。
2. 有候选模型的列进入 `candidates`。
3. 并行失败列会转换成 `ColumnModelTrainingResult` 的失败状态。
4. 只要还有候选模型，整体可能返回 `PARTIAL_SUCCESS`。

### 5. 已发布模型检测并行

`RahaDetectService.detect` 会对每个有特征字典的字段并行加载模型并预测。

业务键是字段名，任务体是 `detectColumn(...)`：

```java
items.add(new ParallelWorkItem<String, DetectColumnOutcome>(
        entry.getKey(), () -> detectColumn(
        request, entry.getKey(), entry.getValue())));
```

检测阶段会将失败字段记录到 `failedColumns`，成功字段的检测结果仍会保存：

```java
ParallelFailure failure = parallel.getFailures().get(columnName);
failedColumns.put(columnName, failure == null
        ? "RuntimeException" : failure.getErrorType());
```

因此检测服务天然支持部分成功：某些字段没有兼容模型或执行失败时，不会抹掉其他字段的检测结果。

## 六、SparkResourceManager 的功能

`SparkResourceManager` 负责 Spark 资源保护，它提供两个方法：

### 1. `broadcastIfAllowed`

只有当估算大小不超过 `ResourceConfig.broadcastThresholdBytes` 时，才创建广播变量。

```java
ResourceConfig config = new ResourceConfig(
        2, 2, 10L, "MEMORY_ONLY", 20L, 5000L);

SparkResourceManager manager = new SparkResourceManager();

Optional<Broadcast<String>> accepted =
        manager.broadcastIfAllowed(spark, "small", 10L, config);

Optional<Broadcast<String>> refused =
        manager.broadcastIfAllowed(spark, "large", 11L, config);
```

上例中：

1. `"small"` 估算为 10 字节，等于阈值，可以广播。
2. `"large"` 估算为 11 字节，超过阈值，返回 `Optional.empty()`。

### 2. `persistIfAllowed`

只有当估算大小不超过 `ResourceConfig.cacheThresholdBytes` 时，才按配置缓存数据集。

```java
Dataset<Row> dataset = spark.range(0L, 5L).toDF();

boolean refused = manager.persistIfAllowed(dataset, 21L, config);
boolean cached = manager.persistIfAllowed(dataset, 20L, config);
```

上例中：

1. 21 字节超过缓存阈值 20 字节，不缓存。
2. 20 字节等于阈值，按 `MEMORY_ONLY` 缓存。

当前支持的缓存级别：

| 配置值 | Spark StorageLevel |
| --- | --- |
| `MEMORY_ONLY` | `StorageLevel.MEMORY_ONLY()` |
| `MEMORY_AND_DISK` | `StorageLevel.MEMORY_AND_DISK()` |
| `DISK_ONLY` | `StorageLevel.DISK_ONLY()` |

其他值会抛出 `IllegalArgumentException`。

## 七、配置来源

并行和资源阈值来自 `ResourceConfig`：

| 字段 | 作用 |
| --- | --- |
| `maxParallelStrategies` | 同时运行的最大策略任务数量 |
| `maxParallelColumns` | 同时运行的最大列任务数量 |
| `broadcastThresholdBytes` | 允许广播的小对象最大估算字节数 |
| `cacheStorageLevel` | Spark 缓存级别 |
| `cacheThresholdBytes` | 允许缓存的数据最大估算字节数 |
| `stageTimeoutMillis` | 单阶段最大运行时间 |

业务服务通常这样使用：

```java
parallelExecutor.execute(items,
        request.getConfig().getResourceConfig().getMaxParallelColumns(),
        request.getConfig().getResourceConfig().getStageTimeoutMillis());
```

也就是说，业务代码不直接决定线程数和超时，而是统一从资源配置读取。

## 八、异常与超时处理示例

### 1. 单任务异常

```java
List<ParallelWorkItem<String, String>> items = Arrays.asList(
        new ParallelWorkItem<String, String>("success", () -> "ok"),
        new ParallelWorkItem<String, String>("failure", () -> {
            throw new IllegalArgumentException("测试失败");
        }));

ParallelBatchResult<String, String> result =
        new BoundedParallelExecutor().execute(items, 2, 1000L);
```

结果：

1. `successes.get("success")` 为 `"ok"`。
2. `failures.get("failure").getErrorType()` 为 `IllegalArgumentException`。
3. `failures.get("failure").isTimeout()` 为 `false`。

### 2. 批次超时

```java
List<ParallelWorkItem<String, String>> items = Arrays.asList(
        new ParallelWorkItem<String, String>("slow", () -> {
            Thread.sleep(1000L);
            return "late";
        }),
        new ParallelWorkItem<String, String>("queued", () -> "quick"));

ParallelBatchResult<String, String> result =
        new BoundedParallelExecutor().execute(items, 1, 50L);
```

结果：

1. `slow` 通常会被标记为超时失败。
2. `queued` 可能已经抢先完成，也可能因为批次截止被取消。
3. 无论哪种情况，两个任务都会被归档到成功或失败集合中。

### 3. 重复任务键

```java
List<ParallelWorkItem<String, String>> items = Arrays.asList(
        new ParallelWorkItem<String, String>("same", () -> "first"),
        new ParallelWorkItem<String, String>("same", () -> "second"));

new BoundedParallelExecutor().execute(items, 2, 1000L);
```

结果：直接抛出 `IllegalArgumentException`，因为结果映射依赖稳定唯一键。

## 九、整体设计价值

这个包的价值可以概括为：

1. 统一并行模型：不同业务阶段不用重复写线程池、超时、异常隔离逻辑。
2. 控制资源上限：通过 `ResourceConfig` 控制策略并发、列并发、广播阈值、缓存阈值和阶段超时。
3. 保持结果可追踪：所有任务都有业务键，成功和失败都能映射回字段名或策略标识。
4. 支持部分成功：训练、检测等阶段可以保留成功列结果，并把失败列汇总到任务状态。
5. 提升可测试性：执行器是构造函数注入的，测试可以替换或验证并发行为。

## 十、使用时的注意事项

1. `Callable` 内部逻辑要尽量响应线程中断，否则超时取消只能标记结果，无法立即停止底层耗时操作。
2. `timeoutMillis` 是整个批次的总时间，不是每个任务单独拥有完整时间。
3. `ParallelWorkItem` 的 `key` 必须唯一，字段名、策略标识这类稳定业务键最合适。
4. `successes` 和 `failures` 都是不可修改映射，调用方需要复制后再加工。
5. `SparkResourceManager` 依赖调用方提供准确或保守的 `estimatedBytes`，估算过低仍可能造成 Spark 内存压力。
6. 对完整性要求高的阶段可以在发现 `failures` 后抛异常；对允许部分成功的阶段可以把失败转换成业务结果。
