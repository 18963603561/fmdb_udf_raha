# 快照检查点 HDFS 列批完整落地说明

## 一、目标

本次改造将采样到训练之间的检查点从“98 万条 Hive 明细记录”调整为“现有表保存清单，HDFS ORC 保存列批明细”。工程按新系统处理，不保留旧检查点格式兼容代码。

## 二、最终架构

| 层次 | 保存内容 | 物理位置 |
| --- | --- | --- |
| 检查点元数据 | 画像、策略摘要、字典、聚类摘要 | `dw.raha_snapshot_checkpoint` |
| 批次索引 | 路径、字段、记录数、内容校验和 | `dw.raha_snapshot_checkpoint` |
| 重明细 | 稀疏特征、聚类版本和簇标识 | HDFS ORC |
| 最终提交 | 完整字段、批次集合和根路径 | `MANIFEST` |
| 清理审计 | 已删除的检查点根路径 | `CLEANED` |

默认 HDFS 路径结构：

```text
/fmdb/raha/checkpoint/
  dataset=<datasetHash>/
  snapshot=<snapshotHash>/
  checkpoint=<checkpointId>/
  column_batch=0001/
```

## 三、批内流程

1. 全表只执行一次数据加载、字段画像和策略计划生成。
2. 按字段序号每 10 列形成一个逻辑批次。
3. 批内执行策略，默认不执行跨批关系策略。
4. 批内生成稀疏特征并执行列聚类。
5. 将特征和聚类归属合并为 10 字段 ORC 记录；值哈希用于维持可信特征坐标，不保存原始值。
6. 写入临时目录，校验记录数后原子重命名。
7. 向现有检查点表写入当前批字典、聚类摘要和 `DETAIL_BATCH`。
8. 将聚类覆盖折叠到按行状态，释放当前批命中、特征和聚类对象。
9. 全部批次完成后生成采样任务并提交最终清单。

## 四、训练恢复

训练列批把目标字段传给检查点仓储。仓储读取最终清单后，只加载与目标字段相交的 HDFS 列批路径，核对每批记录数，再通过 `column_name` 裁剪并恢复特征、聚类归属、字段字典和策略摘要。

标签传播语义保持不变：仍按 `column_name + cluster_version + cluster_id` 传播标签；模型训练仍使用完整稀疏向量。

## 五、配置

| 配置 | 默认值 | 用途 |
| --- | --- | --- |
| `raha.persistence.snapshot-checkpoint.detail-base-path` | `/fmdb/raha/checkpoint` | HDFS 明细根目录 |
| `raha.persistence.snapshot-checkpoint.column-batch-size` | `10` | 逻辑列批大小 |
| `raha.persistence.snapshot-checkpoint.orc-partition-count` | `4` | 每批 ORC 分区数 |
| `raha.persistence.snapshot-checkpoint.retention-millis` | `604800000` | 已完成检查点保留时间 |
| `raha.persistence.snapshot-checkpoint.cleanup-enabled` | `true` | 训练成功后清理过期明细 |

## 七、失败边界

1. ORC 临时目录写入失败时删除临时路径，不写批次清单。
2. 策略失败超过任务容忍阈值时中止检查点，删除本次检查点 HDFS 根目录。
3. 缺少最终清单的元数据不能被训练恢复。
4. 清理失败不会回滚已经发布的模型，下次训练成功后再次尝试。
5. 已追加但未形成最终清单的轻量元数据属于审计残留，不包含单元格重明细。

## 八、部署要求

这是不兼容的模式升级。部署前应删除或归档旧的 `dw.raha_snapshot_checkpoint`，再使用新版 `raha-fmdb-schema.sql` 创建模式版本 2 空表。旧检查点和旧表字段不在代码中保留恢复分支。

Spark 运行身份必须同时具备 HDFS 明细根目录的创建、读取、重命名和递归删除权限。
