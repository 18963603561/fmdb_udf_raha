# Java Raha 关闭 RVD 性能与效果分析

## 一、结论

如果业务明确只关心 OD、PVD，不关心跨字段关系冲突 RVD，关闭 RVD 会显著提高当前 Java 工程的性能。

在 `hospital` 数据集上，使用相同最终 Jar、相同数据、相同随机种子、相同 20 行标注预算和相同 Spark 资源进行实测：

| 指标 | OD、PVD、RVD | 仅 OD、PVD | 变化 |
| --- | ---: | ---: | ---: |
| 端到端耗时 | 277.566 秒 | 163.052 秒 | 减少 41.26% |
| 端到端速度 | 1.00 倍 | 1.70 倍 | 提升 70% |
| 特征准备耗时 | 121.128 秒 | 44.931 秒 | 减少 62.91% |
| 特征准备速度 | 1.00 倍 | 2.70 倍 | 提升 170% |
| 策略数 | 436 | 94 | 减少 78.44% |
| 策略命中 | 526593 | 5601 | 减少 98.94% |
| 特征数 | 738 | 214 | 减少 71.00% |

因此，“总体性能会提高很多”在 hospital 上成立，但应准确表述为：特征准备阶段提升很大，端到端性能显著提升，不是整个任务缩短到原来的五分之一。

本次效果指标没有下降，反而有所提高：

| 全量指标 | OD、PVD、RVD | 仅 OD、PVD | 变化 |
| --- | ---: | ---: | ---: |
| 精确率 | 0.838828 | 0.853659 | +0.014831 |
| 召回率 | 0.449902 | 0.550098 | +0.100196 |
| F1 | 0.585678 | 0.669056 | +0.083378 |
| 候选模型 | 9 | 10 | +1 |
| 模型字段覆盖率 | 47.37% | 52.63% | +5.26 个百分点 |

但该效果提升不能直接推广到所有数据集。它同时包含“移除 RVD 特征”和“聚类、主动采样结果发生变化”两类影响。

## 三、为什么关闭 RVD 能提速

### 3.1 RVD 占绝大多数策略计划

完整基准的策略组成：

| 策略族 | 策略数 | 占比 |
| --- | ---: | ---: |
| OD | 25 | 5.73% |
| PVD | 69 | 15.83% |
| RVD | 342 | 78.44% |
| 合计 | 436 | 100% |

关闭 RVD 后只剩 94 个 OD、PVD 策略。

### 3.2 RVD 占策略命中的 98.94%

| 策略族 | 命中数 | 占比 |
| --- | ---: | ---: |
| OD | 1566 | 0.30% |
| PVD | 4035 | 0.77% |
| RVD | 520992 | 98.94% |
| 合计 | 526593 | 100% |

hospital 中大量字段值形成一对多关系，RVD 为 19000 个业务单元格产生了超过 52 万次命中。关闭 RVD 后，策略命中量从 526593 降至 5601。

命中量减少会同时降低：

1. Spark RVD 字段对计算量。
2. 策略命中对象数量和内存占用。
3. 特征组装时的分组和映射量。
4. 特征字典维度。
5. 聚类距离计算维度。
6. 模型训练和预测的稀疏特征规模。

### 3.3 Java 已批量执行 RVD，所以不是减少 78% 总耗时

Java 将 342 个 RVD 合并为 22 个 Spark 子批次，完整基准中 RVD 批次实际墙钟耗时约 38 秒。它不是 342 个完全独立的 Spark 作业。

关闭 RVD 后的额外收益主要来自后续阶段：

| 项目 | 完整基准 | 仅 OD、PVD |
| --- | ---: | ---: |
| 特征数 | 738 | 214 |
| 删除常量特征 | 420 | 241 |
| 特征准备总耗时 | 121.128 秒 | 44.931 秒 |

因此，不能用 342/436 直接推断端到端会减少 78%。实际端到端减少 41.26%，符合固定开销仍然存在的情况。

### 3.4 不受 RVD 影响的固定开销

以下工作关闭 RVD 后仍然存在：

1. 脏表、净表读取和画像。
2. 25 个 OD 与 69 个 PVD 执行。
3. 20 轮主动采样。
4. 标签传播。
5. 10 个字段模型训练和发布。
6. 特征字典、模型和检测结果持久化。
7. 10000 个单元格模型评分。
8. 真值评估和结果文件输出。

所以端到端提升为 1.70 倍，而不是 4.64 倍。

## 四、为什么效果反而提高

### 4.1 RVD 命中过密，可能压制 OD/PVD 聚类结构

RVD 贡献了 98.94% 的策略命中。虽然 342 个 RVD 与 Python 坐标完全一致，但“一致”只说明实现正确，不代表这些关系特征适合 hospital 的错误类型。

hospital 的主要错误是字段内部的字符替换、拼写破坏和格式异常。OD、PVD 与这种错误更直接相关；大量跨字段一对多冲突可能成为高密度背景信号，主导余弦距离和聚类结果。

关闭 RVD 后，聚类和采样更受 OD、PVD 影响，20 行主动采样随之变化。

### 4.2 两次采样只有 4 行重合

完整基准采样行：

```text
119,12,371,399,120,536,320,851,380,297,824,450,998,326,452,596,514,83,543,483
```

OD/PVD-only 采样行：

```text
119,397,370,12,120,820,324,537,325,45,346,451,382,121,829,83,635,404,516,377
```

双方只有 `119`、`12`、`120`、`83` 四行重合。

因此效果提升不是“保留相同训练样本后删掉 RVD 特征”的纯模型消融结果，而是整个主动学习流程发生了变化。

### 4.3 模型字段发生变化

完整基准模型字段：

```text
address_1,state,zip,county,owner,measure_code,measure_name,sample,state_average
```

OD/PVD-only 模型字段：

```text
state,zip,county,phone,type,emergency_service,condition,measure_code,measure_name,score
```

关闭 RVD 后：

1. 失去 `address_1`、`owner`、`sample`、`state_average` 四个模型。
2. 新增 `phone`、`type`、`emergency_service`、`condition`、`score` 五个模型。
3. 模型总数从 9 增加到 10。

### 4.4 字段收益与损失

| 字段 | 完整基准 TP/FP | OD/PVD-only TP/FP | 结论 |
| --- | ---: | ---: | --- |
| `phone` | 0/0 | 34/0 | 明显改善 |
| `type` | 0/0 | 32/0 | 明显改善 |
| `emergency_service` | 0/0 | 16/0 | 改善 |
| `condition` | 0/0 | 32/9 | 明显改善 |
| `score` | 0/0 | 23/0 | 明显改善 |
| `measure_code` | 23/0 | 29/8 | 召回改善，增加误报 |
| `zip` | 30/1 | 30/0 | 小幅改善 |
| `address_1` | 31/8 | 0/0 | 明显退化 |
| `owner` | 27/0 | 0/0 | 明显退化 |
| `sample` | 17/15 | 0/0 | 退化 |
| `state_average` | 16/12 | 0/0 | 退化 |
| `county` | 39/0 | 38/23 | 精确率明显退化 |

新增字段的真实检出收益大于丢失字段和新增误报，所以总体 F1 提高。

## 五、适用边界

### 5.1 适合关闭 RVD 的情况

1. 业务明确只关心单字段异常和格式异常。
2. 错误主要是拼写、字符、长度、类型、占位符、低频值或数值离群。
3. 字段之间没有稳定、可信的函数依赖。
4. RVD 命中量远高于 OD、PVD，导致特征过密。
5. 优先目标是缩短 Spark 作业时间和减少特征内存。

hospital 符合这些条件。

### 5.2 不适合关闭 RVD 的情况

1. 需要检测城市与省份、代码与名称、机构与地址等跨字段一致性。
2. 单字段值本身合法，但字段组合不可能或不一致。
3. 数据中存在明确的一对一或多对一业务规则。
4. OD、PVD 无法仅靠字段自身发现关系错误。

这些场景关闭 RVD 会直接失去关系异常检测能力。

### 5.3 当前结果不能证明所有数据集都会提高 F1

本次随机种子相同，但关闭 RVD 改变了聚类和主动采样。要严格评价 RVD 的模型贡献，还需要使用同一批冻结直接标签分别训练：

1. OD、PVD、RVD 全量特征。
2. 仅 OD、PVD 特征。

这样才能排除采样差异。目前可以确认的是：对于完整 Java 主动学习流程，hospital 关闭 RVD 后性能和效果都更好。

## 六、生产配置建议

如果业务确定不需要 RVD，应在策略计划生成阶段关闭：

```properties
raha.strategy.families=OD,PVD
```

不要只把 RVD 模型权重设为 0。权重设为 0 仍会生成和执行 342 个 RVD，无法获得本次性能收益。

配置变更后必须：

1. 重新生成策略计划。
2. 重新组装特征字典。
3. 重新聚类和采样。
4. 重新训练并发布模型。
5. 不能复用原 OD、PVD、RVD 模型，因为策略计划版本和特征字典版本已经变化。

建议按数据集或任务配置策略族，不建议全局永久删除 RVD 代码。生产可以保留两种配置：

| 场景 | 配置 |
| --- | --- |
| 单字段质量检测 | `OD,PVD` |
| 跨字段一致性检测 | `OD,PVD,RVD` |

## 七、最终建议

如果当前需求明确只关心 OD、PVD，可以关闭 RVD。hospital 实测表明：

1. 特征准备耗时减少 62.91%，提升到原来的 2.70 倍。
2. 端到端耗时减少 41.26%，提升到原来的 1.70 倍。
3. 策略命中减少 98.94%，特征数减少 71.00%。
4. 全量 F1 从 0.585678 提升到 0.669056。

因此，对 hospital 当前数据和错误类型，RVD 的计算成本明显高于实际收益，关闭是合理的。

生产上应将 RVD 做成按数据集启停的可选能力，而不是删除实现。上线前至少再选择一个具有明确跨字段依赖的数据集做对照，确认该数据集是否仍需要 RVD。
