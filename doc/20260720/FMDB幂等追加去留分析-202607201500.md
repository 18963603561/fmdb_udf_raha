# FMDB 幂等追加去留分析

## 1. 结论摘要

不建议把 FMDB 九张标准物理表的幂等追加全局去掉并改成无条件直接插入。

当前 FMDB 基于 Spark SQL 追加 ORC 表实现，物理表本身没有在代码中体现唯一约束、事务主键或原子合并能力。`appendIdempotent` 在应用层承担了“同一业务主键只写一次”的职责。它确实有性能成本，但直接删除会把系统从“至少可重试”退化成“重复写入后需要人工清理”。

更合适的方向不是简单直接插入，而是按表和场景分层优化：

- 元数据、状态、标注、采样、模型表继续保留幂等保护。
- 高数据量派生表优先改成批次级幂等、分区替换或批次提交清单。
- 开发和 toy 验证可以提供显式不安全直插模式，但必须先清理目标表或目标分区，并且禁止用于生产。

一句话判断：幂等检查是正确性保护，性能问题要优化检查范围和写入协议，不能直接拿掉保护。

## 2. 当前代码位置和实现方式

当前 FMDB 写入统一收口在：

`src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/gateway/SparkSqlFmdbTableGateway.java`

核心方法：

```java
appendIdempotent(String tableName, Dataset<Row> rows, List<String> keyColumns)
```

主要流程如下：

1. 校验目标表名、写入数据和业务主键字段。
2. 如果目标表不存在，直接 `saveAsTable`。
3. 如果目标表存在，读取目标表 `existing`。
4. 校验目标表和待写入数据的字段兼容性。
5. 将待写入数据按目标表字段顺序对齐为 `incoming`。
6. 按 `dataset_id`、`partition_month`、`partition_date`、`training_batch_id` 等字段裁剪 `existing`。
7. 从裁剪后的 `existing` 中取业务主键并去重。
8. 用 `left_anti` 过滤掉已经存在的主键，只保留 `pending`。
9. 对 `pending` 计数。
10. 有新增数据时再执行追加写入。

性能主要消耗在第 3、6、7、8、9、10 步，尤其是目标表历史分区多时。

## 3. 为什么追加前要查业务主键

Spark SQL 对普通 ORC 表的追加写入不会自动保证业务唯一性。即使表结构里字段名叫 `job_id`、`sample_batch_id`、`model_version`，它们也只是普通字段，不是数据库主键。

因此，不查业务主键会带来两个问题：

1. 同一批数据重复执行时会重复落库。
2. 后续读取、计数、恢复和训练会把重复记录当成真实记录。

当前项目不是单纯“写结果表”，而是有任务幂等、阶段重试、状态快照、训练物化、检测验证等机制。代码中已经明确存在重试和恢复设计：

- `RahaJobOrchestrator.submit` 根据幂等键识别重复任务提交。
- `RahaJobOrchestrator.execute` 支持阶段失败后重试。
- `StageCheckpointRunner` 支持检查点复用和可恢复失败重试。
- `FmdbResultPersistenceVerifier` 会按物理表记录数校验采样、训练、检测结果。
- `TrainingArtifactMaterializationService` 会在训练物化后校验目标表记录数是否等于本次期望数量。

这说明 FMDB 物理表并不是一次性临时输出，而是任务恢复和结果审计的一部分。

## 4. 九张表是否都需要幂等追加

下面按当前代码中的写入主键和重复影响分析。

| 物理表 | 当前业务主键 | 数据性质 | 是否建议保留幂等 | 直接插入后的主要风险 |
| --- | --- | --- | --- | --- |
| `dw.raha_sample_record` | `sample_batch_id, row_id` | 采样 c1 行快照 | 建议保留 | 同一采样批次重复行，标注模板重复，批次回读数量异常 |
| `dw.raha_annotation_record` | `annotation_batch_id, row_id` | 人工标注批次 | 建议保留 | 重复标注行污染训练标签，重复导入或并发导入时风险高 |
| `dw.raha_training_column_artifact` | `training_batch_id, column_name` | 训练列级产物 | 建议保留或批次级幂等 | 列画像、特征字典、策略计划重复，训练验证数量异常 |
| `dw.raha_training_cell` | `training_batch_id, dataset_id, cell_id` | 全量训练单元格中间状态 | 若开启则必须保留或改批次替换 | 数据量大且重复后恢复特征、聚类、标签会膨胀 |
| `dw.raha_training_example` | `model_set_version, row_id, column_name` | 最终入模样本 | 必须保留或改批次级幂等 | 训练样本重复，模型训练权重被放大，结果不可信 |
| `dw.raha_model_artifact` | `model_set_version, column_name, model_version, state_version` | 模型载荷和状态快照 | 建议保留 | 同版本模型重复，发布状态读取和唯一发布模型校验变复杂 |
| `dw.raha_detection_result` | `detection_batch_id, cell_id, model_version` | 检测错误单元格 | 必须保留或改批次级幂等 | 错误数量虚高，导出、告警和结果查询重复 |
| `dw.raha_job_run` | `job_id, state_version` | 任务状态快照 | 必须保留 | 任务状态快照重复，幂等提交和恢复结果不稳定 |
| `dw.raha_job_stage_attempt` | `job_id, stage_id, attempt_id, state_version` | 阶段尝试状态 | 必须保留 | 阶段审计重复，恢复和失败定位结果混乱 |

需要注意：`dw.raha_training_cell` 默认配置为不保存，但一旦开启，它通常是最大数据量表，反而最不能用无保护直插。

## 5. 不检查会出现哪些具体问题

### 5.1 采样重复

采样批次标识由数据集、快照、采样版本和行标识确定性生成。相同输入重跑时会得到相同 `sample_batch_id`。

如果第一次写入成功，但应用在后续验证、任务状态保存或进程退出时失败，再次重跑会重新尝试写同一个 `sample_batch_id`。如果直接插入，同一 `row_id` 会出现两次。

影响：

- 标注模板可能出现重复样本。
- `findByBatchId` 这类全局批次定位会更容易遇到异常。
- 结果验证按普通 `count` 统计，会发现实际数量大于期望数量。

### 5.2 标注重复

标注导入有文件指纹防重，但这是导入服务层的前置检查，不等于物理写入层的唯一约束。

仍可能重复的情况：

- 两个进程并发导入同一个文件，同时通过指纹检查。
- 标注批次写入成功后，应用在返回结果前异常退出。
- 写入过程中上层重试同一个导入请求。

影响：

- 同一行标注重复进入训练。
- 训练读取最新标注批次时可能得到重复行。
- 同一 `annotation_batch_id` 的物理数量异常。

### 5.3 训练样本重复

训练样本表最敏感，因为它直接影响模型训练。

`model_set_version` 是根据数据集、快照、策略计划、配置指纹和训练批次确定性生成的。同一训练输入重跑时，最终训练样本业务键大概率相同。

如果直接插入，同一个 `model_set_version + row_id + column_name` 会重复出现。

影响：

- 模型训练读取冻结样本时样本数量膨胀。
- 某些正例或负例被重复计数，模型边界偏移。
- 训练物化后的数量校验失败。
- 后续复现训练读到的不是原始训练集，而是重复污染后的训练集。

### 5.4 检测结果重复

检测结果表只保存错误单元格，但同一检测批次重跑时 `detection_batch_id` 和 `cell_id` 可能相同。

如果直接插入：

- 同一个错误单元格会重复出现。
- `find(jobId, cellId)` 当前发现多条会抛出异常。
- 错误统计虚高。
- 下游导出和人工处理会重复处理同一个错误。

### 5.5 任务状态和阶段状态重复

任务状态和阶段状态是追加式快照，不是简单结果表。

直接插入会使同一 `state_version` 可能出现多条快照。即使内容相同，审计和恢复也会变得不干净。如果内容因为时间字段或错误消息差异而不同，问题会更严重。

影响：

- 重复提交识别依赖 `raha_job_run`，该表被写脏后会影响任务幂等。
- 阶段恢复依赖 `raha_job_stage_attempt`，重复快照会增加恢复判断复杂度。
- 失败定位和统计可能被重复记录放大。

## 6. 重复场景和概率估算

代码本身不能给出真实概率。真实概率需要线上指标，例如写入失败率、重试次数、人工重跑次数、多实例并发量、Spark 作业失败率。下面是基于当前代码路径和常见 Spark 运维场景的工程估算。

概率等级说明：

- 极低：正常单次执行几乎不会发生。
- 低：需要异常、网络抖动或少量人工重复操作。
- 中：在失败恢复、并发提交或测试重跑中经常可能遇到。
- 高：该操作模式本身就会触发重复业务键。

| 场景 | 是否会产生相同业务键 | 概率估计 | 说明 |
| --- | --- | --- | --- |
| 空表首次执行且全流程成功 | 否或极少 | 极低 | 这时直接插入通常没问题 |
| 同一配置重复提交 | 是 | 中到高 | 任务幂等键会识别重复请求，若绕过或状态未完整保存，物理写入可能重入 |
| 手工停止后台任务后同参数重跑 | 是 | 高 | 旧的部分物理数据可能已经写入，新任务会再次写同一批业务数据 |
| 阶段失败后自动重试 | 是 | 中 | 默认配置允许阶段重试一次，副作用已提交但阶段返回失败时最危险 |
| Spark 写入成功但应用未收到成功结果 | 是 | 低到中 | 分布式写入常见问题，单次概率不一定高，但属于必须防护的故障模型 |
| 多服务实例并发提交同一请求 | 是 | 中 | 单 JVM `synchronized` 不能防跨进程竞争 |
| 标注文件重复上传 | 是 | 低 | 文件指纹可拦截大多数重复，但并发窗口和异常退出仍存在 |
| 训练或检测物理验证失败后人工重跑 | 是 | 高 | 已写入的派生产物会被再次写入 |
| 目标分区预先清理后再跑 | 否 | 极低 | 这是开发测试直插能成立的前提之一 |

对于当前“停掉后台测试任务后重新跑”的场景，如果旧任务已经写入部分 FMDB 表，又没有清理对应批次或分区，则重复概率不能按随机小概率看，而应视为高风险。因为同一输入会生成相同或高度相关的业务键。

## 7. 能不能避免重复

可以避免，但需要明确采用哪种写入语义。

### 7.1 方式一：继续使用行级幂等

这是当前方案。

优点：

- 正确性最好。
- 对部分写入、重复提交、阶段重试都安全。
- 不要求 FMDB 底层提供唯一约束。

缺点：

- 需要扫描目标表已有业务键。
- Spark 会执行 join、distinct、count 和 write，成本较高。
- 当前实现中 `pending.count()` 和后续写入可能导致计算重复触发。

适用：

- 生产默认模式。
- 业务不可重复的数据表。
- 没有外部事务主键能力的 ORC 表。

### 7.2 方式二：批次级幂等

思路是把“逐行业务键去重”改成“整个批次要么已完整存在，要么完全插入”。

典型流程：

1. 根据批次字段读取目标范围，例如 `sample_batch_id`、`annotation_batch_id`、`training_batch_id`、`model_set_version`、`detection_batch_id`。
2. 只做数量和必要校验。
3. 如果目标批次数量等于期望数量，认为已经完整写入，直接返回已存在。
4. 如果目标批次数量为零，直接插入。
5. 如果目标批次数量介于零和期望之间，认为是脏批次，失败并要求清理或修复。

优点：

- 比行级 anti join 便宜很多。
- 保留重试安全性。
- 非常适合采样、训练物化、检测结果这类批次型数据。

缺点：

- 要求批次内写入是全量语义，不适合真正增量补行。
- 需要调整部分调用方对 `writtenCount` 的理解。
- 遇到半批次残留时需要明确清理策略。

适用表：

- `dw.raha_sample_record`
- `dw.raha_annotation_record`
- `dw.raha_training_column_artifact`
- `dw.raha_training_cell`
- `dw.raha_training_example`
- `dw.raha_detection_result`

### 7.3 方式三：分区替换或批次删除后插入

如果 FMDB 底层支持可靠的事务删除、分区覆盖或按条件覆盖，可以在写入前删除本批次范围，然后直接插入。

优点：

- 写入路径快。
- 不需要逐行 join。
- 重跑同一批次时最终结果唯一。

缺点：

- 删除和插入必须具备原子性或可恢复机制。
- 如果删除成功、插入失败，会留下空批次或半批次。
- 对同一分区内其他批次必须安全，不能误删。

适用：

- `dataset_id + training_batch_id` 这种分区刚好等于批次边界的表。
- 支持事务覆盖的 FMDB 表。

不建议直接用于：

- 只有 `dataset_id + partition_month` 的表，除非删除条件能精确到批次字段。

### 7.4 方式四：临时表加提交清单

先把本批次数据写到临时位置，再写一条提交清单或完成标记。读取时只读取已提交批次。

优点：

- 可以规避部分写入。
- 能支持快速直插临时数据。
- 批次提交状态清晰。

缺点：

- 改造较大。
- 读路径需要识别提交清单。
- 需要垃圾清理临时数据。

适用：

- 未来生产级高吞吐写入。
- 希望避免行级去重扫描的长期方案。

### 7.5 方式五：底层唯一约束或合并语义

如果 FMDB 平台支持类似主键、唯一索引、事务合并或 `MERGE INTO`，可以把幂等职责下沉到底层。

优点：

- 应用代码更简单。
- 性能可能由存储层优化。

缺点：

- 当前代码和建表 SQL 没体现这类约束。
- 需要确认 FMDB 真实能力和失败语义。
- 仍需处理并发、部分提交和冲突错误。

适用：

- FMDB 平台明确支持并验证过唯一性约束的生产环境。

## 8. 是否可以直接插入

可以，但只能在受控条件下使用，不能作为默认生产策略。

直接插入成立的前提：

1. 目标表为空，或目标批次、目标分区已被明确清理。
2. 本次运行不会和其他实例并发写同一批次。
3. 禁用或接受阶段重试带来的重复风险。
4. 写入失败后不自动重跑，或重跑前先清理残留。
5. 结果表只用于一次性验证，不用于后续训练、检测、审计和恢复。

适合场景：

- 本地 toy 验证。
- 一次性性能压测。
- 已显式清表的临时环境。

不适合场景：

- 生产任务。
- 有人工标注导入的任务。
- 需要后续训练复现的任务。
- 需要任务状态恢复的任务。
- 多实例或后台任务可能被停止后重跑的环境。

## 9. 性能损耗来源

当前幂等追加性能损耗主要来自 Spark 作业层面。

| 成本点 | 说明 | 影响 |
| --- | --- | --- |
| 读取目标表 | 目标表存在时必须读取 `existing` | 历史分区多时最明显 |
| 分区值收集 | 从 `incoming` 收集分区值 | 分区列越多，触发动作越多 |
| 主键去重 | `existing.select(...).distinct()` | 会触发 shuffle |
| 反连接 | `incoming left_anti existingKeys` | 需要 join 计划 |
| 写前计数 | `pending.count()` | 会触发一次 Spark action |
| 追加写入 | `insertInto` 或 `saveAsTable` | 可能再次计算 `pending` |

已经加入的分区裁剪能明显减少 `existing` 的扫描范围，但不能完全消除 Spark action 和 join 成本。

## 10. 推荐优化路线

### 10.1 短期建议

保留当前行级幂等作为默认策略，不要全局改成直接插入。

同时继续做以下优化：

- 已有的按 `incoming` 分区值裁剪 `existing` 应保留。
- 对 `pending` 做缓存或检查能否避免 `count` 和写入重复计算。
- 为幂等写入增加耗时日志，包括目标表、输入行数、裁剪分区、已有主键数、新增行数。
- 对高频大表优先评估批次级幂等。

### 10.2 中期建议

在 `FmdbTableGateway` 增加写入策略，而不是只有一个 `appendIdempotent`。

建议策略：

| 策略 | 语义 | 默认使用场景 |
| --- | --- | --- |
| `IDEMPOTENT_BY_KEY` | 当前行级主键幂等 | 状态、模型、标注、采样默认 |
| `BATCH_SKIP_IF_COMPLETE` | 批次完整则跳过，空批次则直插 | 训练样本、检测结果、采样批次 |
| `REPLACE_BATCH` | 删除或覆盖批次后插入 | 支持事务删除的高数据量表 |
| `DIRECT_APPEND_UNSAFE` | 无检查直接插入 | 仅开发测试，必须显式开启 |

### 10.3 长期建议

把批次提交语义显式建模：

- 增加批次提交清单或完成标记。
- 读路径只读取完成批次。
- 写路径先写临时数据，再提交清单。
- 清理未提交或过期临时数据。

这种方式能比行级 anti join 更适合 Spark 大批量写入。

## 11. 对当前问题的建议答案

如果目标只是让 toy 验证快：

- 可以提供开发专用直插开关。
- 每次运行前清理九张表或清理目标批次。
- 不要把直插作为默认行为提交到生产路径。

如果目标是减少生产性能损耗：

- 不建议直接去掉幂等。
- 优先改成批次级幂等或批次替换。
- 对高数据量表单独优化，低数据量元数据表继续使用当前幂等。

如果目标是“后台任务停掉后重跑”：

- 不应该直插。
- 因为这正是重复概率最高的场景。
- 如果旧任务已经有部分物理写入，直插会把同一业务键再次写入。
- 正确做法是保留幂等，或先清理目标批次再用直插跑临时验证。

## 12. 最终判断

当前代码中，幂等追加不是可有可无的性能装饰，而是用来弥补 Spark ORC 表缺少业务主键约束的应用层保护。

全局去掉后，系统在正常成功路径可能看不出问题，但在重试、手工重跑、并发提交、写入后异常退出等场景会产生重复数据。重复数据会污染训练样本、检测结果、采样标注、任务状态和恢复逻辑。

因此建议：

1. 生产默认保留幂等。
2. 对大表从行级幂等升级为批次级幂等或批次替换。
3. 开发测试允许显式直插，但必须先清理目标数据。
4. 增加写入耗时和重复跳过指标，用真实数据决定后续优化优先级。
