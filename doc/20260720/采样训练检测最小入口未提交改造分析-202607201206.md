# 采样训练检测最小入口未提交改造分析

## 1. 分析范围

本文分析当前工作区未提交代码，与以下改造目标的对应关系：

- 7.1 采样最小入口
- 7.2 训练最小入口
- 7.3 检测最小入口
- 8. 推荐实施顺序

当前未提交变更集中在四类文件：

| 类型 | 代表文件 | 作用 |
| --- | --- | --- |
| 最小任务入口 | `RahaTaskRequestFactory`、`FmdbInputSpec`、`SamplingRequestOptions`、`TrainingRequestOptions`、`DetectionRequestOptions` | 把调用方少量参数解析成完整 `RahaTaskExecutionRequest` |
| 采样与标注批次仓储 | `SampleRecordRepository`、`AnnotationRecordRepository`、`FmdbSampleRecordRepository`、`FmdbAnnotationRecordRepository` | 支持按全局采样批次定位采样数据和最新可训练标注 |
| 模型集合版本 | `ModelSetManifest`、`ModelSetRepository`、`DefaultModelSetRepository`、`RahaColumnModel`、`ModelPersistenceContext` | 将多个列模型绑定成一次不可变发布集合 |
| 检测链路贯穿 | `RahaDetectRequest`、`RahaDetectService`、`PublishedColumnModelLoader`、`DetectionResultSaveContext`、`FmdbDetectionResultRepository` | 检测时显式指定 `modelSetVersion` 并持久化 |

## 2. 7.1 采样最小入口对应关系

### 2.1 新增 FMDB 表名规范化和 datasetId 生成器

对应代码：

- `src/main/java/com/fiberhome/ml/raha/service/task/FmdbInputSpec.java`
  - `datasetIdFromTable(String tableName)`
  - `table(String tableName)`

改造原因：

原来如果调用方只传 FMDB 表名，系统缺少稳定的逻辑 `datasetId`。直接把原始表名作为数据集标识会受到大小写、空格、反引号、双引号等写法影响，导致同一张物理表生成多个逻辑数据集。新增规范化生成器后，表名只在物理读取时保留原文，逻辑标识统一转为稳定格式，便于幂等、快照、采样批次、模型版本和检测结果关联。

当前实现特征：

- 表输入使用 `fmdb-table:` 前缀，避免与调用方自定义 SQL 逻辑 `datasetId` 冲突。
- 按点号分层处理完整表名，去除外层引用符号。
- 规范化结果使用小写，保证大小写不敏感场景下稳定。

### 2.2 新增 FmdbInputSpec 或等价内部构建方法

对应代码：

- `src/main/java/com/fiberhome/ml/raha/service/task/FmdbInputSpec.java`
- `src/main/java/com/fiberhome/ml/raha/data/loader/DataLoadRequest.java`

改造原因：

采样、训练、检测都需要描述“读哪里、算作哪个数据集、用什么行身份规则、是否指定版本”。如果这些字段散落在不同请求构造器里，后续很容易出现采样和训练使用不同 `datasetId`、检测和模型集合使用不同身份规则的问题。`FmdbInputSpec` 把 FMDB 表和只读 SQL 的输入规则抽成同一内部规格，再统一转换为 `DataLoadRequest`。

当前实现特征：

- 只允许 `FMDB_TABLE` 与 `FMDB_SQL`。
- SQL 入口要求以 `SELECT` 或 `WITH` 开头，降低写入型 SQL 被误传的风险。
- 支持行身份规则、快照、来源版本、列白名单、列黑名单、敏感列和数据源选项。
- `includedColumns` 与 `excludedColumns` 做冲突校验。

### 2.3 从 RahaConfigFactory 获取完整默认值，只覆盖调用方明确提供的字段

对应代码：

- `src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskRequestFactory.java`
  - `sampling(FmdbInputSpec input, SamplingRequestOptions options)`
  - `training(List<String> sampleBatchIds, TrainingRequestOptions options)`
  - `detection(FmdbInputSpec input, String modelSetVersion, DetectionRequestOptions options)`
- `src/main/java/com/fiberhome/ml/raha/service/task/SamplingRequestOptions.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/TrainingRequestOptions.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/DetectionRequestOptions.java`

改造原因：

最小入口的目标不是绕过配置体系，而是减少调用方必须理解的参数数量。默认值仍应来自 `RahaConfigFactory`，否则容易出现“完整入口”和“最小入口”两套默认值漂移。当前代码先调用 `configFactory.jobConfig(...)` 生成完整配置，再只覆盖调用方显式提供的采样预算、快照、传播方法、模型名前缀、缺失模型策略等字段。

收益：

- 新入口与老入口使用同一套默认配置来源。
- 调用方没有传的参数不会被构造器硬编码成另一套默认值。
- 后续配置项调整时，最小入口不需要同步维护大量默认值。

### 2.4 明确 sourceVersion 与 snapshotId 的优先级

对应代码：

- `src/main/java/com/fiberhome/ml/raha/data/loader/metadata/SnapshotMetadataFactory.java`
- `src/main/java/com/fiberhome/ml/raha/data/loader/metadata/DatasetContentFingerprinter.java`
- `src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/FmdbDatasetLoader.java`
- `src/main/java/com/fiberhome/ml/raha/data/loader/FileRahaDatasetLoader.java`

当前优先级：

1. 调用方显式传入 `snapshotId` 时，直接使用该值。
2. 未传 `snapshotId` 但传入 `sourceVersion` 时，用 `datasetId + sourceVersion + schemaHash + rowCount` 生成稳定快照。
3. `snapshotId` 和 `sourceVersion` 都未传时，读取数据内容指纹，以 `content-<fingerprint>` 参与快照生成。
4. 内容指纹也不可用时，降级为加载时间戳，稳定性最弱。

改造原因：

采样、训练和检测必须能追溯“这次处理的到底是哪一版数据”。如果只用当前时间生成快照，同一份数据每次加载都会产生不同快照，训练和检测无法复现；如果只用表名，又无法区分同表不同版本数据。当前优先级让平台快照优先，业务来源版本其次，最后才用内容指纹或时间兜底。

### 2.5 分别提供表和 SQL 方法，避免不可靠的文本猜测

对应代码：

- `RahaTaskRequestFactory.samplingTable(String tableName)`
- `RahaTaskRequestFactory.samplingSql(String datasetId, String sql)`
- `RahaTaskRequestFactory.detectionTable(String inputReference, String modelSetVersion)`
- `RahaTaskRequestFactory.detectionSql(String datasetId, String sql, String modelSetVersion)`
- `FmdbInputSpec.table(...)`
- `FmdbInputSpec.sql(...)`

改造原因：

用字符串猜测“这是表名还是 SQL”不可靠。例如表名可能包含关键字，SQL 可能带换行、注释、公共表表达式。拆成表方法和 SQL 方法后，调用方显式表达意图，系统不需要做脆弱的文本推断。

## 3. 7.2 训练最小入口对应关系

### 3.1 为采样仓储增加按全局批次标识查询的方法

对应代码：

- `src/main/java/com/fiberhome/ml/raha/repository/port/SampleRecordRepository.java`
  - `findByBatchId(String sampleBatchId)`
- `src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbSampleRecordRepository.java`
  - 覆盖 `findByBatchId`
- `src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskRequestFactory.java`
  - `training(String sampleBatchId)`

改造原因：

训练最小入口只让调用方传 `sampleBatchId`。但旧的采样查询通常需要 `datasetId + partitionMonth + sampleBatchId`，这要求调用方记住采样记录的物理分区，违背“最小入口”的目标。新增按全局批次查询后，训练入口可以从仓储反查数据集、月份分区、来源表或 SQL、行身份规则等信息。

### 3.2 明确采样批次标识的全局唯一性约束

对应代码：

- `SampleRecordRepository.findByBatchId(String sampleBatchId)` 注释说明该标识为全局唯一。
- `RahaTaskRequestFactory.training(...)` 直接以 `sampleBatchId` 作为唯一入口。
- `FmdbSampleRecordRepository.findByBatchId(...)` 在 FMDB 侧定位完整批次。

改造原因：

如果 `sampleBatchId` 只在某个 `datasetId` 或月份下唯一，最小训练入口会出现歧义：同一个批次号可能对应不同数据集或不同月份。全局唯一约束是“只传一个批次号就能训练”的前提。

注意：

当前代码层面已经把全局唯一作为接口语义，但是否在 FMDB 物理表上有唯一索引或等价约束，需要继续核对建表脚本和平台能力。如果没有物理约束，仍可能出现脏数据导致解析不唯一。

### 3.3 自动选择最新有效标注修订，并明确 PARTIAL 策略

对应代码：

- `src/main/java/com/fiberhome/ml/raha/repository/port/AnnotationRecordRepository.java`
  - `findLatestTrainableForSample(String sampleBatchId, boolean allowPartial)`
- `src/main/java/com/fiberhome/ml/raha/repository/adapter/fmdb/repository/FmdbAnnotationRecordRepository.java`
  - 覆盖 `findLatestTrainableForSample`
- `src/main/java/com/fiberhome/ml/raha/service/task/TrainingRequestOptions.java`
  - `allowPartialAnnotation`
- `src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskRequestFactory.java`
  - `validateAnnotationStatus(...)`

当前策略：

- 默认 `TrainingRequestOptions.defaults()` 中 `allowPartialAnnotation=false`，只接受 `IMPORTED`。
- 显式允许时，`PARTIAL` 可作为可训练标注批次。
- 工厂会再次校验状态，避免仓储实现返回不符合策略的批次。

改造原因：

同一个采样批次可能被多次上传标注修订。训练必须选定一个确定版本，否则同一个 `sampleBatchId` 在不同时间训练出不同模型。自动选择最新有效标注修订可以减少调用参数，同时保证结果可解释。默认不接受 `PARTIAL` 是保守策略，避免用不完整标注污染训练；允许 `PARTIAL` 时必须由调用方显式承担风险。

### 3.4 从采样记录还原输入来源和行身份规则

对应代码：

- `src/main/java/com/fiberhome/ml/raha/sampling/domain/SampleRecord.java`
  - 新增来源版本或来源类型相关字段
- `src/main/java/com/fiberhome/ml/raha/sampling/service/SampleRecordService.java`
  - 保存采样时的输入来源、模式、行身份规则
- `RahaTaskRequestFactory.resolveTrainingInput(...)`
- `TrainingMergeRequest`
  - 新增 `RowIdentityConfig`
  - 新增 `TrainingBatchReference` 列表

改造原因：

训练时需要重新加载原始输入，再把采样记录与标注合并成训练样本。若训练阶段重新猜测表名、SQL、行主键或内容哈希规则，就可能找不到当时采样的行，或者把标注贴到错误行上。当前改造从采样记录还原输入来源和行身份规则，保证训练使用与采样一致的行定位规则。

### 3.5 第一阶段支持单批次，第二阶段再支持多批次合并

对应代码：

- `RahaTaskRequestFactory.training(String sampleBatchId)` 提供单批次最小入口。
- `RahaTaskRequestFactory.training(List<String> sampleBatchIds, TrainingRequestOptions options)` 已有多批次方法雏形。
- `TrainingMergeRequest` 已改为 `List<TrainingBatchReference>`。
- `TrainingInputMergeService` 已出现批次列表和上下文中 `sampleBatchIds`、`annotationBatchIds` 的处理。

当前状态判断：

代码已经开始为多批次打地基，但你给的推荐顺序是第一阶段先做单批次。原因是多批次合并不仅是多读几批数据，还需要定义跨批次冲突策略：同一行同一列多次标注冲突怎么办、不同批次来源版本不一致怎么办、同一业务键跨快照是否允许合并。当前 `resolveTrainingInput(...)` 已经对数据集、来源类型、模式哈希、来源版本和行身份规则做一致性校验，这是为了先拦住高风险合并。

## 4. 7.3 检测最小入口对应关系

### 4.1 新增 ModelSetManifest 及其仓储接口

对应代码：

- `src/main/java/com/fiberhome/ml/raha/model/domain/ModelSetManifest.java`
- `src/main/java/com/fiberhome/ml/raha/repository/port/ModelSetRepository.java`
- `src/main/java/com/fiberhome/ml/raha/repository/adapter/DefaultModelSetRepository.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskApplicationServiceFactory.java`
  - 装配 `DefaultModelSetRepository`

改造原因：

检测不能只说“用最新模型”。Raha 是列级模型，整表检测需要一组列模型具有共同的数据集、模式、策略计划、行身份规则和发布状态。`ModelSetManifest` 把这些列模型聚合成一个不可变清单，检测入口只需指定 `modelSetVersion`，系统即可知道应该加载哪一套模型。

### 4.2 为模型仓储增加按 modelSetVersion 查询整套列模型的能力

对应代码：

- `ModelMetadataRepository.findByModelSetVersion(String modelSetVersion)`
- `DefaultModelMetadataRepository.findByModelSetVersion(...)`
- `FmdbModelMetadataRepository.findByModelSetVersion(...)`
- `PublishedColumnModelLoader.load(String modelSetVersion, String columnName, ...)`

改造原因：

按列查“当前已发布模型”会产生模型集合漂移：一张表的某些列可能来自昨天训练，另一些列来自今天训练。按 `modelSetVersion` 查询整套列模型，可以保证检测使用同一次训练发布出来的模型集合。

### 4.3 将 modelSetVersion 贯穿请求、检测阶段、模型加载器和结果持久化

对应代码：

- `RahaTaskExecutionRequest`
  - 新增 `modelSetVersion`、`missingModelPolicy`
- `RahaDetectRequest`
  - 新增 `modelSetVersion`、`missingModelPolicy`
- `PublishedModelDetectionStageHandler`
  - 将 `modelSetVersion` 传给检测服务
- `RahaDetectService`
  - 按 `modelSetVersion + columnName` 加载模型
- `PublishedColumnModelLoader`
  - 新增按模型集合加载指定列模型的方法
- `DetectionResultSaveContext`
  - 新增 `modelSetVersion`
- `FmdbDetectionResultRepository`
  - 保存和校验检测结果归属的模型集合版本

改造原因：

如果检测请求里有 `modelSetVersion`，但结果表没有保存它，后续无法审计“这次结果来自哪一套模型”。如果持久化有版本，但模型加载器仍按最新模型加载，也无法保证结果真实可复现。因此必须从请求、阶段、服务、加载、结果持久化整条链路贯穿。

### 4.4 将 modelSetVersion 纳入任务幂等键

对应代码：

- `RahaTaskRequestFactory.detection(...)`
  - `withExecutionInputFingerprint(inputFingerprint("detection", input, modelSetVersion + "|" + options.getMissingModelPolicy()))`
- `RahaTaskExecutionRequest`
  - 检测请求带 `modelSetVersion`

改造原因：

同一张检测表、同一个快照，用不同模型集合检测，应该是两个不同任务。如果幂等键不包含 `modelSetVersion`，系统可能把新模型检测误判为旧任务重复执行，导致结果被跳过或复用错误。

### 4.5 在模型集合中保存行身份规则，并在检测时校验覆盖兼容性

对应代码：

- `RahaColumnModel`
  - 新增 `rowIdentityConfig`
- `ModelPersistenceContext`
  - 新增 `rowIdentityConfig`
- `ColumnModelMetadataFactory`
  - 保存训练输入行身份规则
- `FmdbModelStore`
  - 模型载荷保存行身份字段
- `FmdbModelMetadataRepository`
  - 从载荷恢复行身份字段
- `ModelSetManifest`
  - 校验集合内列模型行身份规则一致
- `RahaTaskRequestFactory.detection(...)`
  - 检测输入行身份规则如有覆盖，必须与模型集合兼容

改造原因：

模型训练时的样本标签依赖行身份规则。如果检测时换了另一套行身份规则，检测结果的行坐标可能无法和训练语义对齐，尤其是内容哈希、业务主键和平台行号混用时风险很高。把行身份规则保存到模型集合并在检测入口校验，可以把错误提前暴露。

### 4.6 校验模型集合的模式哈希、特征字典和策略计划版本完整一致

对应代码：

- `ModelSetManifest`
  - 校验 `datasetId`、`schemaHash`、`strategyPlanVersion`、`rowIdentityConfig` 一致
- `RahaColumnModel`
  - 保存 `schemaHash`、`featureDictionaryVersion`、`strategyPlanVersion`、`modelSetVersion`
- `PublishedColumnModelLoader`
  - 加载时调用兼容性校验
- `ColumnModelCompatibilityValidator`
  - 校验模式哈希、特征字典版本、策略计划版本

改造原因：

特征字典和策略计划共同决定模型输入向量的含义。同样长度的向量，如果字典版本或策略计划版本不同，模型预测就会变成“用旧语义解释新特征”。模式哈希用于保证输入列结构一致。三者必须完整一致，才能认为模型集合可以用于当前检测输入。

## 5. 8. 推荐实施顺序与当前代码状态

### 5.1 先实现 samplingTable、samplingSql 和统一输入规格

当前对应：

- `RahaTaskRequestFactory.samplingTable(...)`
- `RahaTaskRequestFactory.samplingSql(...)`
- `FmdbInputSpec`
- `SamplingRequestOptions`
- `SnapshotMetadataFactory`
- `DatasetContentFingerprinter`

为什么先做：

采样是闭环源头。训练和检测后续依赖采样阶段落下来的 `datasetId`、`sampleBatchId`、`schemaHash`、`sourceVersion`、行身份规则。如果采样入口不先稳定，后面训练入口只能继续猜测来源，模型集合也缺少可靠血缘。

### 5.2 再实现单个 sampleBatchId 的训练解析工厂

当前对应：

- `RahaTaskRequestFactory.training(String sampleBatchId)`
- `SampleRecordRepository.findByBatchId(...)`
- `AnnotationRecordRepository.findLatestTrainableForSample(...)`
- `TrainingBatchReference`
- `TrainingMergeRequest`
- `TrainingInputMergeService`

为什么第二步做：

训练需要消费采样产物和标注产物。只有采样批次可以全局定位之后，训练入口才能从一个 `sampleBatchId` 推导出全部必要参数。默认只接受最新 `IMPORTED` 标注批次，是为了先保证训练数据完整，避免第一阶段引入部分导入和多批次冲突两个变量。

### 5.3 最后建立模型集合清单并实现显式 modelSetVersion 检测

当前对应：

- `ModelSetManifest`
- `ModelSetRepository`
- `ModelMetadataRepository.findByModelSetVersion(...)`
- `PublishedColumnModelLoader.load(modelSetVersion, columnName, ...)`
- `RahaTaskRequestFactory.detectionTable(...)`
- `RahaTaskRequestFactory.detectionSql(...)`
- `RahaDetectService`
- `DetectionResultSaveContext`
- `FmdbDetectionResultRepository`

为什么第三步做：

检测入口依赖训练阶段产出的模型集合版本。如果训练尚未稳定生成 `modelSetVersion` 和模型集合兼容信息，检测阶段即使加了参数也没有可靠对象可加载。放在第三步可以让检测只接受已发布、完整、一致的模型集合。

### 6.1 中文注释疑似乱码

在当前终端读取多份新增或修改文件时，中文注释显示为类似 `灏?FMDB 鏈€灏忓弬鏁` 的乱码。受影响文件至少包括：

- `src/main/java/com/fiberhome/ml/raha/service/task/RahaTaskRequestFactory.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/FmdbInputSpec.java`
- `src/main/java/com/fiberhome/ml/raha/model/domain/ModelSetManifest.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/SamplingRequestOptions.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/TrainingRequestOptions.java`
- `src/main/java/com/fiberhome/ml/raha/service/task/DetectionRequestOptions.java`
- `src/main/java/com/fiberhome/ml/raha/service/train/TrainingBatchReference.java`
- `src/main/java/com/fiberhome/ml/raha/service/train/TrainingMergeRequest.java`

这与项目约定“所有新增/修改注释必须为中文简体、UTF-8 无 BOM、不可乱码”冲突。提交前建议用编辑器确认实际文件编码；如果文件内容确实已乱码，需要修复注释文本并统一保存为 UTF-8 无 BOM、LF。

### 6.2 多批次训练能力已经露出接口，但策略仍需收口

`RahaTaskRequestFactory.training(List<String>...)` 和 `TrainingMergeRequest` 已支持批次列表，但需求中建议第一阶段只支持单批次。建议在第一阶段对外文档和入口上明确“生产推荐只传单个 `sampleBatchId`”，多批次接口暂作为内部演进基础，直到冲突策略完全定义。

### 6.3 全局唯一性需要物理约束兜底

接口注释已声明 `sampleBatchId` 全局唯一，但还需要确认 FMDB 表结构是否有唯一约束、批次索引表或等价去重逻辑。否则一旦出现重复批次号，训练最小入口可能无法确定唯一数据来源。

### 6.4 检测默认策略存在新旧入口差异

新增 `DetectionRequestOptions.defaults()` 默认 `MissingModelPolicy.FAIL`，而部分兼容构造路径仍可见 `PARTIAL` 默认值。建议后续统一说明：显式 `modelSetVersion` 的最小检测入口默认失败优先，历史未显式模型集合入口如需兼容可继续部分成功。

## 7. 一句话结论

这批修改的核心原因是把“调用方少传参数”从简单封装升级为可复现的闭环：采样阶段固定数据来源和行身份，训练阶段从全局采样批次还原来源并选择确定标注修订，检测阶段用不可变 `modelSetVersion` 固定整套列模型，最终让采样、训练、检测结果都能按同一套版本语义追溯和幂等。
