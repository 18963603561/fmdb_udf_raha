# FMDB 九张物理表幂等追加必要性分析

## 1. 结论摘要

FMDB 九张表不应该一刀切全部使用 `appendIdempotent`。是否必须幂等，取决于这张表的数据是否满足以下条件：

- 失败重试、重复提交、服务重启后补偿写入时，可能再次写入同一业务事实；
- 下游查询会按业务主键、状态版本、模型版本或批次版本读取；
- 重复记录会改变训练样本数量、模型状态判断、任务状态判断或用户标注事实；
- 数据不可重算或不可安全覆盖，例如人工标注、任务审计、模型发布状态。

按当前代码和业务语义，建议分成三类：

| 分类 | 表 | 建议 |
| --- | --- | --- |
| 必须或强烈建议保留幂等 | `dw.raha_job_run`、`dw.raha_job_stage_attempt`、`dw.raha_model_artifact`、`dw.raha_annotation_record`、`dw.raha_sample_record` | 保留 `appendIdempotent` |
| 可以去掉幂等，但需要前置约束 | `dw.raha_training_column_artifact`、`dw.raha_training_example` | 可改直接追加，但要关闭训练重试、保证批次唯一、避免验证阶段重复写 |
| 不必须幂等，适合直接追加 | `dw.raha_detection_result`、`dw.raha_training_cell` | 可以直接追加；检测结果已按本轮改造切到直接追加 |

## 2. 当前九表和幂等主键

| 表 | 当前写入位置 | 当前幂等主键 | 数据语义 |
| --- | --- | --- | --- |
| `dw.raha_sample_record` | `FmdbSampleRecordRepository` | `sample_batch_id,row_id` | 采样批次内的样本行 |
| `dw.raha_annotation_record` | `FmdbAnnotationRecordRepository` | `annotation_batch_id,row_id` | 用户标注记录 |
| `dw.raha_training_column_artifact` | `FmdbTrainingArtifactRepository` | `training_batch_id,column_name` | 列画像、策略计划、特征字典等列级训练产物 |
| `dw.raha_training_cell` | `FmdbTrainingArtifactRepository` | `training_batch_id,dataset_id,cell_id` | 全量训练单元格中间明细 |
| `dw.raha_training_example` | `FmdbTrainingArtifactRepository` | `model_set_version,row_id,column_name` | 最终入模训练样本 |
| `dw.raha_model_artifact` | `FmdbModelStore`、`FmdbModelMetadataRepository` | `model_set_version,column_name,model_version,state_version` | 模型载荷和模型状态版本 |
| `dw.raha_detection_result` | `SparkSqlFmdbResultWriter` | `detection_batch_id,cell_id,model_version` | 预测错误结果 |
| `dw.raha_job_run` | `SparkSqlFmdbResultWriter.writeJob` | `job_id,state_version` | 任务状态快照 |
| `dw.raha_job_stage_attempt` | `FmdbStageRepository` | `job_id,stage_id,attempt_id,state_version` | 阶段尝试和状态快照 |

## 3. 必须或强烈建议保留幂等的表

### 3.1 `dw.raha_job_run`

建议：必须保留幂等。

原因：

- 这是任务状态快照表，`job_id,state_version` 表示一次任务状态版本；
- 如果直接追加，同一状态版本重复写入，会让审计记录、状态数量和结果查询变脏；
- 任务状态本身是流程控制数据，不是普通明细数据；
- 当前 `writeJob` 已经会读取最新状态判断是否变化，说明它天然依赖“状态版本唯一”。

如果不幂等的风险：

- 同一个 `job_id,state_version` 出现多条；
- 查询最新状态时虽然可能还能取到正确状态，但状态历史不可信；
- 任务重复提交、服务重启补写、状态写入超时重试时会产生重复快照。

### 3.2 `dw.raha_job_stage_attempt`

建议：必须保留幂等。

原因：

- 阶段状态主键是 `job_id,stage_id,attempt_id,state_version`；
- 当前 `FmdbStageRepository` 写入后要求 `written == 1`，这说明代码期望每个阶段状态版本只写一次；
- 阶段状态用于失败定位、检查点恢复、重试判断和执行审计。

如果不幂等的风险：

- 阶段尝试记录重复；
- 检查点、阶段历史、失败定位不可信；
- 同一阶段状态被重复统计，排查时会误判为多次执行。

### 3.3 `dw.raha_model_artifact`

建议：必须保留幂等。

原因：

- 模型表同时承载模型载荷和模型状态；
- `model_version` 是不可变模型产物标识，重复写相同模型没有业务价值；
- `state_version` 表示模型发布、禁用、恢复等状态流转版本；
- 模型加载、模型集合查询、发布状态判断都依赖这张表。

如果不幂等的风险：

- 同一个模型版本重复出现；
- 同一个模型状态版本重复出现；
- 查询发布模型时可能出现重复候选；
- 模型集合版本和列模型版本的唯一关系变弱，预测加载可能变得不稳定。

注意：

- `FmdbModelStore` 写模型载荷；
- `FmdbModelMetadataRepository` 写模型状态；
- 两者都写 `dw.raha_model_artifact`，所以这张表尤其不适合直接盲追加。

### 3.4 `dw.raha_annotation_record`

建议：强烈建议保留幂等。

原因：

- 标注记录来自用户或外部标注文件，不是可以随便重算的中间数据；
- 主键 `annotation_batch_id,row_id` 表示一个标注批次里一行标注事实；
- 训练合并时如果重复读到标注，可能造成训练样本重复或标签权重异常。

如果不幂等的风险：

- 用户重复上传同一个标注批次会写入重复标注；
- 文件导入失败后重试会重复写入；
- 后续训练把重复标注当成更多证据，影响模型。

只有在以下条件全部满足时，才可以考虑直接追加：

- 标注服务入口禁止重复批次号；
- 上传系统保证同一 `annotation_batch_id` 只处理一次；
- 失败后不允许自动重试；
- 下游训练合并对标注记录按 `annotation_batch_id,row_id` 去重。

当前系统不建议这么冒险。

### 3.5 `dw.raha_sample_record`

建议：强烈建议保留幂等。

原因：

- 采样记录是人工标注任务的上游依据；
- 主键 `sample_batch_id,row_id` 表示一个采样批次中的一行；
- 采样重复会直接导致标注任务重复、样本池重复、训练输入重复。

如果不幂等的风险：

- 同一个采样批次重复写入相同行；
- 标注任务生成重复；
- 训练样本被重复采入；
- 采样数量校验不可信。

如果后续要优化，优先做分区裁剪或批次级写入约束，不建议直接去掉幂等。

## 4. 可以去掉幂等但需要前置约束的表

### 4.1 `dw.raha_training_column_artifact`

建议：不是绝对必须，但建议谨慎。

数据内容：

- 列画像；
- 策略计划；
- 特征字典；
- 聚类摘要；
- 标签传播摘要。

当前主键是 `training_batch_id,column_name`。

为什么不是绝对必须：

- 这些数据属于训练产物，可以通过同一次训练重新生成；
- 如果系统保证一个训练批次只写一次，直接追加是可行的；
- 写入量相对检测结果通常小很多，不一定是最大的性能瓶颈。

为什么仍建议保留或至少做批次约束：

- 训练结果验证会按数量校验列级产物；
- 重复写入会让列级产物数量翻倍；
- 特征字典和策略计划是训练、预测和解释的关键依赖；
- 一旦重复批次进入表里，后续按 `training_batch_id` 查询会多出重复列记录。

折中建议：

- 如果追求性能，可以改成直接追加；
- 但前提是训练任务不允许失败重试；
- 同一个 `training_batch_id` 不允许重复执行；
- 验证阶段不能再次写入；
- 读取时最好按 `training_batch_id,column_name` 做唯一性校验。

### 4.2 `dw.raha_training_example`

建议：不是绝对必须，但不建议无保护直接追加。

数据内容：

- 最终入模训练样本；
- 特征向量；
- 标签；
- 样本权重；
- 模型集合版本。

当前主键是 `model_set_version,row_id,column_name`。

为什么不是绝对必须：

- 它是训练过程产物；
- 如果 `model_set_version` 全局唯一且训练不重试，直接追加不会重复；
- 写完后通常用于复现和审计，不一定参与在线预测主链路。

为什么仍有风险：

- 如果训练阶段失败重试，同一个 `model_set_version` 可能重复写入；
- 重复训练样本会导致物理数量校验失败；
- 后续如果基于该表做模型复现、评估或导出，会把重复样本当成真实样本。

折中建议：

- 可作为第二批优化对象；
- 如果改直接追加，必须保证 `model_set_version` 每次重试重新生成，或者彻底禁止训练重试；
- 更稳的方式是按 `model_set_version` 做批次级唯一约束或分区级替换。

## 5. 不必须幂等，适合直接追加的表

### 5.1 `dw.raha_detection_result`

建议：可以直接追加。

原因：

- 检测结果是预测输出，属于最终明细；
- 用户已经倾向关闭重复提交和失败重试；
- 本轮改造已经去掉了“结果持久化验证阶段二次写入”；
- 默认配置已经改为 `DIRECT_APPEND`；
- 这张表数据量最大，幂等扫描历史主键的性能损耗最明显。

直接追加成立的前提：

- 不允许同一个检测任务重复提交；
- 检测阶段失败后不自动重试；
- 验证阶段只读校验，不再写入；
- `detection_batch_id` 每次真实检测唯一。

如果这些前提不满足，重复场景包括：

- 同一检测任务被重复调用；
- 检测服务写入成功，但上层认为失败并重跑；
- 外部调度系统重复投递；
- toy 或验证代码再次调用 `writeDetectionResults`。

当前本轮已处理其中最关键的“验证阶段二次写入”。

### 5.2 `dw.raha_training_cell`

建议：可以直接追加，甚至默认不入库。

原因：

- 当前默认配置中 `raha.persistence.table.training-cell.enabled=false`；
- 它是全量训练单元格中间状态，数据量大；
- 主要用于调试、审计或中间过程复现；
- 不属于预测加载模型的必要表。

如果启用该表，直接追加的前提：

- `training_batch_id` 每次训练唯一；
- 训练失败不重试，或者重试生成新的 `training_batch_id`；
- 下游读取时能容忍或校验重复。

如果不满足前提，重复会导致：

- 训练单元格数量膨胀；
- 调试和审计误判；
- 如果后续从该表反查训练明细，可能重复读取同一 `cell_id`。

## 6. 推荐保留策略

建议不要把 9 张表统一改成直接追加，而是按表级配置控制。

推荐默认：

| 表 | 推荐写入方式 |
| --- | --- |
| `dw.raha_sample_record` | `appendIdempotent` |
| `dw.raha_annotation_record` | `appendIdempotent` |
| `dw.raha_training_column_artifact` | 暂保留 `appendIdempotent`，后续可评估直接追加 |
| `dw.raha_training_cell` | 默认关闭；启用时可直接追加 |
| `dw.raha_training_example` | 暂保留 `appendIdempotent`，后续可评估直接追加 |
| `dw.raha_model_artifact` | `appendIdempotent` |
| `dw.raha_detection_result` | `appendDirect` |
| `dw.raha_job_run` | `appendIdempotent` |
| `dw.raha_job_stage_attempt` | `appendIdempotent` |

## 7. 性能优化优先级

按投入产出比排序：

1. 优先优化 `dw.raha_detection_result`。
   - 数据量大；
   - 查询历史主键最伤性能；
   - 已经具备直接追加前提；
   - 本轮已经落地。

2. 其次评估 `dw.raha_training_cell`。
   - 默认关闭；
   - 如果未来打开，数据量会很大；
   - 更适合直接追加或不入库。

3. 再评估 `dw.raha_training_example`。
   - 训练样本量可能较大；
   - 但会影响模型复现和验证；
   - 需要训练批次唯一约束配合。

4. 最后评估 `dw.raha_training_column_artifact`。
   - 数据量通常按列数计；
   - 性能压力小于检测结果和训练样本；
   - 但它是模型依赖链路的重要产物，风险更高。

5. 不建议动任务、阶段、模型、采样、标注五类表。
   - 这些表不是主要性能瓶颈；
   - 重复数据的业务副作用明显；
   - 保留幂等更稳。

## 8. 最终建议

当前最合理的落地边界是：

- 检测结果表：直接追加；
- 训练单元格表：默认关闭，启用时可以直接追加；
- 训练样本和列级产物：先保留幂等，后续如果确认训练不重试、批次强唯一，再切直接追加；
- 采样、标注、模型、任务、阶段：保留幂等。

这套边界能解决主要性能问题，同时不会牺牲任务状态、模型加载、标注事实和采样事实的正确性。
