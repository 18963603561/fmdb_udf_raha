# FMDB 取消幂等追加直接写入方案

## 1. 结论

如果目标是“不再出现幂等追加，不再做写入前扫描、主键去重和反连接”，可以改成直接插入。

但直接插入不能天然保证没有重复数据。它只能把当前语义从“同一业务主键只保存一次”改成“每次运行都生成一份新的运行结果”。

因此需要先明确两种重复：

| 重复类型 | 含义 | 直接插入能否避免 |
| --- | --- | --- |
| 物理主键重复 | 同一 `sample_batch_id + row_id` 等业务键重复出现 | 只有所有批次标识都改成运行级唯一时才能避免 |
| 业务语义重复 | 同一输入、同一配置被重复处理，产生多份结果 | 不能避免，只能接受或在查询层按批次隔离 |

如果允许重复提交，又不做幂等追加，那么同一业务输入重复产生多份结果是预期行为，不是异常。

## 2. “允许重复提交 + 不允许失败重试”是否足够

不够。

这两个条件只能降低一部分重复来源，不能消除全部重复来源。

### 2.1 允许重复提交

允许重复提交不会减少重复，反而会让同一配置、同一输入可以多次运行。

如果每次重复提交都生成新的 `job_id`、新的批次标识、新的模型集合版本，那么不会出现同一个业务主键重复，但会出现多份语义相同的结果。

如果批次标识仍然是确定性生成，那么重复提交仍会写出相同业务键，直接插入会产生物理重复。

当前代码中需要注意：

- `sample_batch_id` 由数据集、快照、采样版本和行标识生成，不包含 `job_id`。
- `annotation_batch_id` 由数据集、采样批次和文件指纹生成，不包含导入运行号。
- `model_set_version` 由数据集、快照、策略计划、配置指纹和训练批次生成。
- `detection_batch_id` 当前通常使用 `job_id`，如果每次提交都新建任务，则天然运行级唯一。

因此，如果只允许重复提交，但不改这些标识生成规则，仍然可能重复。

### 2.2 不允许失败重试

不允许失败重试只能关闭应用层自动重试，例如把阶段最大重试次数设为零。

它不能避免：

- 用户手工重新提交同一请求。
- 后台任务被停止后重新执行。
- 写入成功但应用在保存状态或返回结果前崩溃。
- 多实例同时提交同一任务。
- Spark 内部任务失败重跑带来的提交不确定性。
- 标注文件被重复导入。

所以“不允许失败重试”是降低风险，不是消除风险。

## 3. 如果坚持直接写入，必须改成运行级唯一

直接写入要避免物理主键重复，核心是：

每次运行都必须拥有唯一运行标识，并且所有物理结果表的批次标识都必须包含这个运行标识。

建议引入：

```text
run_id
```

或者直接使用：

```text
job_id
```

作为所有派生批次的唯一边界。

## 4. 需要调整的标识生成规则

### 4.1 采样批次

当前 `sample_batch_id` 是确定性批次。

如果直接写入，建议改为：

```text
sample-{job_id}-{短哈希}
```

或：

```text
sample-{run_id}
```

这样同一输入重复提交会生成不同采样批次。

影响：

- 训练最小入口如果只传 `sample_batch_id`，仍然可用。
- 但同一输入会有多个采样批次，需要用户明确选择哪一个。

### 4.2 标注批次

当前 `annotation_batch_id` 由文件指纹确定。这个设计用于防止重复导入。

如果允许重复导入并直接写入，需要改为：

```text
ann-{import_run_id}
```

或：

```text
ann-{sample_batch_id}-{import_run_id}
```

影响：

- 同一个文件重复导入会生成多个标注批次。
- 训练读取“最新可训练标注”时，必须按导入时间选择。
- 人工误重复上传会成为真实数据，而不是重复拦截。

如果不希望人工标注重复，标注导入仍应保留文件指纹防重，只是底层写入可以直接插入。

### 4.3 训练批次和模型集合

训练批次建议使用 `job_id` 或 `run_id`。

`model_set_version` 如果继续确定性生成，同一训练输入重跑会写同一个模型集合版本。直接插入会重复模型行和训练样本。

建议改为：

```text
model_set_version = hash(dataset_id + snapshot_id + plan_version + config + training_batch_id + run_id)
```

或者直接把 `training_batch_id` 保证为运行级唯一。

影响：

- 同一输入重复训练会产生多个模型集合。
- 发布模型时必须明确发布哪一个模型集合。
- 查询“当前已发布模型”必须继续保证同一字段只有一个已发布模型。

### 4.4 检测批次

检测结果的 `detection_batch_id` 使用 `job_id` 时已经比较适合直接写入。

重复提交会产生多个检测批次，而不是同一检测批次重复写入。

影响：

- 结果查询必须按 `detection_batch_id`。
- 不能只按数据集统计，否则会把重复检测批次都算进去。

## 5. 需要关闭的重试能力

如果要降低重复写入风险，应关闭应用层失败重试。

建议配置：

```properties
raha.failure.max-retry-count=0
raha.failure.fail-fast=true
```

含义：

- 阶段失败后不创建新的自动尝试。
- 出现可恢复失败也尽快终止任务。

但这不影响 Spark 自己的任务重试。若要进一步降低内部重试，需要评估 Spark 参数，例如任务失败次数。但这会降低作业容错能力，不建议轻易全局设置。

## 6. 任务提交语义需要二选一

### 6.1 方案一：允许重复提交

语义：

同一配置每次提交都创建新任务。

优点：

- 不需要查 `raha_job_run` 做提交幂等。
- 用户反复点击或重跑都会生成新结果。
- 直接写入更顺。

缺点：

- 同一输入会有多份采样、训练、检测结果。
- 存储会增长更快。
- 查询和发布必须明确批次。

适合：

- 离线实验。
- toy 验证。
- 每次运行都视为独立实验的场景。

### 6.2 方案二：不允许重复提交

语义：

同一配置已存在任务时直接拒绝新任务。

优点：

- 从入口减少重复运行。
- 比允许重复提交更不容易产生语义重复。

缺点：

- 仍然需要查状态表判断是否存在。
- 用户如果想重跑，需要显式创建新运行号或清理旧任务。

适合：

- 更严格的生产任务。
- 不希望同一输入产生多份结果的场景。

如果你的目标是“不要幂等追加”，但又希望尽量不重复，第二种比第一种更安全。

## 7. 推荐直接写入模式

建议新增写入模式，而不是删除 `appendIdempotent`。

```java
public enum FmdbWriteMode {
    IDEMPOTENT_BY_KEY,
    DIRECT_APPEND
}
```

新增网关方法：

```java
long appendDirect(String tableName, Dataset<Row> rows, long expectedCount)
```

实现语义：

1. 校验表名和 schema。
2. 不读取 existing。
3. 不做主键反连接。
4. 不做幂等跳过。
5. 直接 `insertInto` 或 `saveAsTable`。
6. 返回调用方传入的期望数量，或在安全模式下写后计数。

为了最大性能，直接写入不应在写入前执行 `rows.count()`。

当前很多仓储在创建 `Dataset<Row>` 前已经有 `List<Row>`，因此天然知道输入数量。

## 8. 各表直接写入建议

| 物理表 | 是否可直接写 | 前提 |
| --- | --- | --- |
| `dw.raha_sample_record` | 可以 | `sample_batch_id` 改成运行级唯一 |
| `dw.raha_annotation_record` | 谨慎 | 要么继续文件指纹防重，要么接受重复导入 |
| `dw.raha_training_column_artifact` | 可以 | `training_batch_id` 运行级唯一 |
| `dw.raha_training_cell` | 可以 | `training_batch_id` 运行级唯一 |
| `dw.raha_training_example` | 可以 | `model_set_version` 或 `training_batch_id` 运行级唯一 |
| `dw.raha_model_artifact` | 谨慎 | `model_set_version` 和 `model_version` 运行级唯一，发布状态需额外保护 |
| `dw.raha_detection_result` | 可以 | `detection_batch_id` 使用新 `job_id` |
| `dw.raha_job_run` | 可以但不推荐盲写 | 状态保存仍需要避免同一 `job_id + state_version` 重复 |
| `dw.raha_job_stage_attempt` | 可以但不推荐盲写 | 阶段状态仍需要避免同一尝试版本重复 |

状态表即使直接写，也建议保留“状态未变化则跳过”的逻辑。否则同一状态保存两次会污染审计。

## 9. 查询口径必须改变

直接写入后，不能再假设一个数据集、一个快照、一个配置只有一份结果。

查询必须显式带上：

- `job_id`
- `sample_batch_id`
- `annotation_batch_id`
- `training_batch_id`
- `model_set_version`
- `detection_batch_id`

否则同一业务输入的多次运行结果会混在一起。

例如检测结果统计必须按 `detection_batch_id`，不能只按 `dataset_id + partition_date`。

## 10. 仍可能重复的场景

即使关闭失败重试并使用直接写入，仍可能出现重复或多份结果：

| 场景 | 结果 |
| --- | --- |
| 用户重复提交 | 生成多份语义相同结果 |
| 用户重复导入同一标注文件 | 如果放开防重，会生成多份标注批次 |
| 任务写入成功但状态保存失败 | 用户重跑会生成新批次或重复批次，取决于标识生成 |
| 多实例并发提交 | 如果没有入口锁，会产生多份运行 |
| Spark 提交不确定 | 仍需依赖底层提交协议保证单次写入完整性 |

因此，直接写入的正确表述不是“不会重复”，而是“允许多次运行，每次运行隔离”。

## 11. 最小改造方案

如果当前目标是尽快去掉 FMDB 幂等追加，可以按最小改造做：

1. 新增 `appendDirect`。
2. 增加全局配置：

```properties
raha.fmdb.write-mode=DIRECT_APPEND
```

3. 关闭应用层重试：

```properties
raha.failure.max-retry-count=0
raha.failure.fail-fast=true
```

4. 让重复提交创建新任务，不再复用旧任务。
5. 把 `sample_batch_id` 改成包含 `job_id`。
6. 确认 `training_batch_id` 使用运行级唯一值。
7. 确认 `model_set_version` 随训练运行变化。
8. 检测结果继续使用新 `job_id` 作为 `detection_batch_id`。
9. 标注导入保留文件指纹防重，避免人工数据重复。

这样可以去掉大多数表的写入前扫描。

## 12. 更安全的直接写入方案

如果希望减少重复语义，建议：

1. 不允许重复提交。
2. 同一配置已存在任务时直接拒绝。
3. 用户需要重跑时必须显式传 `forceNewRun=true`。
4. 只有 `forceNewRun=true` 时才生成新批次。

这种方式没有幂等追加，但入口仍然可控。

## 13. 对当前需求的建议

你现在明确希望“不出现幂等追加”，我建议选择下面这个语义：

```text
FMDB 写入改为直接追加；
每次提交都是独立运行；
所有结果必须按运行批次隔离；
应用层不自动失败重试；
标注导入保留文件指纹防重；
状态表保留最小状态变化判断；
不承诺同一输入只产生一份结果。
```

这套语义是自洽的。

但需要接受：

- 同一输入重复提交会产生多份业务结果。
- 存储增长会更快。
- 查询必须带批次。
- 后续如果要清理，需要按批次清理。

如果你接受这些边界，就可以推进代码改造，完全绕开 `appendIdempotent` 的历史扫描和反连接。
