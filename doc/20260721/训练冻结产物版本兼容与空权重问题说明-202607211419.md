# 训练冻结产物版本兼容与空权重问题说明

## 背景

本说明针对以下三处代码中的注释和告警进行解释：

- `src/main/java/com/fiberhome/ml/raha/model/release/ColumnModelCompatibilityValidator.java`
- `src/main/java/com/fiberhome/ml/raha/service/train/TrainingArtifactMaterializationService.java`
- `src/main/java/com/fiberhome/ml/raha/strategy/plan/StrategyPlanService.java`

这些问题的共同背景是：训练阶段会把策略计划、特征字典、训练样本和模型元数据冻结下来；检测阶段可能基于同一个数据快照重新准备特征，也可能加载已经发布的模型进行预测。只要训练和检测两边使用的策略定义或特征字典不是同一个冻结版本，就可能出现版本号不同、特征维度不一致、特征编号语义错位等问题。

## 一、模型字典版本与运行时字典版本不同

对应代码在 `ColumnModelCompatibilityValidator.validate` 中：

```java
if (!metadata.getFeatureDictionaryVersion().equals(featureDictionaryVersion)) {
    LOGGER.warn(...);
}
```

### 为什么会出现

模型训练时使用的是训练阶段生成并冻结的特征字典版本。检测阶段再次执行特征准备时，会根据当前策略命中、特征配置、字段名和特征定义重新生成特征字典版本。即使输入数据看起来是同一份，只要特征定义生成过程中的摘要内容发生变化，版本号就可能不同。

常见原因包括：

- 检测阶段重新跑了策略计划和特征组装，没有直接使用训练冻结产物。
- 特征配置发生变化，例如是否裁剪空白、是否小写、是否归一化宽度、最大特征数等。
- 策略命中集合或特征定义顺序发生变化。
- 某些画像统计值影响了策略配置摘要，间接导致特征定义变化。
- 同字段的特征维度相同，但字典版本号因生成批次或配置摘要不同而不同。

### 会有什么问题

模型参数中的权重向量是按训练时的特征编号学习出来的。检测时如果特征编号和模型训练时不一致，就会出现“同一个下标代表不同含义”的问题。

例如：

| 特征下标 | 训练时含义 | 检测时含义 |
| --- | --- | --- |
| 0 | `is_empty` | `regex_email_hit` |
| 1 | `regex_email_hit` | `is_empty` |

如果模型认为下标 0 权重很高，训练时它代表空值特征；检测时却把邮箱正则命中特征放在下标 0，那么预测分数会被错误解释，结果可能误报或漏报。

代码中当前处理并不是立刻失败，而是先记录告警。原因是 `RahaDetectService.adaptRowsToModelDictionary` 中还有一道维度校验：如果运行时字典定义数量和模型特征维度不同，会直接抛出异常；如果维度一致但版本不同，会把运行时特征行的 `featureDictionaryVersion` 改写为模型字典版本，继续交给模型预测。

这意味着当前实现假设：在相同策略计划下，只要维度一致，特征编号可以复用。但这个假设只有在特征定义顺序和语义完全一致时才安全。

### 有没有影响

有潜在影响，严重程度取决于字典是否只是版本号不同，还是实际结构也不同。

- 如果只是版本号不同，但特征定义、顺序、维度、含义完全一致，通常没有预测影响，只会产生告警。
- 如果维度不同，检测会失败，属于显式阻断。
- 如果维度相同但编号含义不同，风险最大，因为流程可能继续执行，但预测结果已经不可信。

### 如何修复

建议按优先级处理：

1. 检测阶段优先复用训练阶段冻结的策略计划和特征字典，不重新生成语义可能不同的版本。
2. 模型加载或检测前增加更强的字典结构校验，不只比较维度，还应比较特征下标、名称、类型、来源和默认值。
3. 如果版本不同但结构完全一致，可以允许映射，并在日志中明确记录“结构一致，仅版本不同”。
4. 如果结构不一致，应直接失败，避免产生看似成功但语义错误的预测结果。
5. 长期方案是将模型元数据关联到冻结训练产物，检测时按模型版本读取对应字典，而不是重新生成字典后尝试适配。

## 二、策略计划版本与运行时策略计划版本不同

对应代码在 `ColumnModelCompatibilityValidator.validate` 中：

```java
if (!metadata.getStrategyPlanVersion().equals(strategyPlanVersion)) {
    LOGGER.warn(...);
}
```

### 为什么会出现

策略计划版本由 `StrategyPlanVersioner.versionOf` 计算，核心输入是每个策略的 `strategyId` 和 `configurationHash`。只要策略集合或策略配置摘要不同，最终计划版本就不同。

常见原因包括：

- 训练阶段生成了一版策略计划，检测阶段又根据当前输入重新生成了一版。
- 策略配置依赖画像统计值，而画像统计值在不同阶段重新计算后有轻微差异。
- 策略启用开关、阈值、采样参数或配置版本变化。
- 同一快照的策略计划没有被持久化复用。

### 会有什么问题

策略计划决定后续策略执行和特征组装。策略计划版本不同，意味着后续特征字典也很可能不同，最终影响模型输入。

例如：

训练阶段策略计划：

| 策略 | 配置摘要 | 作用 |
| --- | --- | --- |
| `null-check` | `aaa` | 检查空值 |
| `pattern-check` | `bbb` | 检查格式 |

检测阶段重新生成后：

| 策略 | 配置摘要 | 作用 |
| --- | --- | --- |
| `null-check` | `aaa` | 检查空值 |
| `range-check` | `ccc` | 检查范围 |

此时即使字段名和数据快照相同，策略命中集合已经变了，特征字典也可能变。模型仍按训练时的特征语义解释输入，就可能输出错误分数。

### 有没有影响

有影响。策略计划版本不一致通常比单纯字典版本不一致更值得关注，因为它说明特征来源上游已经发生变化。轻则触发告警并依赖后续维度校验兜底，重则导致字段检测失败或预测结果语义错位。

### 如何修复

建议做法：

1. 同一数据快照下，如果已经存在训练冻结策略计划，检测阶段直接复用。
2. 策略计划持久化时保存完整策略定义，而不是只保存版本号。
3. 检测任务应携带模型的 `strategyPlanVersion`，并从仓储中读取匹配版本的策略计划。
4. 当无法读取匹配计划时，应视为不兼容，而不是静默重建。
5. 对策略计划版本告警增加任务号、快照号、模型版本，方便追溯是哪一次训练和检测不一致。

## 三、空权重与三元表达式拆箱空指针

对应代码在 `TrainingArtifactMaterializationService.materialize` 中：

```java
Double weight = null;
if (direct != null) {
    weight = Double.valueOf(direct.getSampleWeight());
} else if (propagated != null) {
    weight = Double.valueOf(propagated.getSampleWeight());
}
```

### 为什么会出现

训练物化时会保存两类数据：

- `FmdbTrainingCellRecord`：记录训练快照中的单元格、特征、聚类和标签信息。
- `FmdbTrainingExampleRecord`：记录最终实际入模的训练样本。

并不是每个单元格都有标签。没有直接标签、也没有传播标签的单元格，仍然可以作为训练单元格记录保存下来，用于后续审计、排查、聚类查看或特征回放。但它不是训练样本，因此没有样本权重。

所以 `FmdbTrainingCellRecord` 的 `weight` 使用 `Double` 对象类型，允许为空。

### 三元表达式为什么会触发空指针

如果写成嵌套三元表达式，很容易不小心触发基本类型 `double` 的自动拆箱。

示例：

```java
Double weight = direct != null
        ? direct.getSampleWeight()
        : propagated == null ? null : propagated.getSampleWeight();
```

这里 `direct.getSampleWeight()` 和 `propagated.getSampleWeight()` 返回的是基本类型 `double`，而另一个分支返回 `null`。Java 会尝试为三元表达式推断统一类型，在某些组合下会把 `null` 当成 `Double` 后再拆箱成 `double`，当实际走到无标签分支时就可能出现 `NullPointerException`。

更直观的例子：

```java
Double value = condition ? 1.0d : null;
```

这段代码看起来想表达“条件不满足时为空”，但因为 `1.0d` 是基本类型，编译器可能把整个表达式按 `double` 处理，运行到 `null` 分支时会尝试拆箱，从而空指针。

当前代码改成显式 `if / else if` 后，只有存在标签时才调用 `Double.valueOf(...)`，没有标签时保留 `null`，避免了三元表达式的拆箱风险。

### 有没有影响

修复前的影响是：只要某个训练快照单元格没有任何标签，物化训练单元格记录时就可能抛出空指针，导致整个训练产物物化失败。这样会影响后续冻结样本恢复、模型训练血缘追踪和检测回放。

修复后的行为是：无标签单元格可以正常落入训练单元格记录，权重字段为空；只有被选为训练样本的记录才要求有标签和权重。

### 如何修复

当前代码已经采用较稳妥写法。后续建议：

1. 保持 `Double weight = null` 加显式分支赋值，不改回嵌套三元表达式。
2. 单元测试覆盖三种情况：直接标签、传播标签、无标签。
3. 对 `FmdbTrainingCellRecord` 的 `sample_weight` 字段确认数据库允许为空。
4. 对最终训练样本表继续保持权重非空，因为入模样本必须来自有效标签。

## 四、同一快照复用训练冻结策略计划

对应代码在 `StrategyPlanService.generateAndSave` 中：

```java
List<StrategyPlan> existing = repository.findPlans(
        dataset.getDatasetId(), dataset.getSnapshotId());
if (!existing.isEmpty()) {
    LOGGER.info(...);
    return existing;
}
```

### 为什么要这样做

同一数据快照代表同一批待处理数据。训练阶段如果已经为该快照生成并保存了策略计划，检测阶段再次处理同一快照时，应复用这份计划，而不是重新生成一份。

原因是策略计划是模型输入链路的上游定义。训练时模型学习的是“某一套策略计划产生的特征”。检测时如果换了策略计划，即使数据没变，特征语义也可能变化。

### 举例说明

假设快照 `snapshot-001` 在训练阶段生成策略计划版本 `plan-A`：

| 阶段 | 快照 | 策略计划 | 字典版本 |
| --- | --- | --- | --- |
| 训练 | `snapshot-001` | `plan-A` | `dict-A` |

模型发布后，检测阶段又对同一快照执行：

| 阶段 | 快照 | 策略计划 | 字典版本 |
| --- | --- | --- | --- |
| 检测 | `snapshot-001` | `plan-B` | `dict-B` |

如果 `plan-B` 是重新生成的，就可能出现模型期望 `dict-A`，运行时却提供 `dict-B` 的情况。这样会触发兼容校验告警，甚至检测失败。

复用已有计划后，检测阶段变为：

| 阶段 | 快照 | 策略计划 | 字典版本 |
| --- | --- | --- | --- |
| 检测 | `snapshot-001` | `plan-A` | `dict-A` |

这样训练和检测的上游定义一致，模型输入更可控。

### 有没有影响

这是正向修复，主要影响是保证同一快照下策略定义稳定，减少策略计划版本和特征字典版本不一致的概率。

需要注意的边界是：如果仓储里已经存在的计划不是训练冻结计划，而是历史错误写入或不完整写入，直接复用会扩大历史错误影响。因此仓储查询最好能区分计划来源、版本状态和完整性。

### 如何修复或增强

当前代码已经实现了按 `datasetId` 和 `snapshotId` 查找并复用已有计划。建议进一步增强：

1. 保存策略计划时增加来源标识，例如训练冻结、检测临时、人工重放。
2. 查询时优先复用训练冻结计划，而不是任意已有计划。
3. 对已有计划数量和字段覆盖范围做完整性校验。
4. 如果已有计划不完整，应记录错误并阻断，而不是重新生成不同版本继续执行。
5. 在检测输出中记录实际使用的策略计划版本，便于和模型元数据对账。

## 五、总体修复建议

整体建议按“先防错、再追溯、最后收敛设计”的顺序推进：

1. 检测阶段默认读取模型绑定的冻结策略计划和冻结特征字典。
2. 字典兼容校验从“维度一致”升级为“结构一致”，比较特征下标、名称、类型、来源和默认值。
3. 对策略计划版本不一致从 `warn` 逐步升级为可配置策略：严格模式下直接失败，兼容模式下允许结构一致后继续。
4. 增加训练产物物化测试，覆盖无标签单元格空权重场景。
5. 增加端到端回放测试：同一快照训练后检测应复用相同策略计划版本和字典版本。
6. 增加运维排查字段：训练批次、模型集合版本、策略计划版本、特征字典版本、快照号。

## 六、结论

这三处代码解决的是同一类问题：保证模型训练时看到的特征定义，与检测时输入模型的特征定义一致。

版本不同本身不一定立刻造成错误，但它是重要风险信号。真正的问题不是字符串版本号不同，而是版本号背后代表的策略定义、特征定义、特征编号和模型权重之间是否仍然一致。

空权重问题则是训练物化层面的健壮性问题。无标签单元格可以作为审计记录存在，但不能被当成训练样本；显式分支赋值可以避免 Java 自动拆箱导致的空指针。
