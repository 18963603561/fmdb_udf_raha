# forceRun 加时间戳简化幂等方案

生成时间：2026-07-21 14:06

## 一、结论

你的思路是对的，但要分清两个问题：

1. 强制创建新任务。
2. 判断哪个结果是当前有效结果。

如果只解决第一个问题：

```text
forceRun=true 时，在 executionInputFingerprint 中加入时间戳
```

基本就够了。

但如果还要解决“覆盖后默认看最新结果”，还需要一个查询规则：

```text
同一业务组内，取最新成功任务对应的批次结果。
```

所以最小方案不是复杂覆盖日志，而是：

```text
forceRun=true + runTimestamp 打散幂等
查询时按同一业务组取最新成功批次
```

## 二、为什么加时间戳能强制新跑

当前关系是：

```text
executionInputFingerprint
  -> 进入 configVersion
    -> 进入 idempotentKey
```

如果 `forceRun=true` 时把时间戳加入 `executionInputFingerprint`：

```text
baseExecutionInputFingerprint
  + forceRun
  + runTimestamp
```

那么每次生成的指纹都不同。

结果就是：

```text
executionInputFingerprint 变
configVersion 变
idempotentKey 变
```

最终 `dw.raha_job_run` 查不到相同 `idempotent_key`，就会创建新任务。

## 三、推荐生成规则

### 1）普通执行

```text
forceRun=false
executionInputFingerprint = sha256(baseInput)
```

重复执行相同请求时：

```text
幂等键相同
复用旧任务
```

### 2）强制执行

```text
forceRun=true
runTimestamp = 当前请求生成时间
executionInputFingerprint = sha256(baseInput + forceRun + runTimestamp)
```

重复执行相同请求时：

```text
幂等键不同
创建新任务
```

### 3）更稳的写法

只用毫秒时间戳理论上可能撞车。

虽然概率很低，但并发场景下最好加一个随机值或调用方传入的运行号。

推荐：

```text
runNonce = request.forceRunId 或 当前毫秒 + 随机 UUID
executionInputFingerprint = sha256(baseInput + forceRun + runNonce)
```

其中 `forceRunId` 可以由调用方传。

如果调用方不传，系统自动生成。

## 四、为什么不能只保留打散后的指纹

如果只保留加了时间戳后的指纹，会丢掉“这些任务本来属于同一组业务输入”的关系。

例如：

第一次检测：

```text
person_info + dw.person_info + model-set-A + runTimestamp=1000
```

第二次强制检测：

```text
person_info + dw.person_info + model-set-A + runTimestamp=2000
```

它们的 `idempotentKey` 不同，这是对的。

但业务上它们仍然属于同一组：

```text
person_info + dw.person_info + model-set-A
```

如果不保存这个基础业务组，查询“当前结果”时就不知道要在哪个范围里取最新。

因此建议同时保留两个值：

| 字段 | 含义 |
| --- | --- |
| `baseExecutionInputFingerprint` | 不含时间戳，用于表示同一业务组 |
| `executionInputFingerprint` | 含时间戳，用于强制创建新任务 |

如果不想新增字段，也至少要在 `result_summary_json` 里保存基础业务组。

## 五、最小落地方案

第一期可以很简单，不需要覆盖日志表。

### 1）请求参数

```json
{
  "forceRun": "true",
  "forceRunId": "202607211406001"
}
```

`forceRunId` 可选。

不传时系统生成：

```text
当前毫秒 + UUID
```

### 2）指纹生成

```text
baseFingerprint = sha256(baseInput)

if forceRun:
    executionInputFingerprint = sha256(baseInput + forceRunId)
else:
    executionInputFingerprint = baseFingerprint
```

### 3）任务摘要

写入 `dw.raha_job_run.result_summary_json`：

```json
{
  "forceRun": true,
  "forceRunId": "202607211406001",
  "baseExecutionInputFingerprint": "sha256-base",
  "executionInputFingerprint": "sha256-force",
  "currentSelectRule": "LATEST_SUCCESS_BATCH"
}
```

### 4）当前结果查询

同一业务组内取最新成功任务。

检测结果推荐按：

```text
datasetId + inputReference + modelSetVersion + baseExecutionInputFingerprint
```

找最新成功任务。

再用最新任务的 `jobId` 作为 `detection_batch_id` 查结果。

## 六、检测示例

### 1）第一次正常检测

请求：

```json
{
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info",
  "modelSetVersion": "model-set-A",
  "forceRun": "false"
}
```

结果：

```text
baseExecutionInputFingerprint = fp-base-A
executionInputFingerprint = fp-base-A
idempotentKey = key-001
detectionBatchId = job-001
```

### 2）第二次相同请求

仍然：

```json
{
  "forceRun": "false"
}
```

结果：

```text
executionInputFingerprint 不变
idempotentKey 不变
复用 job-001
```

### 3）第三次强制重跑

请求：

```json
{
  "forceRun": "true"
}
```

系统生成：

```text
forceRunId = 202607211406001
```

结果：

```text
baseExecutionInputFingerprint = fp-base-A
executionInputFingerprint = fp-force-001
idempotentKey = key-002
detectionBatchId = job-002
```

当前结果：

```text
job-002
```

历史结果：

```text
job-001 仍保留
```

## 七、这个方案解决了什么

可以解决：

1. 不用删除旧数据。
2. 不用覆盖日志表。
3. 不用更新旧记录状态。
4. 用户可以明确强制新跑。
5. 相同请求默认仍然幂等复用。
6. 当前结果可以取最新成功批次。

## 八、这个方案还需要注意什么

### 1）时间戳必须是请求级

不要每行生成一个时间戳。

必须在请求开始时生成一次：

```text
runTimestamp
```

然后整个任务共用。

### 2）强制重跑不等于覆盖旧数据

`forceRun=true` 只表示：

```text
创建新任务。
```

是否把新任务结果当成当前结果，需要查询规则决定。

推荐规则：

```text
最新成功任务就是当前结果。
```

### 3）失败任务不能成为当前结果

如果强制重跑失败：

```text
job-002 = FAILED
```

当前结果仍然应该是：

```text
job-001
```

所以查询当前结果必须过滤：

```text
status = SUCCEEDED
```

或业务允许时包含：

```text
PARTIAL_SUCCESS
```

### 4）并发时要有稳定排序

如果两个强制任务同时成功，当前结果按：

```text
finished_at desc, created_at desc, job_id desc
```

取第一条。

这样至少结果确定。

更严格的做法是同一业务组不允许并发强制重跑。

## 九、是否还需要 `overwrite`

如果你的目标只是：

```text
重新跑一遍，并让用户默认看到最新结果
```

那可以不需要 `overwrite`。

直接使用：

```text
forceRun=true
```

再配合：

```text
查询最新成功批次
```

就够了。

只有当你需要表达：

```text
这次新结果明确替代某个旧结果，并记录替代原因
```

才需要 `overwrite` 或覆盖日志。

## 十、最终推荐

推荐第一期采用你的简化方案：

```text
forceRun=true 时，把请求级 runNonce 加入 executionInputFingerprint。
```

同时保留基础业务组：

```text
baseExecutionInputFingerprint
```

最终规则：

1. `forceRun=false`：按原幂等逻辑复用。
2. `forceRun=true`：生成新指纹、新 `configVersion`、新 `idempotentKey`、新任务。
3. 当前结果查询：按同一基础业务组取最新成功任务。
4. 旧结果不删除。
5. 失败的新任务不影响当前结果。

这样已经能解决绝大多数“重复执行”和“重新跑最新结果”的问题，不需要上复杂覆盖日志表。
