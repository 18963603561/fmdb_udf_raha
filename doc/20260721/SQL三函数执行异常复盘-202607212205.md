## 三、异常总览

本次异常并不是单一原因导致，而是多个历史状态、运行方式和数据准备问题叠加。

主要异常分为六类：

1. Spark SQL 直接调用 UDF 时的 Driver 与 Task 线程边界问题。
2. 容器内外服务名、命令和权限环境不一致。
3. Raha 历史库表存在旧格式数据。
4. 清库方式与 Hive 外部表机制不匹配。
5. 训练依赖人工标注，清库后缺少标注输入。
6. SQL 构造标注 JSON 时字段语义不完整。

## 四、异常一：Spark SQL 直接调用 UDF 的线程问题

### 4.1 现象

直接执行：

```sql
SELECT F_DW_DETCOLLECT('...');
```

时，最初出现异常：

```text
SparkSession should only be created and accessed on the driver.
```

### 4.2 原因

Spark SQL 中 `SELECT UDF(...)` 即使没有 `FROM` 表，也会进入 Spark 物理执行计划。

Hive `GenericUDF` 的初始化和求值可能发生在 Spark task 线程，而不是普通 Driver 主线程。

Raha 三函数内部不是普通轻量 UDF，而是会再次执行数据读取、画像、策略执行、模型训练、结果写库和 ZIP 发布的完整工作流。

因此它需要合法访问 `SparkSession` 并发起内部 Spark 作业。

普通 Spark SQL UDF 执行模型天然不适合承载这种 Driver 侧工作流。

### 4.3 本次处理

为了满足“只能使用 SQL，不用 app”的要求，前面已经在 `AbstractRahaGenericUdf` 中增加了本地 Spark SQL 验证兼容逻辑：

1. 在 task 线程初始化 UDF 时，短暂绕开 `TaskContext` 以复用当前 JVM 的 Spark 会话。
2. 在真正执行 Raha 工作流时，切到同 JVM 的独立线程，避免在 Spark task 线程里嵌套驱动 Spark。

这样 `spark-sql --master local[2]` 可以完成验证。

### 4.4 风险

这个方式是为了满足当前 SQL 验证约束，不等价于生产级通用 UDF 运行模型。

如果切换到 Standalone executor 独立 JVM、YARN、Kubernetes 或真正分布式 executor，该兼容逻辑仍有失效风险。

### 4.5 如何避免

长期建议仍然是：

1. 三函数工作流应放在 Driver 侧过程入口执行。
2. 如果外部必须是 SQL 形态，优先做 FMDB 存储过程或 Spark SQL 扩展命令。
3. 不建议长期依赖普通 `SELECT UDF(...)` 承载训练和预测工作流。
4. 验证环境如果必须用 SQL，应固定使用 `spark-sql --master local[2]`，并使用同一份函数 jar。

## 五、异常二：函数包是否一致的疑问

### 5.1 现象

训练和预测失败后，怀疑采样函数包与训练、预测函数包不一致。

### 5.2 核查结果

三份 SQL 均使用同一个 jar：

```sql
ADD JAR /opt/spark/work-dir/fmdb-udf-raha-1.0.0-SNAPSHOT-all.jar;
```

三函数类也在同一包下：

```text
com.fiberhome.ml.raha.udf.F_DW_DETCOLLECT
com.fiberhome.ml.raha.udf.F_DW_DETTRAIN
com.fiberhome.ml.raha.udf.F_DW_DETRUN
```

日志中三阶段都出现相同的 `AbstractRahaGenericUdf.java` 行号，说明使用的是同一版 UDF 基类。

### 5.3 结论

失败不是函数包不一致导致，而是数据状态和 SQL 入参依赖不匹配导致。

### 5.4 如何避免

后续每次执行前固定校验三点：

1. SQL 中 `ADD JAR` 路径完全一致。
2. 容器内 jar 的 SHA-256 校验值记录到输出目录。
3. 三个 SQL 输出中都能看到同一版 UDF 基类行号或版本标识。

## 六、异常三：历史数据中存在 `SHA-256` 采样指纹

### 6.1 现象

执行旧训练 SQL 时出现：

```text
不支持的采样行指纹算法：SHA-256
```

### 6.2 原因

旧批次 `sample-15d1d5b6681c79f83f4adef1` 在 `dw.raha_sample_record` 中已有 300 条历史记录。

这些记录的 `row_fingerprint_algorithm` 为 `SHA-256`。

当前代码只支持 `MD5` 行指纹算法。

### 6.3 过程

曾短暂评估过补充 `SHA-256` 兼容，但随后明确要求不要补兼容，可以直接删除库中数据。

因此最终没有走兼容代码路线，而是清理旧库表数据后重新采样。

### 6.4 如何避免

后续版本应坚持一个原则：

1. 如果不计划兼容旧数据，执行前必须清空 Raha 产物表。
2. 如果要复用历史数据，必须提前做数据版本迁移。
3. 不允许在不清库的情况下混用旧批次和新代码。

建议执行前增加 SQL 检查：

```sql
SELECT row_fingerprint_algorithm, count(*)
FROM dw.raha_sample_record
GROUP BY row_fingerprint_algorithm;
```

如果出现非 `MD5`，应先清库或迁移。

## 七、异常四：直接删库后旧 SQL 固定 ID 失效

### 7.1 现象

清理旧批次后，再执行原始训练 SQL，返回：

```json
{
  "status": "FAILED",
  "errorcode": "SAMPLE_NOT_FOUND"
}
```

### 7.2 原因

原始训练 SQL 写死了旧采样批次：

```json
"sampleBatchId":"sample-15d1d5b6681c79f83f4adef1"
```

清库后这个批次已经不存在。

### 7.3 如何避免

采样、训练、预测必须作为同一轮闭环执行。

不能在清库后继续使用旧 SQL 中写死的 `sampleBatchId` 和 `modelSetVersion`。

正确流程是：

1. 先执行采样 SQL。
2. 从采样输出提取新的 `sampleBatchId`。
3. 用新的 `sampleBatchId` 生成训练 SQL。
4. 从训练输出提取新的 `modelSetVersion`。
5. 用新的 `modelSetVersion` 生成预测 SQL。

## 八、异常五：Hive 外部表不能 `TRUNCATE`

### 8.1 现象

清空表时执行：

```sql
TRUNCATE TABLE dw.raha_sample_record;
```

返回：

```text
Operation not allowed: TRUNCATE TABLE on external tables
```

### 8.2 原因

Raha 相关表是 Hive 外部表，Spark 不允许对外部表执行 `TRUNCATE TABLE`。

### 8.3 中间尝试

曾尝试用 `INSERT OVERWRITE` 保留非目标记录，但遇到动态分区非空约束和外部表读写同表风险。

最后改为在 HDFS 容器中删除表 location 下的数据目录。

### 8.4 最终处理

在 `fmdb-hdfs` 容器中使用：

```sh
/opt/hadoop/bin/hdfs dfs -rm -r -skipTrash /user/hive/warehouse/dw.db/raha_xxx
```

随后重建空目录并赋权：

```sh
/opt/hadoop/bin/hdfs dfs -mkdir -p /user/hive/warehouse/dw.db/raha_xxx
/opt/hadoop/bin/hdfs dfs -chmod -R 777 /user/hive/warehouse/dw.db/raha_xxx
```

### 8.5 如何避免

后续清库不要使用 `TRUNCATE TABLE`。

推荐提供一个固定脚本：

1. 使用 `SHOW TABLES IN dw LIKE 'raha_*'` 列出表。
2. 通过 `DESCRIBE FORMATTED` 获取表 location。
3. 在 HDFS 容器统一删除 location 数据目录。
4. 重建目录和权限。
5. 用 SQL 验证关键表计数为 0。

## 九、异常六：容器命令环境不一致

### 9.1 现象

在 `fmdb_udf_schmatch` 容器中执行 `hdfs` 命令失败：

```text
hdfs: command not found
```

### 9.2 原因

`fmdb_udf_schmatch` 是 Spark SQL 执行容器，里面有 `/opt/spark/bin/spark-sql`，但没有 Hadoop 命令行。

HDFS 命令位于 `fmdb-hdfs` 容器：

```text
/opt/hadoop/bin/hdfs
```

### 9.3 如何避免

后续分工固定如下：

| 操作 | 容器 |
| --- | --- |
| 执行 `spark-sql` | `fmdb_udf_schmatch` |
| 清理 HDFS warehouse 数据 | `fmdb-hdfs` |
| 查看 Spark master 状态 | `fmdb-spark-master` |
| 查看 executor 状态 | `fmdb-spark-worker` |

不要假设所有命令都在同一个容器中存在。

## 十、异常七：清库后训练缺少标注

### 10.1 现象

清库后如果直接训练，会返回：

```text
MANUAL_ANNOTATION_NOT_FOUND
```

或找不到可训练标注批次。

### 10.2 原因

`F_DW_DETCOLLECT` 只生成待标注模板，不会自动生成训练标签。

`F_DW_DETTRAIN` 需要以下两种输入之一：

1. HDFS 或本地路径中存在对应 `sampleBatchId` 的已标注 Excel。
2. `dw.raha_annotation_record` 中存在该 `sampleBatchId` 的可训练标注记录。

清库后两者都没有。

### 10.3 本次处理

为了满足“只能使用 SQL”的约束，本次没有用 Python 自动标注，也没有用 app。

而是通过 Spark SQL 向 `dw.raha_annotation_record` 插入 300 条标注记录。

标注规则如下：

1. `name`：非空。
2. `age`：小于 18 或大于 90 判为异常。
3. `phone`：不匹配手机号正则判为异常。
4. `id_card`：不匹配身份证正则判为异常。
5. `email`：不匹配邮箱正则判为异常。
6. `home_address`：长度过短判为异常。

### 10.4 风险

这属于 SQL 自动规则标注，仅用于跑通闭环。

它不是人工审核后的高质量标签，也不是严格使用原始 ground truth 文件。

因此模型质量和字段失败情况会受规则标签影响。

### 10.5 如何避免

如果要稳定训练，应准备正式标注流程：

1. 采样后导出 Excel。
2. 人工或固定程序补充标签。
3. 上传到约定标注目录。
4. 再执行训练 SQL。

如果仍要求纯 SQL，应提前准备标准 SQL 标注脚本，并把规则和字段校验写成固定版本。

## 十一、异常八：SQL 构造标注 JSON 的两次错误

### 11.1 第一次错误

训练返回：

```text
标注 JSON 字段列表无效
```

原因是最初使用：

```sql
array_remove(array(...), CAST(NULL AS STRING))
```

生成 `errorColumns` 时，空数组场景下字段被省略或解析结果不符合仓储反序列化要求。

### 11.2 第二次错误

训练返回：

```text
正常行不能包含异常字段
```

原因是为了避免字段缺失，曾将 `errorColumns` 固定写成所有字段。

这会导致 `rowLabel=0` 的正常行仍然携带异常字段列表，违反 `RowAnnotation` 约束。

### 11.3 最终修正

最终使用 Spark SQL 高阶函数：

```sql
filter(array(...), x -> x is not null)
```

保证：

1. 异常行的 `errorColumns` 是实际异常字段数组。
2. 正常行的 `errorColumns` 是空数组。
3. `reviewedColumns` 始终是完整字段数组。
4. `cellLabels` 始终包含六个字段标签。

### 11.4 如何避免

后续 SQL 标注脚本必须先做单条 JSON 验证：

```sql
SELECT annotation_json
FROM dw.raha_annotation_record
WHERE sample_batch_id = '...'
LIMIT 1;
```

并确认 JSON 至少包含：

1. `cellLabels`
2. `reviewedColumns`
3. `errorColumns`
4. `rowLabel`
5. `sourceSnapshotId`
6. `importFingerprint`

## 十二、异常九：训练阶段部分字段失败

### 12.1 现象

训练最终返回整体 `SUCCESS`，但字段级有失败：

| 字段 | 状态 |
| --- | --- |
| `name` | `PUBLISHED` |
| `age` | `PUBLISHED` |
| `phone` | `FAILED` |
| `id_card` | `PUBLISHED` |
| `email` | `PUBLISHED` |
| `home_address` | `FAILED` |

### 12.2 原因

`phone` 训练时出现 HDFS 临时文件写入异常。

`home_address` 被模型质量门禁拒绝，日志中可见：

```text
模型质量门禁拒绝退化模型
```

原因是 SQL 规则标注中 `home_address` 正例过少，导致模型质量门禁判断退化。

### 12.3 影响

模型集合仍然生成：

```text
dw.person_info@20260721135853.933-job-43d61f46-cee7-43b0-ae80-1202caebe51c
```

但其中只有 4 个字段有可发布模型。

预测必须使用：

```json
"missingModelPolicy":"PARTIAL"
```

否则会因缺少字段模型失败。

### 12.4 如何避免

1. 提高标注质量，保证每个字段有足够正负样本。
2. 对 `phone` 这类字段重跑训练前清理 HDFS 临时目录。
3. 检查 HDFS NameNode 和 DataNode 是否存在文件租约残留。
4. 如果只是验证链路，可设置字段白名单，先训练高质量字段。
5. 如果要全字段上线，不应接受部分模型成功。

## 十三、异常十：预测阶段部分字段失败

### 13.1 现象

预测最终返回整体 `SUCCESS`：

```json
{
  "status": "SUCCESS",
  "modelfieldcount": 4,
  "failedfieldcount": 2,
  "detectedcellcount": 1600,
  "detectederrorcount": 134
}
```

但日志中可见 `phone` 和 `home_address` 缺少字段模型。

### 13.2 原因

训练阶段这两个字段未发布模型。

预测 SQL 使用 `PARTIAL` 策略后，系统允许跳过缺失模型字段，仅对 4 个已发布字段执行预测。

### 13.3 影响

预测结果不是 6 个字段全量预测，而是 4 个字段部分预测。

`detectedcellcount=1600`，对应 400 行乘以 4 个可用模型字段。

### 13.4 如何避免

如果目标是全字段预测，必须先保证训练阶段 6 个字段全部 `PUBLISHED`。

预测前应执行模型完整性检查：

```sql
SELECT column_name, model_set_status, model_version
FROM dw.raha_model_artifact
WHERE model_set_version = '...';
```

只有字段数等于源表有效字段数时，才用严格预测。

## 十四、关于日志中大量 HDFS 分区警告

### 14.1 现象

日志中多次出现：

```text
The directory ... was not found. Was it deleted very recently?
```

### 14.2 原因

清库时删除了 HDFS 目录，但 Hive Metastore 中分区元数据可能仍有缓存或残留。

Spark 在列举旧分区 location 时发现目录不存在，于是打印警告。

### 14.3 影响

这些警告大多不影响最终写入和查询。

本次采样、训练和预测均继续执行。

### 14.4 如何避免

清理 HDFS 目录后应增加分区元数据修复步骤：

```sql
MSCK REPAIR TABLE dw.raha_sample_record;
MSCK REPAIR TABLE dw.raha_annotation_record;
MSCK REPAIR TABLE dw.raha_detection_result;
```

或者在清理时同步执行 `ALTER TABLE DROP PARTITION`，避免元数据指向已删除路径。

## 十五、推荐标准执行流程

后续若要重复本次“纯 SQL 三函数闭环”，建议使用如下标准流程。

### 15.1 前置检查

1. 确认容器 `fmdb_udf_schmatch` 存在并运行。
2. 确认 jar 路径存在：

```text
/opt/spark/work-dir/fmdb-udf-raha-1.0.0-SNAPSHOT-all.jar
```

3. 确认源表存在：

```sql
SELECT count(*) FROM dw.person_info;
```

4. 确认 Web 服务存在：

```text
/opt/software/fmdb/manager/web/jetty/webapps/fmdb_report
```

### 15.2 清库

1. 在 `fmdb-hdfs` 容器清理 `dw.db/raha_*` 数据目录。
2. 重建空目录并赋权。
3. 回到 `fmdb_udf_schmatch` 用 SQL 验证关键表为 0。

### 15.3 采样

执行采样 SQL：

```sql
ADD JAR /opt/spark/work-dir/fmdb-udf-raha-1.0.0-SNAPSHOT-all.jar;

CREATE TEMPORARY FUNCTION F_DW_DETCOLLECT
AS 'com.fiberhome.ml.raha.udf.F_DW_DETCOLLECT';

SELECT F_DW_DETCOLLECT('...');
```

提取：

1. `sampleBatchId`
2. `snapshotId`
3. `schemaHash`
4. `partitionMonth`

### 15.4 标注

如果不能人工上传 Excel，则必须用 SQL 插入 `dw.raha_annotation_record`。

插入前必须保证 JSON 字段完整。

### 15.5 训练

用新 `sampleBatchId` 执行训练 SQL。

训练后必须检查字段级结果，不要只看整体 `status`。

### 15.6 预测

如果训练有字段失败，预测必须使用：

```json
"missingModelPolicy":"PARTIAL"
```

如果要求全字段预测，则训练阶段必须先修复所有失败字段。

## 十六、后续改进建议

### 16.1 工程侧

1. 给三函数增加明确的运行模式标识，区分验证模式和生产模式。
2. 给 UDF 返回结果增加 `overallStatus` 和 `fieldStatusSummary`，避免字段失败被整体 `SUCCESS` 淹没。
3. 给 SQL 入口增加模型完整性检查函数。
4. 给清库提供受控脚本，避免手工删除 HDFS 路径。
5. 给标注 JSON 增加 SQL 示例模板和校验 SQL。

### 16.2 数据侧

1. 不要混用旧 `SHA-256` 采样记录和新 `MD5` 行身份逻辑。
2. 每轮验证前固定清理 Raha 产物表。
3. 采样、标注、训练、预测必须使用同一轮 ID。
4. 不要复用旧 SQL 中写死的 `sampleBatchId` 和 `modelSetVersion`。

### 16.3 运维侧

1. 清库后执行分区修复或分区删除。
2. 训练前清理 HDFS `_temporary` 残留。
3. 长时间训练失败后检查 HDFS lease。
4. 固定容器职责，不在 Spark 执行容器里执行 HDFS 管理命令。

## 十七、最终结论

这次异常多的根本原因是：把一个完整 Driver 侧工作流强行放进 Spark SQL 普通 UDF 调用模型中，同时又混入了历史库表数据、旧固定 ID、外部表清理限制和缺少正式标注的问题。

本次最终闭环已经跑通，但属于 SQL 验证闭环，不应直接等同生产执行方案。

要稳定避免这些异常，应坚持三点：

1. 每轮验证先清空 Raha 产物数据，避免历史 ID 和旧格式数据干扰。
2. 采样、标注、训练、预测使用同一轮动态返回的 ID，不复用旧 SQL 写死值。
3. 如果长期要求 SQL 入口，应建设真正的 Driver 侧 SQL 过程或扩展命令，而不是依赖普通 UDF 承载完整训练预测工作流。
