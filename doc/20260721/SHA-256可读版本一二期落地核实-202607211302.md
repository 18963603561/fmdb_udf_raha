# SHA-256 可读版本一二期落地核实

## 一、背景

本次按《SHA-256 字段梳理与模型可读版本改造分析》和《SHA-256 可读版本改造涉及文件清单》落地两期改造。

核心原则如下：

1. 不新增冗余哈希字段。
2. 将可读值本身设计为唯一值。
3. 表输入按库名和表名生成来源。
4. 字段模型按库名、表名、字段名和版本生成。
5. SQL 输入统一解析第一个来源表；多表或多段 SQL 只取第一个表。
6. 检测不传模型集合版本时，按表或 SQL 第一个来源表自动选择最新完整已发布模型集合。

## 二、一期落地内容

一期目标是解决模型集合、字段模型、模型文件名、SQL 来源解析和默认选模问题。

### 2.1 已完成能力

1. 新增可读标识工具 `ReadableIdUtils`，统一生成可读、安全、稳定的标识片段。
2. 新增模型来源键 `ModelSourceKey`，优先用表名生成模型来源，缺失时兼容数据集标识。
3. 新增模型可读版本工具 `ModelReadableVersioner`。
4. 模型集合版本改为 `库名.表名@时间-任务号`。
5. 字段模型版本改为 `库名.表名.字段名@时间-任务号`。
6. 模型名称改为 `前缀-库名.表名-字段名`。
7. 模型文件名由可读模型版本转换为路径安全文件名。
8. FMDB 模型路径直接保留可读模型版本，避免路径不可逆。
9. 新增 SQL 首表解析器 `FmdbSqlSourceTableResolver`。
10. SQL 输入不再要求外部传 `datasetId`，统一按首个来源表推导。
11. 模型元数据仓储新增按数据集查找最新完整已发布模型集合能力。
12. 检测请求支持不传 `modelSetVersion`，自动选最新已发布模型集合。
13. 训练成功后自动发布候选模型，保证三函数训练后可直接检测。
14. 阈值评测允许更新已发布模型的阈值元数据，检测加载时仍使用发布元数据阈值。

5. 自动发布后，训练输出模型可直接用于检测。

## 三、二期落地内容

二期目标是进一步处理快照、采样批次、标注批次、训练快照和 SQL 来源引用，减少 SHA-256 在业务主标识里的使用。

### 3.1 已完成能力

1. `DataLoadRequest` 新增 `sourceReference`，区分实际读取输入和持久化展示引用。
2. `DatasetSnapshot.inputReference` 对 SQL 记录为首个来源表，实际 SQL 原文保存到采样上下文。
3. 默认快照 ID 改为 `snapshot_来源@版本` 形式。
4. 采样批次 ID 改为 `sample_库名.表名@时间` 形式。
5. 标注批次 ID 改为 `ann_采样批次@时间` 形式。
6. 训练合并快照 ID 改为 `train_库名.表名@时间` 形式。
7. SQL 采样时在上下文保存 `readInputReference`，训练恢复时仍可用 SQL 原文读取数据。
8. 旧 SQL 记录中如果 `inputReference` 仍为 SQL 原文，训练入口继续兼容并解析首表。
9. README 已更新 SQL 首表、默认选模和 `modelSetVersion` 可选规则。

## 六、结论

一、二期均已完成落地并通过重点核实。

本次改造没有通过新增冗余哈希字段解决问题，而是把已有业务主标识改为可读且唯一：模型集合版本、字段模型版本、快照 ID、采样批次、标注批次和训练快照均可直接从值本身看出来源。SQL 输入已统一按第一个来源表归一，检测入口可以只传表名或 SQL，系统自动选择最新完整已发布模型集合；需要指定历史模型时仍可传入 `modelSetVersion`。
