# FMDB 追加加最新时间戳覆盖简化方案

生成时间：2026-07-21 14:01

## 一、结论

可以简化。

在 FMDB 不支持删除的前提下，直接追加是可行的。

但是不要简单按“每行最大时间戳”取结果。

正确方式应该是：

```text
直接追加新批次
  -> 用批次级时间选出最新批次
  -> 查询当前结果时返回这个最新批次的全部明细
```

一句话：

```text
可以用最新时间戳解决，但必须按批次取最新，不能按单行取最新。
```

## 二、为什么不能只按每行最新时间戳

以检测结果表为例，当前 `dw.raha_detection_result` 有这些关键字段：

| 字段 | 说明 |
| --- | --- |
| `detection_batch_id` | 检测批次，通常等于 `jobId` |
| `dataset_id` | 数据集 |
| `input_reference` | 输入表或 SQL |
| `model_set_version` | 模型集合版本 |
| `cell_id` | 单元格标识 |
| `detected_at` | 检测时间 |
| `partition_date` | 分区日期 |

如果写：

```sql
select *
from dw.raha_detection_result
where detected_at = (
  select max(detected_at)
  from dw.raha_detection_result
)
```

这通常是错的。

原因：

一次检测有很多单元格。

每个单元格写出的 `detected_at` 可能不是完全同一个毫秒。

如果只取最大 `detected_at`，可能只取到最后几个单元格，丢掉同一批次里的其他单元格。

所以必须先找到最新批次，再取这个批次的全部结果。

## 三、最简推荐方案

不新增覆盖日志表。

不删除旧数据。

不改旧数据状态。

只追加新结果。

查询当前结果时按最新检测批次读取。

核心规则：

```text
同一业务覆盖组内，finished_at 最大的成功任务，就是当前有效批次。
```

如果不想关联任务表，也可以用：

```text
同一业务覆盖组内，max(detected_at) 最大的 detection_batch_id，就是当前有效批次。
```

但更推荐使用任务表的 `finished_at`，因为它是批次级完成时间。

## 四、为什么推荐用任务表时间

检测明细表的 `detected_at` 是单元格级时间。

任务表 `dw.raha_job_run` 的 `finished_at` 是任务级时间。

覆盖语义应该是任务级的：

```text
哪一次检测任务是最新完成的
```

而不是：

```text
哪个单元格最后被处理
```

因此推荐当前结果查询时以：

```text
dw.raha_job_run.finished_at
```

作为批次排序字段。

## 五、最小字段依赖

### 1）不新增字段也能做

当前已有字段已经可以支持简化方案：

| 表 | 字段 | 用途 |
| --- | --- | --- |
| `dw.raha_detection_result` | `detection_batch_id` | 批次标识 |
| `dw.raha_detection_result` | `dataset_id` | 覆盖组条件 |
| `dw.raha_detection_result` | `input_reference` | 覆盖组条件 |
| `dw.raha_detection_result` | `model_set_version` | 覆盖组条件 |
| `dw.raha_job_run` | `job_id` | 关联 `detection_batch_id` |
| `dw.raha_job_run` | `status` | 只取成功任务 |
| `dw.raha_job_run` | `finished_at` | 判断最新批次 |

### 2）建议新增但非必需的字段

如果后续可以改表，建议检测结果表增加：

| 字段 | 说明 |
| --- | --- |
| `batch_finished_at` | 批次完成时间 |
| `overwrite_group_key` | 覆盖组 |

这样查询更简单，不必关联任务表。

但是第一期可以不加。

## 六、当前结果查询方案

### 1）按任务表选择最新检测批次

逻辑 SQL：

```sql
select r.*
from dw.raha_detection_result r
join (
  select r2.dataset_id,
         r2.input_reference,
         r2.model_set_version,
         r2.detection_batch_id
  from dw.raha_detection_result r2
  join dw.raha_job_run j
    on r2.detection_batch_id = j.job_id
  where j.status = 'SUCCEEDED'
    and r2.dataset_id = 'person_info'
    and r2.input_reference = 'select * from dw.person_info'
    and r2.model_set_version = 'model-set-A'
  order by j.finished_at desc, j.created_at desc, j.job_id desc
  limit 1
) latest
on r.detection_batch_id = latest.detection_batch_id
```

说明：

先找最新 `detection_batch_id`。

再取这个批次所有结果。

### 2）不关联任务表的退化方案

如果当前查询不方便关联 `dw.raha_job_run`，可以先按检测明细表求最新批次：

```sql
select r.*
from dw.raha_detection_result r
join (
  select detection_batch_id
  from dw.raha_detection_result
  where dataset_id = 'person_info'
    and input_reference = 'select * from dw.person_info'
    and model_set_version = 'model-set-A'
  group by detection_batch_id
  order by max(detected_at) desc, detection_batch_id desc
  limit 1
) latest
on r.detection_batch_id = latest.detection_batch_id
```

这个方案也能用。

但它用的是批次内最大 `detected_at`，不如任务表的 `finished_at` 清晰。

## 七、直接追加如何模拟覆盖

覆盖请求：

```json
{
  "overwrite": "true",
  "forceRun": "true"
}
```

实际执行：

```text
创建新检测任务
  -> 生成新 detection_batch_id
  -> 追加写入 dw.raha_detection_result
  -> 旧 detection_batch_id 不删除
  -> 当前结果查询取最新成功 detection_batch_id
```

第一次检测：

| detection_batch_id | finished_at | 当前 |
| --- | --- | --- |
| detect-001 | 1000 | 是 |

第二次覆盖检测：

| detection_batch_id | finished_at | 当前 |
| --- | --- | --- |
| detect-001 | 1000 | 否 |
| detect-002 | 2000 | 是 |

旧数据还在。

只是查询当前结果时不再取它。

## 八、这个方案能解决什么

可以解决：

1. FMDB 不能删除的问题。
2. 重跑后结果重复展示的问题。
3. 用户只关心最新结果的问题。
4. 历史结果审计保留的问题。
5. 实现复杂度过高的问题。

不需要：

1. 删除旧批次。
2. 更新旧批次状态。
3. 单独维护复杂覆盖关系链。
4. 给每行结果更新 `is_current`。

## 九、这个方案不能解决什么

它不能解决所有覆盖语义。

### 1）不能物理减少存储

旧结果一直保留。

数据会越来越多。

需要后续做归档或离线清理。

### 2）不能自动让所有旧查询变正确

如果调用方直接查：

```sql
select * from dw.raha_detection_result
where dataset_id = 'person_info'
```

它仍然会看到所有历史批次。

必须提供“当前结果查询”或“当前结果视图”。

### 3）并发覆盖仍需要处理

如果两个覆盖任务同时执行，可能都会成功追加。

最终谁是当前结果，要靠：

```text
finished_at desc, created_at desc, job_id desc
```

来稳定排序。

更严格的方案仍然需要同覆盖组加锁。

### 4）失败任务不能作为当前结果

必须只选择：

```text
job.status = SUCCEEDED
```

或者根据业务允许：

```text
SUCCEEDED、PARTIAL_SUCCESS
```

不要选失败任务的结果。

## 十、是否还需要 `overwrite_enabled`

可以不需要独立物理列。

第一期建议：

1. 请求里保留 `overwrite=true`。
2. 任务表不新增 `overwrite_enabled` 列。
3. 把覆盖意图写入 `result_summary_json`。
4. 查询当前结果时按最新成功批次判断。

示例：

```json
{
  "overwrite": true,
  "overwriteMode": "APPEND_LATEST_BATCH",
  "currentSelectRule": "latest_success_job"
}
```

这样不改表结构也能表达清楚。

## 十一、是否还需要覆盖日志表

第一期可以不需要。

如果只要求：

```text
展示最新结果
```

那么直接追加加最新批次查询足够。

但如果未来要求：

1. 明确知道哪次覆盖了哪次。
2. 显示覆盖原因。
3. 做覆盖审批。
4. 做跨批次审计链。

那时再加覆盖日志表。

所以推荐：

```text
第一期不用覆盖日志表。
第二期再加。
```

## 十二、对采样和训练是否适用

### 1）检测最适合

检测结果天然是批次型结果。

同一输入、同一模型集合下，最新检测批次可以作为当前结果。

所以检测最适合“追加 + 最新批次”。

### 2）采样谨慎适用

采样也可以按最新 `sampleBatchId` 看当前结果。

但训练通常会显式传 `sampleBatchId`。

如果采样覆盖后自动切换最新批次，可能导致训练输入变化不明显。

建议采样仍然显式返回新 `sampleBatchId`，训练由用户选择。

### 3）训练不建议覆盖

训练产生模型集合。

模型集合应该不可变。

同一批数据重新训练，应生成新 `modelSetVersion`。

检测时选择最新发布模型或显式模型集合。

不要用“覆盖”去修改旧模型。

## 十三、最简落地方案

### 第一步：新增当前检测结果查询

新增仓储方法：

```text
findLatestDetectionBatch(datasetId, inputReference, modelSetVersion)
```

返回：

```text
detectionBatchId
```

再按该批次读取结果。

涉及文件：

| 文件 | 修改 |
| --- | --- |
| `DetectionResultRepository.java` | 增加查询最新批次方法 |
| `FmdbDetectionResultRepository.java` | 实现按最新成功任务查询 |
| `RahaDetectionUdfService.java` | 当前结果导出时使用最新批次 |

### 第二步：`overwrite=true` 直接转成强制新跑

规则：

```text
overwrite=true 等价于 forceRun=true + 当前结果取最新批次
```

不做删除。

不做旧数据更新。

涉及文件：

| 文件 | 修改 |
| --- | --- |
| `RahaTaskExecutionRequest.java` | 增加 `overwrite` |
| `RahaTaskRequestFactory.java` | 解析覆盖参数 |
| `RahaJobOrchestrator.java` | overwrite 时创建新任务 |

### 第三步：返回中说明当前批次

检测返回增加：

| 字段 | 说明 |
| --- | --- |
| `overwrite` | 是否按覆盖语义执行 |
| `currentBatchId` | 当前有效检测批次 |
| `currentSelectRule` | `LATEST_SUCCESS_BATCH` |

如果不想加列，可以先写入 ZIP 报告和 `result_summary_json`。

## 十四、推荐查询视图

如果 FMDB 支持建视图，建议创建：

```text
dw.raha_detection_result_current
```

视图逻辑：

```text
每个 dataset_id + input_reference + model_set_version
只返回最新成功 detection_batch_id 的结果
```

这样用户不用每次手写复杂 SQL。

如果不支持视图，就在 Java 仓储里封装。

## 十五、最终建议

你的简化思路是对的：

```text
直接追加 + 最新时间戳
```

但建议改成更准确的说法：

```text
直接追加 + 最新成功批次
```

最终推荐规则：

1. 检测覆盖不删除旧结果。
2. `overwrite=true` 创建新检测批次。
3. 当前结果按最新成功任务批次读取。
4. 旧结果保留用于审计。
5. 第一期不加覆盖日志表。
6. 第一期不加 `overwrite_enabled` 物理列。
7. 后续需要审计链时，再加覆盖日志表。

这样实现最简单，也最符合 FMDB 不能删除的现实约束。
