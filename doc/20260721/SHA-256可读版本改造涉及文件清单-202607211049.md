## 一、修订结论

本次修订采纳“可读唯一值优先，减少冗余字段”的方案。

如果字段本身承担的是业务定位、版本选择、文件定位职责，并且可读值已经能够做到全局唯一或业务范围内唯一，就不需要额外保留一份哈希摘要字段。否则会带来字段膨胀、对象构造复杂、仓储读写复杂、兼容分支增加等问题。

因此，第一期不建议新增 `model_set_key`、`model_set_digest`、`model_key`、`model_digest`、`is_latest` 这类字段。推荐直接复用现有字段：

| 现有字段 | 新取值规则 | 是否新增哈希字段 |
|---|---|---|
| `model_set_version` | 改为可读且唯一的表级模型集合版本 | 否 |
| `model_version` | 改为可读且唯一的字段级模型版本 | 否 |
| `model_path` | 使用可读安全文件名或路径 | 否 |
| `modelName` | 使用库名、表名、字段名生成可读名称，继续放在现有模型载荷中 | 否 |
| `snapshot_id` | 可改为可读唯一快照号，但要保证同一业务版本稳定 | 否 |
| `training_snapshot_id` | 可改为可读唯一训练快照号 | 否 |

需要继续保留哈希的字段，只限于内容指纹、隐私保护、幂等、行身份、单元格身份等场景，例如 `row_content_hash`、`cell_id`、`schema_hash`、`config_version`、`idempotent_key`、`valueHash`。

## 二、为什么不再建议额外保留模型哈希

前一版文档偏保守，建议给模型链路新增可读键和摘要字段。重新评估后，这对当前工程收益不大：

1. `model_set_version` 已经是模型集合主标识，直接把它改成可读唯一值即可。
2. `model_version` 已经是字段模型主标识，直接把它改成可读唯一值即可。
3. 历史哈希版本仍可作为旧值保留在同一个字段里，不需要新字段兼容。
4. 模型完整性不依赖模型版本是否是哈希，真正需要校验时可以校验 `model_payload_json`、模型文件内容或训练元数据。
5. 新增摘要字段会让领域对象、表结构、仓储接口、测试断言都变复杂。

结论是：模型链路的业务标识应直接改成可读唯一值，哈希不再作为模型版本的默认形态。

## 三、唯一性规则建议

### 3.1 表级模型集合版本

推荐格式：

```text
<database>.<table>@<yyyyMMddHHmmss>
```

并发训练或同秒重复训练时，追加递增序号：

```text
<database>.<table>@<yyyyMMddHHmmss>.<seq>
```

示例：

```text
dw.customer@202607211127.001
```

唯一性边界：

- 同一 `dataset_id` 下 `model_set_version` 必须唯一。
- 如果不同库表可能共用一个逻辑 `dataset_id`，则版本里必须包含完整库表名。
- 如果存在同秒并发训练，必须通过序号或任务内计数保证不冲突。

### 3.2 字段级模型版本

推荐格式：

```text
<database>.<table>.<column>@<yyyyMMddHHmmss>.<seq>
```

示例：

```text
dw.customer.phone@202607211127.001
```

唯一性边界：

- 同一 `dataset_id`、同一 `column_name` 下 `model_version` 必须唯一。
- 字段级版本应与所属 `model_set_version` 使用同一版本时间和序号。
- 文件名使用安全编码后的字段级版本，不需要再生成哈希文件名。

### 3.3 快照和训练快照

快照可以可读化，但要注意它会参与 `cell_id` 的生成。为了避免同一业务版本反复加载后单元格身份变化，建议优先使用业务版本：

```text
snapshot_<database>.<table>@<sourceVersion>
train_<database>.<table>@<modelVersion>
```

如果没有 `sourceVersion`，才使用当前时间和序号：

```text
snapshot_dw.customer@202607211127.001
train_dw.customer@202607211127.001
```

这类字段同样不需要并行保存哈希字段。

## 四、仍然必须保留哈希的场景

| 字段或对象 | 保留原因 |
|---|---|
| `row_content_hash` | 表示行内容指纹，用于变更检测和去重。 |
| `row_id` | 当前无业务键时等于内容指纹，有业务键时也用于稳定行身份，不建议直接暴露原始主键。 |
| `cell_id` | 单元格坐标稳定标识，避免长文本拼接和敏感信息暴露。 |
| `valueHash` | 值级隐私保护和特征回放需要。 |
| `schema_hash` | 表结构摘要，适合做兼容校验。 |
| `config_version` | 任务配置摘要，用于幂等和复现。 |
| `idempotent_key` | 任务幂等键，不应改成可读值。 |
| `strategyId`、`configurationHash` | 策略配置去重和复现。 |
| `feature_dictionary_version` | 如果当前用作特征字典内容摘要，可以继续保留。 |
| `cluster_version` | 如果当前用作聚类结果摘要，可以继续保留。 |

判断标准：如果字段表达的是“内容是否相同”，保留哈希；如果字段表达的是“业务对象是谁、版本是哪次”，使用可读唯一值。

## 九、推荐修改顺序

1. 新增 `ModelSourceKey` 和 `ModelReadableVersioner`，先固定可读唯一规则。
2. 新增 `FmdbSqlSourceTableResolver`，统一从 SQL 中提取第一个库表名。
3. 修改 `ColumnModelVersioner` 和 `RahaTrainService`，让新训练直接产出可读 `model_set_version`、`model_version`、`modelName`。
4. 修改 `FileColumnModelStore`，去掉 64 位哈希强校验，生成可读安全文件名。
5. 修改 `FmdbModelStore`，确认模型表和 `model_payload_json` 写入可读版本。
6. 扩展模型仓储接口和 FMDB 实现，支持按 `dataset_id` 查最新已发布模型集合。
7. 修改 `RahaTaskRequestFactory`，未传 `modelSetVersion` 时先解析出统一模型集合版本。
8. 修改 `RahaDetectionUdfService`，把 `modelSetVersion` 入参改为可选，输出实际使用版本。
9. 调整 `RahaDetectService` 默认逻辑，避免版本为空时按字段混用模型。
10. 更新 README 和相关测试。

## 十、验收标准

1. 新训练产出的 `model_set_version` 直接可读，并且同一数据集下唯一。
2. 新训练产出的 `model_version` 直接可读，并且同一字段下唯一。
3. 本地模型文件名不再是纯 64 位哈希。
4. 不新增模型摘要类字段，也不新增模型键类字段。
5. `F_DW_DETRUN` 表输入不传 `modelSetVersion` 时，可以自动选择该表最新已发布模型集合。
6. `F_DW_DETRUN` 显式传历史 64 位哈希 `modelSetVersion` 时，历史调用仍可用。
7. `F_DW_DETRUN` SQL 输入不传 `modelSetVersion` 时，`input_reference` 使用 SQL 中第一个 `库名.表名`，并按该表自动选择最新模型集合。
8. SQL 中存在多个表时，只取解析顺序中的第一个表，不再要求额外模型查找参数。
9. 检测服务实际加载的是同一个模型集合里的字段模型。
10. 内容指纹、行身份、单元格身份、幂等键等哈希字段保持不变。

## 十一、最终建议

可读唯一值可以直接作为模型链路主标识，不必再加一层哈希摘要。当前工程更适合采用“替换值形态，不扩字段数量”的改造方式：

- 新模型的 `model_set_version` 直接写可读唯一值。
- 新模型的 `model_version` 直接写可读唯一值。
- 新模型文件名直接使用字段级可读安全名。
- SQL 输入的 `input_reference` 直接写第一个解析到的 `库名.表名`。
- 历史哈希版本继续在原字段中兼容读取。
- 哈希只保留在确实需要内容指纹、隐私保护和幂等的字段上。

这样可以显著降低数据库结构变更、领域对象膨胀和仓储兼容分支数量，更符合新工程保持简洁的要求。
