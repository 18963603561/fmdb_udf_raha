## 七、为什么会这么慢

### 7.1 每次函数都重复跑公共流水线

采样、训练、预测都重新执行了 `LOAD_DATA`、`PROFILE`、`GENERATE_STRATEGY`、`RUN_STRATEGY`、`GENERATE_FEATURE` 等公共阶段。对于同一个 `dw.person_info limit 400`、同一个 schema 和同一个数据快照，这些中间结果理论上可以复用，但当前 SQL 调用每次都重新生成。

### 7.2 聚类阶段实现成本高

`CLUSTER` 对 6 个字段、约 2400 个单元格特征执行字段级聚类。当前默认配置：

| 配置 | 值 |
| --- | --- |
| `raha.clustering.distance-metric` | `COSINE` |
| `raha.clustering.target-cluster-count` | 2 |
| `raha.clustering.max-sample-count` | 500 |
| `raha.resource.max-parallel-columns` | 4 |

即使数据只有 400 行，聚类仍然占用 2 分半以上，说明这里更可能受算法实现、Spark action 次数、列级并发与本地 `local[2]` 资源不匹配影响，而不是单纯数据量过大。

### 7.3 策略执行被采样、训练、预测重复触发

策略配置默认启用 `OD,PVD`，本次训练生成 36 个策略计划，分布为 `OD=12, PVD=24`。采样、训练、预测都经历 `RUN_STRATEGY`，导致重复开销：

| 任务 | `RUN_STRATEGY` 耗时 |
| --- | ---: |
| 采样 | 31.786 秒 |
| 训练 | 48.196 秒 |
| 预测 | 28.273 秒 |

预测阶段本来只需要使用已发布模型打分，但现在仍然重新跑画像和策略，导致预测前置准备远大于真正预测。

### 7.4 本地 Spark SQL 启动开销固定存在

三次都是独立 `spark-sql --master local[2]` 进程，每次都会启动 SparkSession、加载 Hive、注册函数、编译计划和执行一个只返回一行 JSON 的 UDF 查询。这个固定成本在小数据场景下非常明显。

### 7.5 任务表写入方式放大了 Hive/HDFS 开销

日志反复出现：

```text
The directory hdfs://hdfs:9000/user/hive/warehouse/dw.db/raha_job_run/... was not found. Was it deleted very recently?
The directory hdfs://hdfs:9000/user/hive/warehouse/dw.db/raha_job_stage_attempt/... was not found. Was it deleted very recently?
```

这些告警来自前面清理过外部表 HDFS 分区目录，任务运行时每个阶段多次追加写任务表和阶段表。因为 `DIRECT_APPEND` 不做幂等合并，`RUNNING` 和完成态各写一行，读取时也能看到重复状态记录。每次追加写都可能触发 Hive metastore 分区查询、连接重试和 HDFS 目录探测，形成阶段间固定开销。

### 7.6 并发参数和实际执行资源不匹配

代码中的 `BoundedParallelExecutor` 对策略和字段都使用固定线程池，默认：

| 配置 | 值 |
| --- | --- |
| `raha.resource.max-parallel-strategies` | 4 |
| `raha.resource.max-parallel-columns` | 4 |
| Spark master | `local[2]` |

本地 Spark 只有 2 个执行线程，但 Raha 内部列并发和策略并发是 4。对于 Spark action 密集型阶段，这会造成线程争用、作业排队和元数据连接竞争。`phone` 模型训练期间出现 HDFS 临时 ORC 文件找不到，也和并发写同一张表、同一 HDFS 临时提交目录的稳定性风险有关。

## 八、优化解决方案

### 8.1 最高优先级：复用采样与训练中间产物

目标：避免训练和预测重复执行 `PROFILE`、`GENERATE_STRATEGY`、`RUN_STRATEGY`、`GENERATE_FEATURE`、`CLUSTER`。

建议：

1. 以 `datasetId + schemaHash + snapshotId + configVersion` 为键缓存画像、策略计划、策略结果、特征字典和聚类结果。
2. 训练阶段如果传入 `sampleBatchId`，优先读取采样阶段已生成的策略结果、特征和聚类，而不是重新计算。
3. 预测阶段只加载模型需要的特征字典和模型，不再重新跑采样用的聚类。

预期收益：采样后训练可少跑约 229 秒以上的重复准备阶段；预测可少跑约 53 秒以上的准备阶段。小数据场景下端到端耗时可能从 12 分钟级降到 3 到 5 分钟级，甚至更低。

### 8.2 优化 `CLUSTER` 阶段

目标：解决最大单点瓶颈。

建议：

1. 对 400 行以内的小数据，增加轻量聚类路径，避免调用完整 Spark 聚类或重复 action。
2. 将 `raha.clustering.max-sample-count` 从 500 下调到与业务标注预算匹配，例如 300 或 200。
3. 对采样和训练复用同一份聚类结果，避免同一批数据聚类两次。
4. 对文本类字段可以先使用策略覆盖分数直接采样，聚类只作为补充，而不是必经路径。
5. 在阶段日志中补充每列聚类耗时和样本量，确认是否某个字段拖慢整体。

预期收益：当前采样和训练聚类合计 320.860 秒，是最值得优化的地方。

### 8.3 优化 `RUN_STRATEGY` 阶段

目标：减少重复策略计算和无效策略数量。

建议：

1. 预测阶段默认不要重新执行策略，只使用训练时发布的模型和特征字典。
2. 开启策略过滤或指定字段白名单，避免对不需要预测或没有模型的字段继续跑策略。
3. 对 `person_info` 这类结构化字段先保留收益最高的策略类型，例如优先保留 PVD 格式、长度、字符集策略，对纯数值列保留必要 OD 策略。
4. 将策略运行结果持久化并按快照复用。

预期收益：三次任务 `RUN_STRATEGY` 合计 108.255 秒，复用后可显著下降。

### 8.4 降低任务表和阶段表写入成本

目标：减少阶段间 27 到 42 秒的固定开销。

建议：

1. 把 `raha_job_run`、`raha_job_stage_attempt` 从追加状态流水表改为快照表或支持按 job/stage 覆盖最新状态。
2. 每个阶段只在开始和结束写必要字段，减少中间 `jobRepository.save` 调用。
3. 分区目录清理后立即执行 `MSCK REPAIR TABLE` 或按需预创建当月分区目录。
4. 对任务表写入使用批量写或延迟汇总写，阶段结束后一次性写完整状态。
5. 对小数据验证场景，可以临时关闭非必要列产物持久化，保留最终结果表即可。

预期收益：每次任务可减少十几到几十秒的 Hive/HDFS 元数据开销。

### 8.5 调整本地验证并发

目标：让 Raha 内部并发与 `local[2]` 匹配。

建议：

1. 本地 SQL 验证时设置 `raha.resource.max-parallel-columns=2`。
2. 本地 SQL 验证时设置 `raha.resource.max-parallel-strategies=2`。
3. 如果使用真实 Spark 集群，再根据 executor core 数把并发调高。
4. 对模型发布写入同一张表的阶段，先串行发布或按字段分区写，避免 HDFS 临时目录竞争。

预期收益：不一定降低所有阶段耗时，但能减少并发写冲突和 Spark 作业争用，提升稳定性。

### 8.6 优化 SQL 调用方式

目标：减少每次独立 `spark-sql` 进程的固定启动成本。

建议：

1. 如果仍要求只能用 SQL，可以使用同一个 `spark-sql` 会话顺序执行采样、标注插入、训练和预测，避免三次独立启动。
2. 将函数注册和固定配置放到初始化 SQL 中，减少每次执行前的准备语句。
3. 保留 SQL 调用，不使用 app 提交，但可以用同一 SQL session 执行多条语句。

预期收益：三次固定 SQL 外层开销合计约 52.295 秒，可明显减少。

### 8.7 改善可观测性

目标：以后能直接定位到字段、策略和 Spark action 级别。

建议：

1. 开启 Spark event log，并挂载持久化目录，例如 `/tmp/spark-events` 或 HDFS 事件目录。
2. 启动 Spark History Server，保留每次 SQL 的 application、job、stage、task 明细。
3. 在 Raha 阶段表 `summary_json` 中写入每列耗时、每个策略族耗时、Spark action 次数和结果行数。
4. 在 `BoundedParallelExecutor` 中记录每个 workKey 的开始、结束、耗时和线程等待时间。
5. 对 `SparkSqlFmdbTableGateway.append` 增加耗时日志，拆分为 metastore 查询、DataFrame 写入、分区刷新三个子耗时。

## 九、最终结论

目前耗时主要不在最终采样选择、模型训练或预测打分本身，而在三类地方：

1. 重复公共流水线，尤其是 `CLUSTER` 和 `RUN_STRATEGY`。
2. 小数据场景下 Spark SQL 独立进程启动和 UDF 执行框架固定成本偏高。
3. 每个阶段频繁追加写 Hive/HDFS 任务表，叠加外部表分区目录曾被清理后的元数据探测和连接重试。

最优先的优化路线是：先让训练复用采样的画像、策略、特征、聚类结果；再让预测跳过采样/训练前置策略流水线，只加载已发布模型和必要特征；最后减少阶段表写入次数并启用 Spark event log。按本次实测，单是避免重复 `CLUSTER` 和 `RUN_STRATEGY`，理论上就能省掉数百秒级耗时，这是当前收益最大的方向。
