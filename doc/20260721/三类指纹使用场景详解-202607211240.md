# 三类指纹使用场景详解

生成时间：2026-07-21 12:40

## 一、先给结论

`executionInputFingerprint`、`configVersion`、`idempotentKey` 是三层不同粒度的标识。

它们不是并列关系，而是递进关系：

```text
executionInputFingerprint
  -> 参与生成 configVersion
    -> 参与生成 idempotentKey
```

一句话理解：

| 字段 | 类比 | 解决的问题 |
| --- | --- | --- |
| `executionInputFingerprint` | 本次业务输入小票 | 区分这次到底用哪个采样批次、标注批次、模型集合 |
| `configVersion` | 完整任务档案袋编号 | 区分完整任务配置是否完全一样 |
| `idempotentKey` | 提交查重码 | 判断这次提交是不是已经存在同一个任务 |

## 二、为什么不能只用一个字段

如果只用一个字段，也能做查重，但排查会很痛苦。

举例：

同样是检测 `dw.person_info`，第二次结果变了。

如果只有一个大哈希，你只能知道“变了”，但不知道为什么变：

1. 是换了模型集合？
2. 是换了 SQL？
3. 是换了策略配置？
4. 是换了资源配置？
5. 是换了缺失模型策略？

现在拆成三层后，可以逐层定位：

1. `executionInputFingerprint` 变了，说明调用级业务输入变了。
2. `configVersion` 变了但 `executionInputFingerprint` 没变，说明基础配置变了。
3. `idempotentKey` 一样，说明系统会认为是同一个任务。

## 三、第一层：`executionInputFingerprint`

### 1）它是什么

`executionInputFingerprint` 是“调用级业务输入指纹”。

它只关心这一次调用额外指定的业务差异。

它不是完整配置版本，也不是最终查重键。

### 2）它在哪里生成

生成位置：

```text
RahaTaskRequestFactory
```

不同任务生成方式不同。

采样：

```text
sha256(
  "sampling"
  + canonicalInput(input)
  + samplingDiscriminator(options)
)
```

训练：

```text
sha256(
  "training"
  + trainingBatchReference
  + canonicalInput(input)
  + propagationMethod
  + modelNamePrefix
)
```

检测：

```text
sha256(
  "detection"
  + canonicalInput(input)
  + modelSetVersion
  + missingModelPolicy
)
```

### 3）它用在哪里

它会写入：

```text
RahaJobConfig.executionInputFingerprint
```

然后进入：

```text
RahaJobConfig.toCanonicalString()
```

最终影响：

```text
configVersion
```

也就是说，它本身不是最终查重键，但它会让配置版本变化。

### 4）它解决什么问题

它解决的是：

同一种任务、同一张表、同一套配置下，不同业务输入不能被误判为同一个任务。

例如：

1. 训练时换了标注批次。
2. 检测时换了模型集合。
3. 采样时换了采样轮次。

这些变化不一定属于基础配置变化，但必须影响任务语义。

## 四、第二层：`configVersion`

### 1）它是什么

`configVersion` 是“完整任务配置版本”。

它把整个 `RahaJobConfig` 做规范化后再哈希。

它比 `executionInputFingerprint` 大。

### 2）它在哪里生成

生成位置：

```text
ConfigVersioner.versionOf(config)
```

底层逻辑：

```text
sha256(config.toCanonicalString())
```

### 3）它包含什么

`configVersion` 包含：

| 内容 | 说明 |
| --- | --- |
| `jobType` | 采样、训练、检测 |
| `datasetId` | 数据集 |
| `snapshotId` | 数据快照 |
| `inputReference` | 表名或 SQL |
| `rowIdentityConfig` | 行身份规则 |
| `randomSeed` | 随机种子 |
| `strategyConfig` | 策略配置 |
| `featureConfig` | 特征配置 |
| `modelConfig` | 模型配置 |
| `clusteringConfig` | 聚类配置 |
| `samplingConfig` | 采样配置 |
| `resourceConfig` | 资源配置 |
| `failureToleranceConfig` | 失败容忍配置 |
| `executionConfigFingerprint` | 运行配置指纹 |
| `executionInputFingerprint` | 调用级业务输入指纹 |

### 4）它用在哪里

它主要用于：

1. 保存到任务表的 `config_version`。
2. 生成任务级 `idempotentKey`。
3. 阶段执行时生成 `ArtifactVersion`。
4. 校验提交任务和执行任务的配置一致。

### 5）它解决什么问题

它解决的是：

这次任务的完整配置是否和上次完全一样。

例如：

同样检测 `dw.person_info`，模型集合也没变，但你把策略配置从 `OD + PVD` 改成 `OD`。

这时：

```text
executionInputFingerprint 可能不变
configVersion 会变
idempotentKey 也会变
```

因为完整配置已经不同，不能复用旧任务。

## 五、第三层：`idempotentKey`

### 1）它是什么

`idempotentKey` 是“任务提交查重键”。

它是系统真正用来判断：

```text
这次提交是不是和之前某个任务一样
```

的字段。

### 2）它在哪里生成

生成位置：

```text
IdempotencyKeyGenerator.generate(config, configVersion)
```

核心逻辑：

```text
sha256(
  jobType
  + datasetId
  + inputReference
  + snapshotId
  + configVersion
)
```

### 3）它用在哪里

使用位置：

```text
JobRepository.findByIdempotentKey(datasetId, idempotentKey)
```

保存位置：

```text
raha_job_run.idempotent_key
```

处理逻辑：

1. 提交任务前先生成 `idempotentKey`。
2. 用 `datasetId + idempotentKey` 查任务表。
3. 如果查到已有任务，就认为重复提交。
4. 如果没有查到，就创建新任务。

### 4）它解决什么问题

它解决的是：

重复提交时不要重复执行。

例如用户连点两次“开始检测”，两次请求完全一样。

系统应该只执行一次，第二次直接复用已有任务。

## 六、场景一：采样重复执行

### 第一次请求

```json
{
  "sourceType": "SQL",
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info limit 450",
  "labelingBudget": "300",
  "samplingRound": "4",
  "requestId": "r1"
}
```

生成过程：

```text
executionInputFingerprint
  = sha256("sampling" + 输入规范化 + samplingRound=4 + 已有标签摘要)

configVersion
  = sha256(完整采样配置 + executionInputFingerprint)

idempotentKey
  = sha256(SAMPLING + person_info + SQL + snapshotId + configVersion)
```

系统查：

```text
raha_job_run
where dataset_id = 'person_info'
  and idempotent_key = 当前 idempotentKey
```

没查到，创建新采样任务。

### 第二次只改 `requestId`

```json
{
  "sourceType": "SQL",
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info limit 450",
  "labelingBudget": "300",
  "samplingRound": "4",
  "requestId": "r2"
}
```

因为 `requestId` 不参与核心业务指纹：

```text
executionInputFingerprint 不变
configVersion 不变
idempotentKey 不变
```

处理结果：

```text
命中已有任务
不重新采样
返回复用任务
```

### 第三次改 `samplingRound`

```json
{
  "samplingRound": "5"
}
```

变化：

```text
executionInputFingerprint 变
configVersion 变
idempotentKey 变
```

处理结果：

```text
创建新采样任务
生成新 sampleBatchId
```

## 七、场景二：训练重复执行

### 第一次训练

```json
{
  "sampleBatchId": "sample-A",
  "modelNamePrefix": "person_info"
}
```

训练入口会解析：

```text
sample-A
  -> 找采样批次
  -> 找最新可训练标注批次 annotation-1
```

生成：

```text
executionInputFingerprint
  = sha256("training" + sample-A + annotation-1 + 输入规范化 + 传播方法 + 模型名前缀)
```

没查到旧任务，就创建训练任务。

输出：

```text
trainingBatchId = train-1
modelSetVersion = model-set-1
```

### 第二次还是同一批次

```json
{
  "sampleBatchId": "sample-A",
  "modelNamePrefix": "person_info"
}
```

如果最新可训练标注还是 `annotation-1`：

```text
executionInputFingerprint 不变
configVersion 不变
idempotentKey 不变
```

处理结果：

```text
复用已有训练任务
不重新训练
```

### 第三次上传了新标注

同一个采样批次下新增：

```text
annotation-2
```

再次训练：

```json
{
  "sampleBatchId": "sample-A",
  "modelNamePrefix": "person_info"
}
```

入口解析到最新可训练标注变成：

```text
annotation-2
```

变化：

```text
executionInputFingerprint 变
configVersion 变
idempotentKey 变
```

处理结果：

```text
创建新训练任务
生成新 modelSetVersion
```

## 八、场景三：检测重复执行

### 第一次检测

```json
{
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info",
  "modelSetVersion": "model-set-A",
  "missingModelPolicy": "PARTIAL"
}
```

生成：

```text
executionInputFingerprint
  = sha256("detection" + 输入规范化 + model-set-A + PARTIAL)

configVersion
  = sha256(完整检测配置 + executionInputFingerprint)

idempotentKey
  = sha256(DETECTION + person_info + SQL + snapshotId + configVersion)
```

没查到旧任务，创建检测任务。

### 第二次完全一样

还是：

```json
{
  "modelSetVersion": "model-set-A",
  "missingModelPolicy": "PARTIAL"
}
```

三者都不变：

```text
executionInputFingerprint 不变
configVersion 不变
idempotentKey 不变
```

处理结果：

```text
复用已有检测任务
不重新写检测结果
```

### 第三次换模型集合

```json
{
  "modelSetVersion": "model-set-B"
}
```

变化：

```text
executionInputFingerprint 变
configVersion 变
idempotentKey 变
```

处理结果：

```text
创建新检测任务
用 model-set-B 重新检测
写入新的 detectionBatchId
```

### 第四次只改导出 ZIP 路径

```json
{
  "localWebRoot": "/tmp/new-web-root"
}
```

如果这个字段不进入任务配置和输入指纹：

```text
executionInputFingerprint 不变
configVersion 不变
idempotentKey 不变
```

处理结果：

```text
复用已有检测任务
不会重新检测
可能也不会重新生成你期望的新 ZIP
```

这就是为什么导出类参数和任务语义参数要分清楚。

## 九、三者变化关系表

| 变化内容 | executionInputFingerprint | configVersion | idempotentKey | 是否新任务 |
| --- | --- | --- | --- | --- |
| 只改 `requestId` | 不变 | 不变 | 不变 | 否 |
| 改采样轮次 | 变 | 变 | 变 | 是 |
| 改训练标注批次 | 变 | 变 | 变 | 是 |
| 改检测模型集合 | 变 | 变 | 变 | 是 |
| 改策略配置 | 不一定变 | 变 | 变 | 是 |
| 改资源配置 | 不一定变 | 变 | 变 | 是 |
| 改输入 SQL | 变 | 变 | 变 | 是 |
| 改快照 `snapshotId` | 变或不变 | 变 | 变 | 是 |
| 只改 ZIP 输出目录 | 通常不变 | 通常不变 | 通常不变 | 否 |

## 十、为什么 `configVersion` 已经有输入引用，还需要 `idempotentKey`

这是个容易混淆的点。

`configVersion` 是“配置内容版本”。

它可以告诉你：

```text
这个任务配置长什么样
```

但任务查重要考虑查询维度。

当前查重是：

```text
datasetId + idempotentKey
```

`idempotentKey` 把几个最关键的任务定位字段再包了一层：

1. `jobType`
2. `datasetId`
3. `inputReference`
4. `snapshotId`
5. `configVersion`

这样任务仓储可以直接用一个稳定查重键判断重复提交。

换句话说：

```text
configVersion 更像配置版本号
idempotentKey 更像任务提交唯一键
```

## 十一、为什么 `executionInputFingerprint` 不直接当幂等键

因为它太小。

它只表达调用级业务差异，不表达完整配置。

举例：

两次检测都使用：

```text
modelSetVersion = model-set-A
missingModelPolicy = PARTIAL
```

所以 `executionInputFingerprint` 可能一样。

但第二次把策略配置、特征配置或资源配置改了。

这时两个任务不能认为完全一样。

所以还需要 `configVersion` 包住完整配置。

## 十二、为什么 `configVersion` 不直接当幂等键

技术上可以，但当前设计没有这么做。

原因是 `idempotentKey` 明确表达“任务提交查重”语义。

它把任务定位字段和配置版本合在一起。

这样后续如果要调整配置版本的组成，不一定要改变任务查重入口的概念。

另外，任务表按：

```text
datasetId + idempotentKey
```

查询，比直接把 `configVersion` 当作业务查重键更清晰。

## 十三、最简单的记忆方式

可以按这个顺序记：

```text
executionInputFingerprint：这次用什么业务输入
configVersion：这次完整任务配置是什么
idempotentKey：这次提交是不是重复任务
```

再换成一句更直白的话：

```text
executionInputFingerprint 管“差异”
configVersion 管“版本”
idempotentKey 管“查重”
```

## 十四、当前工程里的实际问题

当前内部已经能算出这些字段，并用 `idempotentKey` 做任务级复用。

但外部 UDF 返回还不够清楚。

当前缺口：

1. 没有把 `executionInputFingerprint` 返回给用户。
2. 没有把 `configVersion` 返回给用户。
3. 没有把 `idempotentKey` 返回给用户。
4. 没有把 `reused=true/false` 返回给用户。
5. 任务复用时业务 payload 恢复还不完整。

建议返回中增加：

```json
{
  "executionInputFingerprint": "sha256...",
  "configVersion": "sha256...",
  "idempotentKey": "sha256...",
  "reused": true,
  "reuseReason": "JOB_IDEMPOTENT_HIT"
}
```

这样用户一眼能看出来：

1. 这次为什么被认为相同。
2. 哪一层发生变化。
3. 是新执行还是复用。

## 十五、最终建议

不建议合并这三个字段。

建议保持三层结构，但把它们暴露到统一返回和日志中。

最小改造建议：

1. 公共 UDF 返回字段增加 `configVersion`。
2. 公共 UDF 返回字段增加 `idempotentKey`。
3. 公共 UDF 返回字段增加 `reused`。
4. 调试模式下返回 `executionInputFingerprint`。

这样既保留工程上的分层，又让用户能看懂重复执行为什么复用。
