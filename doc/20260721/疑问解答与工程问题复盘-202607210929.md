## 三、10 个疑问逐条解答

### 1）`inline` 去掉是不是就没有问题

结论：

不是。

问题不在 `inline` 这个关键字本身，而在返回值形态和 SQL 写法是否匹配。

`F_DW_DETCOLLECT` 这类函数返回的不是普通单值，而是一个结构化结果。它更像下面这种形态：

```text
array<struct<
  status:string,
  datasetId:string,
  rowCount:int,
  ...
>>
```

如果直接写：

```sql
SELECT F_DW_DETCOLLECT('...');
```

Spark 会把它当成一列复杂对象。

结果类似：

| result |
| --- |
| `[{"status":"SUCCESS","datasetId":"person_info",...}]` |

如果写：

```sql
SELECT inline(F_DW_DETCOLLECT('...'));
```

Spark 会把 `array<struct>` 展开成普通列。

结果类似：

| status | datasetId | rowCount |
| --- | --- | --- |
| `SUCCESS` | `person_info` | `450` |

真正出问题的是这种写法：

```sql
SELECT inline(F_DW_DETCOLLECT('...')) AS result;
```

这会让 Spark 以为展开结果只对应一个别名 `result`，但实际结构体里有很多字段，于是字段数量对不上。

所以：

1. 如果只想看原始复杂对象，可以去掉 `inline`。
2. 如果想得到一行多列的函数返回结果，不能去掉 `inline`。
3. 正确写法是 `SELECT inline(...)`，不要给 `inline(...)` 整体只起一个别名。

当前状态：

本次已经通过正确展开或驱动侧调用绕过了这个 SQL 形态问题。

### 2）不能单独执行这 3 个函数是什么意思

结论：

可以单独执行，但要在 Spark Driver 侧执行，不能把内部还要跑 Spark 作业的函数下发到 Executor 里执行。

这里的关键点是 Spark 的执行模型。

普通 SQL UDF 通常在 Executor 里执行。

但这三个函数内部做的事情不是简单计算，而是会再次触发 Spark 作业，例如：

1. 执行 SQL 加载表数据
2. 做字段画像
3. 做策略计算
4. 写 Hive 或 HDFS
5. 生成 Excel 和 ZIP
6. 训练模型
7. 检测并写结果

如果外层 SQL 把 UDF 下发到 Executor，而 UDF 内部又尝试启动新的 Spark 作业，就会变成：

```text
Driver 提交 SQL
  Executor 执行 UDF
    UDF 内部再次提交 Spark 作业
```

这种嵌套 Spark 作业很容易卡住、超时，或者无法进入业务逻辑。

本次新增的驱动侧入口，本质是：

```text
Driver main 方法
  创建 SparkSession
  直接调用 F_DW_DETCOLLECT 实现
  直接调用 F_DW_DETTRAIN 实现
  直接调用 F_DW_DETRUN 实现
```

它仍然调用三函数的同一套实现，只是不让这些函数在 Executor 里被动执行。

举例：

不推荐：

```sql
SELECT inline(F_DW_DETCOLLECT('...'));
```

在当前实现中，更推荐：

```text
java main
  -> new SparkSession
  -> call collect udf implementation
```

当前状态：

已经通过 `RahaUdfDriverApp` 这种薄驱动入口解决本次闭环执行问题。

剩余问题：

如果未来要让业务方直接用 SQL 调三函数，需要把这类函数包装成真正的 Driver 侧服务、存储过程或任务提交接口，而不是普通 Executor UDF。

### 3）`commons-io` 冲突为什么要用 `maven-shade-plugin`

结论：

应该使用 `maven-shade-plugin`，当前 `pom.xml` 已经加入 shade 配置，并对 `POI`、`commons-io` 等依赖做了 relocation。

问题背景：

Spark 自带很多依赖。

例如 Spark 3.3.1 自己带了一套较旧版本的 `commons-io`。

本工程生成 Excel 时使用 `Apache POI`。新版本 POI 可能调用新版 `commons-io` 才有的方法。

当 Spark 的旧依赖优先级更高时，运行时会出现类似现象：

```text
代码编译通过
运行时生成 Excel 失败
异常指向 commons-io 或 POI 方法不存在
```

常见错误形态：

```text
NoSuchMethodError
ClassNotFoundException
NoClassDefFoundError
```

中文解释：

代码编译时看到的是新版本依赖，但运行时 Spark 把旧版本依赖放在更前面，导致实际加载的类不是编译时那套类。

`maven-shade-plugin` 的作用是把依赖重命名打包。

例如：

```text
org.apache.poi      -> raha.org.apache.poi
org.apache.commons.io -> raha.org.apache.commons.io
```

这样 Spark 自带的 `org.apache.commons.io` 和本工程需要的 `raha.org.apache.commons.io` 就不会再抢同一个类名。

当前状态：

当前 `pom.xml` 已经存在 `maven-shade-plugin`，并配置了：

1. `org.apache.poi`
2. `commons-io`
3. `net.lingala.zip4j`
4. 若干 POI 依赖链

本次 Excel 和 ZIP 已成功生成，说明本次闭环已经不再被该问题阻断。

注意事项：

1. 运行时必须使用 shade 后的包。
2. 如果部署时仍然使用未 shade 的原始 jar，依赖冲突可能重新出现。
3. `spark.driver.userClassPathFirst=true` 是补充手段，shade 才是更稳定的隔离方案。

### 4）为什么 1000 行表会被聚类上限挡住

结论：

`raha.clustering.max-sample-count=500` 挡住的是聚类输入规模，不是最终聚类个数。

你理解的“聚类”方向是对的：

聚类确实是把相似数据归到一起。

但是在真正归类之前，算法要先计算很多数据之间的相似度。

对 1000 行数据来说，最坏情况下需要比较的组合数接近：

```text
1000 * 999 / 2 = 499500
```

这还只是一个字段或一个聚类空间里的数量级。

如果有多个字段、多个特征，计算量会继续放大。

所以默认上限 `500` 的含义是：

```text
如果输入样本超过 500 行，就不进入重聚类逻辑，避免测试环境被拖死。
```

这不是说 1000 行不会被聚类，也不是说提升到 2000 就不做聚类。

提升到 2000 的意思是：

```text
允许最多 2000 行进入聚类计算。
```

如果表有 1000 行，且上限是 2000，那么仍然会做聚类。

只是不会因为“输入行数超过上限”而直接失败。

举例：

| 表行数 | 上限 | 是否进入聚类 |
| --- | --- | --- |
| 450 | 500 | 进入 |
| 1000 | 500 | 被挡住 |
| 1000 | 2000 | 进入 |
| 3000 | 2000 | 被挡住 |

当前状态：

本次闭环最终把采样 SQL 改成：

```sql
select * from dw.person_info limit 450
```

这样采样阶段控制在默认上限内，检测阶段仍然对完整 1000 行执行。

该问题对本次闭环已解决。

工程层面仍建议保留上限保护，因为聚类成本不是线性增长。

### 5）高 CPU 卡在 1000 行列内聚类，生产资源足够是否就没问题

结论：

资源更多会缓解，但不能从根本上消除复杂度问题。

这里的高 CPU 主要来自聚类阶段的相似度计算。

采样聚类不是简单扫一遍数据，而是要做类似下面的事情：

```text
取出字段值
计算画像
生成候选
计算相似度或距离
分组
选择代表样本
写采样任务
```

如果算法里存在大量两两比较，规模会接近二次增长。

简单估算：

| 行数 | 两两比较数量 |
| --- | --- |
| 500 | 124750 |
| 1000 | 499500 |
| 2000 | 1999000 |
| 5000 | 12497500 |

所以行数从 1000 到 2000，不是多一倍工作量，而可能接近多四倍。

生产环境资源足够时，可以这么优化：

1. 保留输入上限，不要无限放大 `raha.clustering.max-sample-count`。
2. 先抽样，再聚类，不要直接对全量数据聚类。
3. 按字段分批并行，但要限制最大并发，避免抢满集群。
4. 对人名、手机号、身份证、邮箱这类强格式字段，优先用 `PVD` 的格式、长度、字符集策略。
5. 对数值字段才重点使用 `OD`。
6. 对大表使用分层采样或随机采样，再做聚类。
7. 对重复值高的字段先按值去重，再聚类代表值。
8. 缓存中间数据，减少重复扫描表。

本次要求只用 `OD` 和 `PVD`，这本身已经比全策略更轻。

当前状态：

本次闭环通过 `limit 450` 解决了测试环境可跑通问题。

生产级性能问题还没有完全解决。

建议后续增加：

1. 聚类前去重
2. 最大候选对数量限制
3. 分层采样
4. 聚类耗时指标
5. 超限时自动降级为随机采样或规则采样

### 6）Java 三元表达式拆箱空指针是什么意思

结论：

这是 Java 自动拆箱导致的空指针，不是业务数据本身一定错。

问题可以用一个简化例子说明。

假设有一段代码：

```java
Double fallback = null;
double direct = 1.0d;
Double weight = useDirect ? direct : fallback;
```

看起来 `weight` 是 `Double`，似乎可以接收 `null`。

但 Java 三元表达式会先推断两个分支的公共类型。

因为一个分支是 primitive `double`，另一个分支是 `Double`，Java 可能会把 `Double` 分支拆箱成 `double`。

如果执行到 `fallback` 分支，就会发生：

```java
fallback.doubleValue()
```

而 `fallback` 是 `null`，于是报空指针。

本工程里的场景是：

1. 有些单元格有人工标注权重。
2. 有些单元格只有标签传播权重。
3. 有些单元格没有直接标签，也没有传播标签，权重应允许为空。

旧写法试图在一个三元表达式里同时处理：

```text
直接标签权重
传播标签权重
空权重
```

结果被 Java 自动拆箱坑到了。

修复后的思路是显式写分支：

```java
Double weight = null;
if (direct != null) {
    weight = Double.valueOf(direct.getSampleWeight());
} else if (propagated != null) {
    weight = Double.valueOf(propagated.getSampleWeight());
}
```

这样没有标签时 `weight` 就保持 `null`，不会触发拆箱。

当前状态：

该问题已经在 `TrainingArtifactMaterializationService` 中修复。

本次训练已经成功完成，说明该问题不再阻断当前闭环。

### 8）编排器 `RUNNING -> SKIPPED` 到底是什么问题

结论：

这是阶段状态机的合法流转不完整。

编排器执行一个阶段时，通常会先把阶段置为：

```text
RUNNING
```

然后交给具体阶段处理器执行。

处理器可能返回：

1. `SUCCEEDED`
2. `FAILED`
3. `PARTIAL_SUCCESS`
4. `SKIPPED`

举例：

假设训练流程里有一个“标签传播阶段”。

如果没有可传播的标签，阶段处理器可能认为：

```text
这个阶段没有必要执行，应该跳过。
```

于是它返回：

```text
SKIPPED
```

旧状态机只允许：

```text
PENDING -> SKIPPED
```

不允许：

```text
RUNNING -> SKIPPED
```

但实际执行链路是编排器先置为 `RUNNING`，处理器再返回 `SKIPPED`。

所以旧状态机就会误报：

```text
状态转换非法
```

这类错误会掩盖真正业务结果。

也就是说，阶段本身可能只是被正常跳过，但编排器把它当成异常。

当前状态：

`StageStatus` 已经允许：

```text
RUNNING -> SKIPPED
```

该问题已修复。

### 9）幂等键是什么，多次执行同一张表会怎样

结论：

幂等键是系统判断“这次请求是不是和之前同一个任务”的唯一指纹。

它的作用是避免重复创建任务、重复写结果、重复训练模型。

关键点：

`requestId` 通常是调用方追踪号，不一定进入幂等键。

所以只改 `requestId`，不一定能强制创建新任务。

举例：

第一次执行：

```json
{
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info",
  "requestId": "r1"
}
```

第二次执行：

```json
{
  "datasetId": "person_info",
  "sqlText": "select * from dw.person_info",
  "requestId": "r2"
}
```

如果幂等键不包含 `requestId`，系统会认为这两次是同一个任务。

可能出现的现象：

1. 第二次直接复用第一次结果。
2. 某些阶段显示 `SKIPPED`。
3. 返回的 `sampleBatchId` 或 `modelSetVersion` 看起来没有变化。
4. 检测结果没有重新生成。
5. 日志里能看到 checkpoint 或幂等记录复用。

这不是一定错误。

如果你的目标是防止重复执行，这是正确行为。

如果你的目标是强制重跑，就需要改变会进入幂等键的字段。

通常会影响幂等键的字段包括：

| 字段 | 作用 |
| --- | --- |
| `datasetId` | 数据集 |
| `snapshotId` | 数据快照 |
| `schemaHash` | 表结构 |
| `configVersion` | 配置指纹 |
| `strategyPlanVersion` | 策略计划版本 |
| `modelSetVersion` | 模型集合版本 |
| `sourceVersion` | 调用方自定义来源版本 |

常见 ID 的含义：

| ID | 含义 |
| --- | --- |
| `requestId` | 调用追踪号 |
| `sampleBatchId` | 采样批次 |
| `annotationBatchId` | 标注批次 |
| `trainingBatchId` | 训练批次 |
| `modelVersion` | 单列模型版本 |
| `modelSetVersion` | 多列模型集合版本 |
| `snapshotId` | 数据快照 |
| `detectionBatchId` | 检测批次 |

当前状态：

本次通过显式快照和批次版本完成闭环。

工程层面仍建议补充：

1. 幂等键组成文档。
2. `forceRun` 或 `overwrite` 参数。
3. 返回结果里明确标记 `reused=true/false`。

### 10）策略计划复用是什么意思，重复执行会复用什么

结论：

你的理解基本正确：重复执行时，系统可以复用数据库里已经保存的策略计划，但前提是快照和配置能对上。

策略计划不是简单的固定配置。

它可能依赖：

1. 字段画像
2. 数据类型判断
3. 空值比例
4. 值分布
5. 字段长度
6. 字符集
7. 策略启用配置

所以策略计划通常和数据快照绑定。

如果同一张表同一个快照重复执行：

```text
datasetId = person_info
snapshotId = snapshot-A
```

可以复用：

```text
snapshot-A 对应的策略计划
```

如果第二次换了快照：

```text
datasetId = person_info
snapshotId = snapshot-B
```

系统会倾向于重新生成策略计划。

本次问题复杂在这里：

训练产物里保存的 `trainingSnapshotId` 是训练阶段冻结后的快照。

它不一定等于原始数据快照。

如果系统用“原始快照”去查，而训练产物里存的是“训练快照”，就可能查不到同一份策略计划。

于是会出现：

```text
明明训练过
检测时却认为策略计划版本不同
```

当前修复思路是：

1. 同一快照优先复用已持久化策略计划。
2. 如果策略计划版本不同，不直接硬失败。
3. 降级为告警。
4. 继续用特征维度和字典映射守住检测输入结构。

可以复用的内容包括：

| 内容 | 复用条件 |
| --- | --- |
| 数据快照 | `snapshotId` 相同 |
| 字段画像 | 数据快照和字段结构相同 |
| 策略计划 | `datasetId + snapshotId` 命中 |
| 特征字典 | 策略计划和字段维度一致 |
| 标注任务 | `sampleBatchId` 相同 |
| 训练样本 | 训练批次和模型集合一致 |
| 候选模型 | 模型版本一致 |
| 已发布模型 | `modelSetVersion` 命中 |
| 检测结果 | 检测批次或幂等键命中 |

不能随便复用的内容包括：

1. 表结构变化后的模型。

5. 只改 `requestId` 但业务输入完全相同的任务。

本次闭环通过兼容性降级和特征字典映射跑通。

但从工程设计上看，快照血缘还需要进一步清晰化。

建议后续显式区分：

## 六、仍然需要关注的问题

### 1）驱动侧入口只是闭环验证方案

本次通过 `RahaUdfDriverApp` 跑通三函数。

这适合验证和批处理，但如果要让业务方直接通过 SQL 调用，还需要正式化。

建议：

1. 做成任务提交 API。
2. 或做成 Driver 侧存储过程。
3. 或把三函数语义变成服务端命令，而不是普通 Executor UDF。

### 2）采样聚类需要生产级降级策略

当前使用 `limit 450` 是为了跑通测试闭环。

生产上不能只靠手工限制 SQL。

建议增加：

1. 超限自动降级。
2. 大字段去重后聚类。
3. 分层采样。
4. 耗时统计。
5. 每字段最大候选对限制。

### 3）自动发布参数还不够产品化

当前已经自动发布，但还缺显式参数。

建议新增：

```json
{
  "autoPublish": "true"
}
```

默认值建议为 `true`。

### 4）幂等语义需要写清楚

目前只看执行结果，很难判断某次是新跑还是复用。

建议返回值增加：

```json
{
  "reused": false,
  "idempotencyKey": "...",
  "checkpointId": "..."
}
```

### 5）快照血缘需要拆清楚

当前训练快照、检测快照、原始表快照容易混用。

建议明确：

1. `sourceSnapshotId`
2. `trainingSnapshotId`
3. `detectSnapshotId`

这样策略计划复用、模型兼容性校验和检测解释都会更稳定。

### 6）ZIP 下载需要正式 Web 服务

本次静态服务地址是：

```text
http://127.0.0.1:18080
```

这只是本机临时下载入口。

长期应该由正式 Web 服务提供：

1. `baseUrl`
2. 鉴权
3. 文件过期策略
4. 下载审计日志
5. 容器端口或网关映射

## 七、会出现什么现象

### 1）如果继续直接用普通 SQL 执行复杂 UDF

可能现象：

1. SQL 卡住。
2. 不进入业务日志。
3. Spark 作业长时间无结果。
4. Executor 侧出现嵌套 Spark 作业问题。

推荐做法：

通过 Driver 侧入口执行。

### 2）如果采样行数超过聚类上限

可能现象：

1. 返回 `INPUT_LIMIT_EXCEEDED`。
2. 聚类阶段失败或跳过。
3. 采样任务无法生成。
4. 后续训练拿不到标注任务。

推荐做法：

控制采样输入规模，或提供自动降级。

### 3）如果没有使用 shade 后的 jar

可能现象：

1. Excel 生成失败。
2. ZIP 生成失败。
3. 运行时报 POI 或 commons-io 相关错误。

推荐做法：

部署 shade 后的 jar。

### 4）如果训练后不发布模型

可能现象：

1. 训练返回成功。
2. 检测返回失败。
3. 错误类似找不到已发布模型。

推荐做法：

默认 `autoPublish=true`。

### 5）如果重复执行但只改 `requestId`

可能现象：

1. 复用旧结果。
2. 部分阶段 `SKIPPED`。
3. 返回旧的批次或模型版本。
4. 结果表没有新增你预期的批次。

推荐做法：

使用明确的 `sourceVersion`、`snapshotId` 或 `forceRun`。

### 6）如果训练和检测快照不一致

可能现象：

1. 策略计划版本不同。
2. 特征字典版本不同。
3. 模型兼容性告警。
4. 严格模式下检测失败。

推荐做法：

显式传递并保存快照血缘。
