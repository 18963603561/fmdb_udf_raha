# SHA-256 字段梳理与模型可读版本改造分析

生成时间：2026-07-21 09:35

修订时间：2026-07-21 11:27

## 修订说明

结合后续评估，本方案调整为“可读唯一值优先，减少冗余字段”。模型集合版本、字段模型版本、模型文件名、快照标识这类业务定位字段，如果可读值本身已经能保证唯一，就不再建议额外新增 `*_digest`、`*_key`、`is_latest` 等并行字段。哈希只保留在内容指纹、隐私保护、幂等、行身份、单元格身份等确有必要的场景。

## 一、结论摘要

当前工程在 FMDB 物理表、JSON 字段、模型文件名和模型路径中都使用了 SHA-256 或 SHA-256 截断值。整体上可以分成两类：

- 第一类是指纹、幂等、隐私保护、单元格坐标、行内容摘要，这类字段继续使用 SHA-256 是合理的。
- 第二类是模型集合版本、字段模型版本、模型文件名、快照标识、训练快照标识，这类字段承担业务定位和人工排查职责，继续只用哈希会显著降低可读性。

建议优先改造的字段和对象是：

- `model_set_version`：当前是完整 SHA-256，建议直接改为可读且唯一的模型集合版本，例如 `dw.customer@202607210935.001`。
- `model_version`：当前是完整 SHA-256，建议直接改为可读且唯一的字段级模型版本，例如 `dw.customer.name@202607210935.001`。
- `model_path` 和本地模型文件名：当前依赖 `model_version`，文件名形如 `<64位哈希>.properties`，建议使用库表字段版本号生成安全文件名。
- `modelName`：当前存在于模型对象、模型载荷 JSON 和检测原因 JSON 中，但不是独立物理列，且默认只由前缀和字段名组成，建议补充库名、表名、字段名。
- `snapshot_id` 和 `training_snapshot_id`：当前默认是前缀加截断哈希，建议改为可读且唯一的快照编码；如果同一业务源版本需要稳定复用，必须让可读编码基于 `sourceVersion` 或显式版本号生成。

预测入口也需要同步调整：当前检测 UDF 明确要求传入 `modelSetVersion`。如果要实现“默认只输入表名或 SQL，系统自动选择最新模型”，需要在模型仓储层新增“按表查最新已发布模型集合”的能力，并把 `modelSetVersion` 改成可选参数。

## 五、当前设计带来的可读性问题

1. 快照标识无法直接看出来自哪个库、哪张表、哪个源版本。
2. 模型集合版本是完整哈希，检测结果里的 `model_set_version` 无法直接定位训练对象。
3. 字段模型版本和模型文件名是完整哈希，无法直接看出模型对应字段。
4. `modelName` 不是物理列，默认形态接近 `raha-字段名`，缺少库表上下文。
5. 检测 UDF 当前把 `modelSetVersion` 作为必填参数，不符合“只输入表名或 SQL 自动选择最新模型”的默认使用方式。
6. SQL 输入场景统一从 SQL 中提取第一个库表名作为 `input_reference` 和默认选模依据；如果存在关联查询、子查询或多个表，也只取解析顺序中的第一个表。

## 六、推荐命名规则

### 6.1 可读唯一值直接作为主标识

建议不要额外新增模型摘要字段，而是直接把现有业务定位字段的取值改成可读唯一值：

| 用途 | 复用字段 | 示例 |
|---|---|---|
| 表级模型集合版本 | `model_set_version` | `dw.customer@202607210935.001` |
| 字段级模型版本 | `model_version` | `dw.customer.name@202607210935.001` |
| 模型文件名 | `model_path` | `dw_customer__name__v202607210935_001.properties` |
| 快照编码 | `snapshot_id` | `snapshot_dw.customer@202607210935.001` |
| 训练快照编码 | `training_snapshot_id` | `train_dw.customer@202607210935.001` |

历史 64 位哈希版本继续作为旧数据保留在同一个字段里，新数据不再默认生成哈希版本。这样可以减少字段数量、对象属性和仓储兼容分支。

### 6.2 表级版本规则

推荐展示形态：

```text
<database>.<table>@<yyyyMMddHHmmss>
```

示例：

```text
dw.customer@202607210935
```

推荐文件安全形态：

```text
<database>_<table>__v<yyyyMMddHHmmss>
```

示例：

```text
dw_customer__v202607210935
```

### 6.3 字段级模型规则

推荐展示形态：

```text
<database>.<table>.<column>@<yyyyMMddHHmmss>
```

示例：

```text
dw.customer.phone@202607210935
```

推荐文件安全形态：

```text
<database>_<table>__<column>__v<yyyyMMddHHmmss>.properties
```

示例：

```text
dw_customer__phone__v202607210935.properties
```

### 6.4 特殊字符处理

库名、表名、字段名进入文件名或 URL 路径前必须做安全编码：

- 统一小写或保留原大小写需要提前确定，不要混用。
- 点号用于层级展示，文件名中替换为下划线。
- 空格、中文、特殊符号需要百分号编码或稳定转义。
- 超长表名和字段名需要长度限制，末尾可追加短摘要防冲突。
- 可读版本不能直接包含敏感值。

## 七、自动选模流程建议

### 7.1 表输入默认流程

```mermaid
graph TD;
  n01[输入表名];
  n02[规范表名];
  n03[查发布模型];
  n04[选择最新];
  n05[加载字段模型];
  n06[执行预测];
  n01 --> n02;
  n02 --> n03;
  n03 --> n04;
  n04 --> n05;
  n05 --> n06;
```

表输入默认逻辑：

1. 根据输入表名生成稳定 `datasetId`，当前表输入已有 `fmdb-table:` 加规范化表名的基础。
2. 如果用户没有传 `modelSetVersion`，按 `datasetId` 或规范化表名查询已发布模型集合。
3. 只允许选择 `PUBLISHED` 状态的模型集合。
4. 最新版本按 `published_at`、`created_at`、`state_version` 降序选择。
5. 如果用户传了 `modelSetVersion`，继续走精确版本加载，保证兼容历史调用。

### 7.2 SQL 输入默认流程

```mermaid
graph TD;
  n01[输入SQL];
  n02[解析首表];
  n03[写入引用];
  n04[查询模型];
  n05[选择最新];
  n06[执行预测];
  n01 --> n02;
  n02 --> n03;
  n03 --> n04;
  n04 --> n05;
  n05 --> n06;
```

SQL 输入建议：

1. SQL 原文仍用于实际读取数据，但持久化和输出的 `input_reference` 统一写第一个解析到的 `库名.表名`。
2. 如果 SQL 中存在多个表、关联查询、子查询或临时视图，只取解析顺序中的第一个表作为默认模型来源。
3. 如果用户显式传入 `modelSetVersion`，仍按指定模型集合版本执行，不受首表规则影响。
4. 如果第一个表没有库名前缀，应按 FMDB 当前默认库或工程约定补齐，保证最终 `input_reference` 是稳定的 `库名.表名`。

## 八、建议的落地改造清单

### 8.1 数据库结构

第一期不建议新增模型相关物理字段，优先复用现有字段承载可读唯一值：

| 表 | 复用字段 | 新语义 |
|---|---|---|
| `dw.raha_model_artifact` | `model_set_version` | 表级模型集合可读唯一版本。 |
| `dw.raha_model_artifact` | `model_version` | 字段级模型可读唯一版本。 |
| `dw.raha_model_artifact` | `model_path` | 可读安全模型文件路径。 |
| `dw.raha_model_artifact` | `model_payload_json` | 保存可读 `modelName` 和现有模型参数。 |
| `dw.raha_detection_result` | `model_set_version` | 实际使用的可读模型集合版本。 |
| `dw.raha_detection_result` | `model_version` | 实际使用的可读字段模型版本。 |

只有在后续确认查询性能不足，或者业务必须按物理列筛选 `model_name`、库名、表名时，才考虑新增字段。默认不要为了展示可读性引入并行字段。

| 模块 | 当前问题 | 改造建议 |

| `ColumnModelVersioner` | 只生成完整 SHA-256 的 `model_version` | 改为生成可读唯一的字段模型版本。 |
| `FileColumnModelStore` | 强校验 `modelVersion` 必须是 64 位十六进制 | 改成安全文件名校验，兼容读取历史哈希文件名。 |
| `FmdbModelStore` | 模型路径只包含模型版本 | 路径加入库名、表名、字段名和可读版本。 |
| `FmdbModelMetadataRepository` | 只按 `model_set_version` 查询 | 新增按表查最新已发布模型集合的查询方法。 |
| `ModelSetRepository` | 只支持精确版本查找 | 新增 `findLatestPublishedByDataset` 或 `findLatestPublishedByTable`。 |
| `FmdbInputSpec` | SQL 输入缺少首表来源 | SQL 原文继续用于读取数据，同时提取首个库表名生成稳定 `datasetId` 和持久化 `input_reference`。 |
| `RahaUdfRequestParser` | SQL 输入引用可能被摘要化 | 改为从 SQL 提取第一个 `库名.表名`，多个表只取第一个表。 |
| `RahaTaskRequestFactory` | 检测入口要求 `modelSetVersion` | 增加默认自动解析模型集合的重载方法。 |
| `RahaDetectionUdfService` | UDF 参数里 `modelSetVersion` 必填 | 调整为可选，未传时按表或 SQL 自动选模。 |
| `README.md` | 示例使用可读 `model-set-...`，代码实际生成 SHA | 更新文档，说明默认选模和手动指定方式。 |

### 8.3 兼容策略

1. 历史哈希版本继续可用，用户显式传 `modelSetVersion` 时按旧逻辑精确加载。
2. 新版本训练直接在 `model_set_version`、`model_version` 中写入可读唯一值。
3. 检测结果继续使用现有 `model_set_version` 和 `model_version` 字段，导出时自然展示可读值。
4. 本地模型文件迁移时支持同时读取旧的 `<64位哈希>.properties` 和新的可读文件名。
5. 对可读版本生成规则增加冲突处理，防止同名表、同名字段、同一时间戳并发训练冲突。

## 九、保留 SHA-256 的字段建议

以下字段不建议直接改成可读值：

- `row_content_hash`
- `cell_id`
- `valueHash`
- `schema_hash`
- `config_version`
- `idempotent_key`
- `strategyId`
- `configurationHash`
- `labelId`
- `sourceFingerprint`
- `annotation_task_id`

原因是这些字段主要承担去重、幂等、隐私保护、复现和稳定引用，不应该为了展示可读性牺牲唯一性和安全性。

## 十、优先级建议

第一优先级：

- 直接把 `model_set_version` 改为可读唯一值。
- 直接把 `model_version` 和 `model_path` 改为可读唯一值。
- 给 `modelName` 增加库表字段上下文，优先继续写入现有 `model_payload_json`。
- 检测入口支持不传 `modelSetVersion` 时自动按表选择最新发布模型。

第二优先级：

- 将 `snapshot_id`、`training_snapshot_id` 的默认生成规则改为可读唯一值。
- 检测结果继续使用现有版本字段展示可读模型信息。
- README 和 UDF 参数说明同步更新。

第三优先级：

- 评估 `sample_batch_id`、`annotation_batch_id` 是否需要可读批次号。
- 对有业务主键的数据增加受控的可读主键展示列，但不要替代 `row_id`。

## 十一、最终建议

这次问题的核心不是“不能用 SHA-256”，而是“业务定位字段不应该用 SHA-256 当默认展示值”。对于模型训练和预测主链路，建议直接把现有业务标识改成可读唯一值，而不是新增一批并行字段：

- `model_set_version` 用库名、表名、版本号定位。
- `model_version` 用库名、表名、字段名、版本号定位。
- `model_path` 使用字段级可读安全文件名。
- 默认预测只输入表名时，系统自动选择最新已发布模型集合。
- SQL 预测统一提取第一个库表名，复杂 SQL 或多表 SQL 也按第一个表选择默认模型。
- SQL 场景的 `input_reference` 不再默认写 SQL 哈希摘要，而是写第一个解析到的 `库名.表名`。

哈希继续留在内容指纹、隐私保护、幂等和稳定坐标字段上。这样既能提升模型排查、文件定位、结果导出和 UDF 调用的可读性，又不会因为新增冗余字段把新工程复杂度拉高。
