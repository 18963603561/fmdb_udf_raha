# Raha 工程配置项分析与外置化改造报告

## 1. 分析结论

本次对 `src/main/java`、`src/main/resources`、`pom.xml` 和现有测试进行了完整配置扫描，并完成运行配置外置化改造。

改造后结论如下：

- 新增统一运行配置文件 `src/main/resources/raha-defaults.properties`。
- 集中外置 148 个运行配置键。
- 原有九类 `defaults()` 默认对象不再在方法中写死数值，统一由 `RahaConfigFactory` 创建。
- 支持类路径默认文件、外部覆盖文件和 JVM 系统属性三级覆盖。
- 外部配置存在未知键、非法整数、非法小数、非法布尔值或非法枚举时立即失败。
- 影响执行语义的资源配置会形成 SHA-256 指纹并进入任务配置版本，防止错误复用旧检查点。
- 显式构造器继续保留，任务级参数仍可覆盖资源默认值，不要求业务代码直接操作 `Properties`。
- 普通 Jar 和完整交付 Jar 均已确认包含 `raha-defaults.properties`。
- 全量 146 个测试通过，失败 0、错误 0、跳过 0。

## 2. 配置边界

本次将工程参数分成四类。

| 类型 | 处理方式 | 典型内容 |
| --- | --- | --- |
| 运行默认配置 | 外置到 `raha-defaults.properties` | 阈值、并发、超时、采样预算、模型参数、保留周期 |
| 任务输入 | 保留在请求或构造器 | 数据集、输入表、行标识字段、快照、调用方、模型版本 |
| 构建与平台基线 | 保留在 `pom.xml` 或已有资源文件 | Java、Spark、Scala、FMDB Jar 组合、Maven 插件版本 |
| 协议与实现常量 | 保留在代码 | 表字段名、错误码、哈希算法、概率边界、数值稳定边界 |

配置外置的目标是减少业务代码侵入，而不是把所有常量都变成可变参数。协议字段和数学不变量如果被任意修改，会破坏持久化兼容、模型可重复性或接口契约，因此不应外置。

## 3. 配置架构

### 3.1 核心组件

| 组件 | 责任 |
| --- | --- |
| `RahaConfigLoader` | 加载类路径默认文件、外部覆盖文件和系统属性，记录文件调用日志 |
| `RahaProperties` | 提供严格的整数、长整数、小数、布尔、枚举、集合和映射解析 |
| `RahaConfigFactory` | 集中创建任务、策略、模型、资源、安全、UDF 和性能配置对象 |
| `RahaDefaultConfigProvider` | 延迟加载默认配置，供现有 `defaults()` 和无参构造器使用 |
| `RahaConfigurationException` | 返回配置错误键和安全错误摘要 |

### 3.2 覆盖优先级

配置按以下顺序合并，后者覆盖前者：

1. 类路径 `raha-defaults.properties`。
2. `-Draha.config.file=<path>` 或环境变量 `RAHA_CONFIG_FILE` 指定的外部文件。
3. JVM 中所有已声明的 `-Draha.*=<value>` 系统属性。
4. 业务代码显式传入的构造器参数或任务请求参数。

`RahaDefaultConfigProvider` 在首次使用时加载并缓存配置。使用启动参数覆盖时，必须在首次访问任何 `defaults()`、UDF 注册器、画像器或默认服务组件之前完成设置。需要在运行期明确装配另一套配置时，应直接创建 `RahaConfigLoader` 和 `RahaConfigFactory`，不要修改共享缓存。

### 3.3 未知键策略

默认文件中的键集合同时充当配置白名单。外部文件或系统属性出现未声明键时，加载器直接抛出 `RahaConfigurationException`，避免拼写错误被静默忽略。

## 10. 依赖与 classpath 代码约定

原 `raha-fmdb-classpath.properties` 和 `raha-dependency-whitelist.properties` 已删除，依赖约束不再通过重复的资源文件维护。

### 10.1 构建依赖约定

Java、Spark、Scala 和 MLlib 版本统一以 `pom.xml` 为准：

- Java 编译目标为 1.8。
- Spark 版本为 3.3.1。
- Scala 二进制版本为 2.12。
- Spark SQL 和 Spark MLlib 使用 `provided` 作用域。
- Maven Enforcer 禁止 Spark 4 和 Scala 2.13 对应依赖进入依赖树。
- Animal Sniffer 校验代码只使用 Java 8 API。

### 10.2 FMDB 运行版本约定

工程不再维护运行时 classpath 精确 Jar 清单，也不在 UDF 启动时扫描或拒绝平台 Jar。

默认运行基线直接跟随 `pom.xml`：

- Spark 版本为 3.3.1。
- Scala 二进制版本为 2.12。
- Spark SQL 和 Spark MLlib 由运行平台提供，不打入业务完整 Jar。
- 构建阶段由 Maven Enforcer 禁止 Spark 4 和 Scala 2.13 依赖。

平台实际部署只需提供与上述 Spark 和 Scala 版本兼容的运行环境，工程不限制 Jar 所在目录，也不校验厂商扩展 Jar 的精确文件名。

## 11. 使用方式

### 11.1 修改工程默认值

直接修改：

```text
src/main/resources/raha-defaults.properties
```

修改后必须重新构建 Jar，并执行全量测试。

### 11.2 使用外部覆盖文件

外部文件只需要填写要覆盖的键，例如：

```properties
raha.resource.max-parallel-strategies=8
raha.resource.max-parallel-columns=6
raha.strategy.max-rvd-column-pairs=300
raha.security.result.mode=HASH_ONLY
```

启动时指定：

```powershell
java -Draha.config.file=D:\conf\raha-production.properties -jar raha.jar
```

也可以使用环境变量：

```powershell
$env:RAHA_CONFIG_FILE='D:\conf\raha-production.properties'
```

### 11.3 使用系统属性覆盖单项

```powershell
java -Draha.resource.max-parallel-columns=8 `
     -Draha.security.result.mode=HASH_ONLY `
     -jar raha.jar
```

### 11.4 显式工厂装配

```java
RahaConfigFactory factory = new RahaConfigFactory(
        new RahaConfigLoader().load(Paths.get("D:/conf/raha-production.properties")));
RahaJobConfig config = factory.jobConfig(
        JobType.DETECTION, "dataset", "source_table", "id");
```

此方式适合需要在同一 JVM 中明确管理多套配置的装配代码。

该指纹进入 `RahaJobConfig.toCanonicalString()`，因此上述配置变化会改变 `ConfigVersioner` 生成的版本。UDF 函数名、安全展示策略、性能基准和资源建议不改变检测计算语义，不纳入执行指纹。

## 15. 最终建议

1. 生产环境使用外部覆盖文件，不直接修改交付 Jar 内默认配置。
2. 外部文件纳入配置中心、变更审批和版本管理，不在日志中打印完整配置内容。
3. 修改执行相关参数后创建新任务，不复用旧任务标识和旧检查点。
4. 分区、并发、RVD 和缓存参数必须结合目标 FMDB 集群压测结果调整。
5. 敏感数据环境优先使用 `HASH_ONLY`；需要展示脱敏值时，明确配置敏感字段和可见长度。
6. 不把数据集、调用方、模型版本或表名写成全局默认值，继续按任务和部署注入。
7. 不将纠正值、修复候选或清洗回写参数加入本配置体系，本工程仍只负责 Raha 数据检测。

本次改造已经将工程中适合运行期调整的主要参数集中外置，同时保留了任务级覆盖、Java 8 兼容、配置版本隔离和原有接口兼容。后续新增可调参数时，应先在 `raha-defaults.properties` 声明，再通过 `RahaConfigFactory` 创建强类型对象，禁止在业务方法中直接读取 `System.getProperty` 或新增散落数值常量。
